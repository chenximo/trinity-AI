# Trinity 原型形态对比与 AI 友好交付

> **用途**：对比 **Figma / 原生 HTML / Vue 五件套** 三种形态在协作与还原上的差异；并约定 **Trinity 后续原型以 Vue 五件套为唯一工程交付物**（不再并行维护「与五件套对等的」独立 HTML 或把 Figma 当第二套长期真源）。  
> **受众**：产品、设计、前端负责人、资深工程师；使用 Cursor 等工具做还原与接 API 的协作者。  
> **与仓库规范的关系**：模块工程形态以 **`docs/Trinity原型模块目录与交付规范.md`** 为准；本页不重复五件套文件表，只讨论**形态对比与交付策略**。

---

## 0. 团队交付约定（后续只做五件套）

| 约定 | 说明 |
|------|------|
| **工程真源** | 新模块、新页面直接在 **`apps/*/src/views/<模块>/`** 按五件套落地；路由、样式、mock、交互、README 与规范对齐。 |
| **设计输入** | Figma、其它设计工具、评审用截图等可作为**输入或附件**，在模块 **`README.md`** 中写明链接、日期、已知偏差；**不要求**在仓库里长期维护一份与五件套「平级」的独立 HTML 原型目录。 |
| **历史静态站** | 根目录 `TrinityAI/` 等静态资产可作为**归档对照或一次性迁移源**；新功能默认不再「先写一整页 HTML 再迁 Vue」作为固定流程。特例（如 **`gen:marketing`** 从 `index.html` 生成骨架）在脚本与模块 README 中写清边界即可。 |
| **评审与走查** | 以 **Vite 跑起来的五件套页面** 为评审主界面；需要像素对比时，可用设计导出图或浏览器截图与 PR 说明，而不依赖单独维护的 HTML 快照站。 |

## 1. 三种形态分别是什么（对比用，非三条并行产线）

| 阶段 | 典型交付物 | 信息形态 |
|------|------------|----------|
| **V1 设计稿**（Figma 等） | 画板、组件、变体、标注、Token（若有） | 视觉与结构意图为主，**非可执行源码** |
| **V2 原生 HTML** | 静态页 + CSS（+ 少量 JS） | **DOM、类名、间距**已落地为浏览器可跑代码 |
| **V3 Vue 五件套**（**当前唯一长期交付**） | `views/<模块>/`：`XxxPage.vue`、`*.css`、`*Interactions.ts`、`mock.ts`、`README.md` | **路由 + 职责拆分 + 假数据 + 交接说明**，与真实工程一致 |

---

## 2. 对「资深工程师 + AI 辅助开发」哪个更好？

**结论（协作效率）**：**V3 ≥ V2 ≫ V1**。在 **「后续只做五件套」** 策略下：

- **长期与 AI 共用的真源只有 V3**；V1、V2 若出现，多为**一次性输入或历史档案**，不必与五件套双轨长期同步。
- **V1**：仍适合定产品范围与视觉方向；对写代码的 AI 而言证据偏软，还原度强依赖标注。
- **V2**：在「曾有一段静态迁 Vue」的历史里很有价值；若不再维护 HTML，**视觉与结构证据**需转移到五件套内的 **CSS + 模板注释 + README 中的设计链/截图说明**。
- **V3**：**可交付、可 diff、可接 API**；README 与 mock 显著降低 AI 幻觉与返工。

**一句话（更新版）**：设计稿表达 **What**；历史上 HTML 曾锁定 **浏览器长什么样**；**今后以 Vue 五件套锁定「仓库里怎么长、改哪里、接什么」**。

## 3. 「页面还原」能到什么程度？（多维度对照）

| 维度 | 设计稿（Figma 等） | 原生 HTML | Vue 五件套 |
|------|-------------------|-----------|------------|
| **视觉 / 版式** | 高潜力，依赖标注与 Token | 高，可直接对照 DOM/CSS | 高；依赖 **`*.css` + 模板** 与全局 token |
| **交互 / 状态** | 中低 | 中 | 中高，`*Interactions.ts` + `mock.ts` |
| **与真实工程对齐** | 低 | 中 | **高** |
| **AI 可用硬证据** | 图 + 零散说明 | 整页源码 | **源码 + README / mock 契约** |
| **接后端 / API** | 弱 | 弱～中 | **强** |

## 5. 只做五件套时，怎样「做得更好」？（可执行清单）

### 5.1 把「设计输入」接进五件套（替代「再维护一份 HTML」）

- **README 必备**：设计稿 / 评审文档 **URL**、**对齐日期**、与实现已知差异（留白、动效未做等）。  
- **关键屏**：可选在 PR 或 `docs/` 放 **截图 + 标注**，便于 AI 与人类 diff，而不依赖单独 HTML 站。  
- **Token 与类名**：依赖 `packages/tokens`、`trinity-base` 的写在 README；若设计侧有特殊 `orc-*` / `or-*` 约定，**一句话指向**设计规范或域说明。  
- **状态与边界**：空态 / 错误 / 加载在模板或 README 中标「未做 / 占位」，减少 AI 自行发明。

### 5.2 五件套工程内要守住的底线

- **README 当契约**：路由 name、懒加载、全局 CSS 依赖、接 API 时先改 `mock.ts` 再动交互（与规范 §4、§6 一致）；**二次开发补充**按 **`docs/Trinity原型模块目录与交付规范.md` §6.1** 在 README 用表格登记。  
- **mock / 交互边界**：`mock.ts` 不写 DOM；`document` / 键盘等进 `*Interactions.ts`。  
- **全局类名与 CSS**：需要 `orc-*`、`or-*` 等稳定类名时，用 **非 `*.module.css` 的全局表**（与 Chat / 规范一致）。  
- **模板分区**：大模板用 HTML 注释分块（侧栏 / 主列 / 弹层），方便评审与 AI 局部修改。  
- **`gen:*` 脚本**：若从静态文件覆盖 Vue，在 **脚本头 + README** 写清覆盖范围，避免误冲手工结构。

### 5.3 仍可能用到设计工具时（可选，非必经产线）

- 若团队仍用 Figma 等出稿：保持 **Token、命名与状态** 齐套即可，**不必**为每个迭代再导出整站 HTML 作为第二真源；差异在 README 闭环。

### 5.4 通用增强（与形态无关）

- **术语表**：`orPage`、`tai-*`、`window.TrinityOR` 等见 **`docs/Trinity开发枢纽与AI协作流程.md`**。  
- **可选视觉回归**：Playwright 截图或 Storybook，锁住「改壳不伤页」。  
- **`.cursor/rules` / AGENTS.md**：缩短新会话的上下文冷启动。  
- **改版脚注**：模块 README 或规范脚注 **对齐日期**。
