# 工程师 · 如何消费原型

> **仓库**：`TrinityAI-web` · **只改 `apps/`**，**不改 `apps_ui/`**  
> **双仓约定与 PM 发布**：[双仓协作与原型交付.md](./双仓协作与原型交付.md) · 五件套 [`§6.2`](../../01-原型与交付规范/Trinity原型模块目录与交付规范.md#62-readme-固定节交付与工程师对齐)

---

## 1. pull 之后看什么

| 材料 | 路径 |
|------|------|
| 原型五件套 | `apps_ui/<app>/src/views/<模块>/` |
| 路由与落点 | 同上 `README.md` → **`## 工程对齐`** |
| 变更摘要（可选） | `apps_ui/CHANGELOG.md` 最新一节 |

本地预览：`pnpm run dev:ui`。

---

## 2. 怎么做

```text
git pull → 读 README 映射 → 只读对照 apps_ui 五件套 → 在 apps 实现
```

| 情形 | 做法 |
|------|------|
| **首次（0→1）** | 映射为「未落地」→ 在 `apps` 新建页，结构参考 `apps_ui` |
| **迭代** | 在已有 `apps` 页**增量改**；可看 CHANGELOG + `git diff apps_ui/该模块/` |

**禁止**：改 `apps_ui`；用 `apps_ui` 整文件覆盖 `apps`（会丢掉 API、鉴权、路由）；同一 commit 同时提交 `apps/` 与 `apps_ui/`。

---

## 3. 给 AI 的一句提示

```text
git pull 后，按 apps_ui/.../README「工程对齐」的映射，只读 apps_ui、只改 apps。
首次建页；迭代按 diff 增量改，保留工程侧 API 与鉴权。
```
