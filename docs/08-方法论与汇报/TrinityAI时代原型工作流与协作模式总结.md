## 0. 现状一句话

当前模式（原型即代码 + 五件套交付 + apps_ui/apps 互斥）在 AI 时代**框架非常先进**，但缺一块关键拼图——**机器可读的变更契约**。补齐这块拼图，就能从「人工交付流」进化为「AI 原生协作流」。

---

## 1. 当前模式评估

### 1.1 已经做对的核心决策

| 决策 | 为什么这在 AI 时代是优势 |
|------|------------------------|
| **Vue 五件套为唯一工程真源** | AI 能直接读代码、diff、迁移；Figma/HTML 对 AI 都是「软证据」 |
| **README 当契约 + mock 数据独立** | 极大降低 AI 幻觉，新会话冷启动成本最低 |
| **apps_ui / apps 职责分离 + 提交互斥** | 避免「混改」导致的不可追溯，AI 助手边界清晰 |
| **packages 共用 + Token 体系** | 设计变更可通过依赖关系自动传导 |
| **.cursor/rules 已建制化** | AI 助手开箱即知的约束 |

### 1.2 真正的瓶颈

瓶颈**不在**「原型即代码」理念本身，而在于：

> **原型更新后，工程师的 AI 缺乏一个「结构化、机器可读的变更摘要」来快速理解「哪里变了、怎么变、为什么变」。**

工程师 AI 目前需要：
1. 自己 diff `apps_ui/` 的代码
2. 试图理解变更意图
3. 再决定如何在 `apps/` 中落地

这第三步对 AI 来说最难——**代码 diff 不承载设计意图**。

### 1.3 分版本开发的额外复杂度

产品经理一次输出的是**「设计全景」**，工程师按**「开发节奏」**分批实现。如果没有显式的版本标记，工程师 AI 面对完整原型时无法判断：
- 这个按钮是 V1.0 必须做的，还是 V2.0 才需要？
- 这个弹窗当前版本只需要静态展示，还是要接真实数据？
- 这个筛选条件是 P0 还是 P1？

## 2. AI 时代的演进方向

当前模式是**「人工交付流」的代码化升级**。下一步应向**「AI 原生协作流」**演进：

```
产品经理侧 AI                工程师侧 AI
     ↓                           ↑
  五件套 + 变更契约  ───────────→  自动感知、建议迁移
```

**关键变化**：产品经理交付给工程师的，不应该只是「更新后的代码」，还应该包括**「机器可读的变更契约」**（changelog、接口 diff、视觉变更清单、版本映射）。

---

## 3. 建议创建的 Skill 体系（产品经理侧）

基于现有的 `原型设计统一.skill.md`，建议产品经理侧补齐 **3 个核心 skill**，形成闭环：

| Skill | 作用 | 解决什么问题 |
|-------|------|-------------|
| **Skill 1：五件套原型生成与自检** | 从设计意图生成符合五件套规范的模块，并自动检查规范一致性 | 保证新原型质量稳定，减少人工 review |
| **Skill 2：设计规范与 Token 治理** | 管理 `packages/tokens`、`trinity-base.css` 的变更，自动检查 apps_ui 中是否有魔法数 | 解决「设计更新后，原型各处是否同步」的问题 |
| **Skill 3：原型变更记录与交付** | 当 apps_ui 更新时，自动生成结构化 changelog（视觉变更 / 交互变更 / 数据契约变更 / 新增模块 / 版本映射） | **这是工程师 AI「快速对比更新」的核心材料** |

---

## 4. 核心解决方案

### 4.1 变更契约（解决「工程师 AI 如何快速感知更新」）

产品经理每次迭代 apps_ui 后，Skill 3 自动生成一份**结构化变更记录**，包含：

| 变更模块 | 变更类型 | 摘要 | 影响范围 | 对接建议 |
|---------|---------|------|---------|---------|
| `views/chat/` | 视觉 | 发送按钮从描边改为渐变 | CSS 类名 `orc-send` | 同步替换 apps/web 中同类样式 |
| `views/models/` | 交互 | 新增筛选抽屉 | `modelsInteractions.ts` 新增 `openFilterDrawer` | 参考实现拷贝到工程侧 |
| `views/home/` | 数据 | Hero 区供应商从 6 个增至 8 个 | `mock.ts` 中 `HOME_HERO_PROVIDERS` | 同步更新接口字段 |

**工程师 AI 的第一步不再是「读代码」，而是「读变更契约」**，然后针对性地去 apps_ui 里看实现细节。

### 4.2 版本映射（解决「分版本开发」问题）

在五件套基础上，增加**「版本 - 优先级 - 状态」三层标记**：

#### 第一层：模块级版本契约（在 README 中）

```markdown
## 开发版本契约

| 功能点 | 所属版本 | 优先级 | 原型状态 | 工程状态 |
|--------|---------|--------|---------|---------|
| 仪表盘布局 | V1.0 | P0 | ✅ 完成 | 🔄 开发中 |
| 实时数据刷新 | V1.0 | P0 | ✅ 完成 | ⏳ 待开发 |
| 图表 Hover 提示 | V1.1 | P1 | ✅ 完成 | ⏸️ 跳过 |
| CSV 导出 | V1.1 | P1 | ✅ 完成 | ⏸️ 跳过 |
| 自定义布局 | V1.2 | P2 | ✅ 完成 | ⏸️ 跳过 |
| AI 分析面板 | V2.0 | P2 | 📝 设计稿 | ⏸️ 跳过 |

**当前工程师应聚焦：V1.0 P0 项**
```

#### 第二层：代码级标记（在 Vue 模板中）

用 HTML 注释在模板中直接标记，便于 AI 搜索和理解：

```vue
<!-- V1.0 必须：顶部统计卡片区 -->
<div class="dashboard-stats">...</div>

<!-- V1.1 新增：图表交互层（当前版本跳过） -->
<!-- VERSION: V1.1 -->
<div class="chart-tooltip-layer" v-if="false">...</div>

<!-- V2.0 规划：AI 分析面板（仅占位，不实现） -->
<!-- VERSION: V2.0 | STATUS: PLANNED -->
<div class="ai-insight-panel-placeholder">
  <!-- 占位：待 V2.0 开发 -->
</div>
```

#### 第三层：交互与 Mock 的标记

```ts
// mock.ts
export const MOCK_DASHBOARD_STATS = [
  // V1.0: 静态展示数据
  { label: '今日请求', value: '12,345', change: '+12%' },
  
  // V1.1: 增加趋势数据（当前 mock 可空，预留结构）
  // VERSION: V1.1
  trendData: null, // 待 V1.1 接入
];

// interactions.ts
export function useDashboardInteractions() {
  // V1.0: 基础导航和刷新
  const refreshData = () => { ... };
  
  // V1.1: 图表 hover 交互（标记为未激活）
  // VERSION: V1.1 | ACTIVE: false
  const showChartTooltip = () => { ... };
  
  return { refreshData };
}
```

#### 版本标记原则

| 原则 | 说明 |
|------|------|
| **标记是产品侧的契约，不是注释** | 产品经理负责维护和更新，不是工程师临时加的 |
| **版本标记跟着原型走** | 原型迭代时，版本映射同步更新 |
| **默认保守原则** | 未标记版本的功能，工程师 AI 默认「当前版本不实现」 |
| **渐进披露** | 工程师只看他需要实现的版本，不被远期设计干扰 |

### 4.3 仓库职责分层

明确 `Trinity/` 与 `TrinityAI-web/` 的关系：

| 仓库/目录 | 角色 | 维护者 | 变更场景 |
|----------|------|--------|---------|
| `Trinity/docs/` | 设计规范与 PRD 真源 | 产品经理 | 产品定义、需求变更、工作流文档 |
| `TrinityAI-web/apps_ui/` | 可执行原型真源 | 产品经理 | 页面级原型迭代 |
| `TrinityAI-web/packages/` | 共用 Token / 组件 | 双方 | 设计规范落地为代码 |
| `TrinityAI-web/apps/` | 工程实现 | 工程师 | 生产级开发 |

产品经理的日常：在 `apps_ui/` 改原型 → Skill 自动生成变更契约 → 交付给工程师。

## 5. 下一步行动建议

### 第一步：创建 / 完善 Skill（产品经理侧）

1. 把现有的 `原型设计统一.skill.md` 升级为更工程化的版本（Skill 2 的基础）
2. 新建**「五件套原型生成与自检」**Skill
3. 新建**「原型变更记录与交付」**Skill（解决核心问题的关键）

### 第二步：在 apps_ui 中试点

1. 选一个即将迭代的模块（如 `trinity-ai-admin` 的新页面）
2. 用新 Skill 走一遍「生成 → 迭代 → 输出变更契约」的完整流程
3. 让工程师用变更契约来驱动 `apps/` 的同步

### 第三步：验证闭环

1. 产品经理更新 apps_ui 原型
2. Skill 自动生成变更记录（含版本映射）
3. 工程师 AI 读取变更记录，快速定位需要同步的范围
4. 验证工程师落地效率是否提升
