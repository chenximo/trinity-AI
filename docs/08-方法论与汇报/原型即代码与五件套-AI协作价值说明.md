# 「原型即代码 + 五件套」· AI 协作价值说明

> **文档类型**：方法论 / 汇报素材（**非工程操作手册**）  
> **受众**：产品经理本人、答辩与汇报听众、合作方说明  
> **工程真源请读**：[`01-原型与交付规范/Trinity原型模块目录与交付规范.md`](../01-原型与交付规范/Trinity原型模块目录与交付规范.md)（含 §6.3 双仓摘要）

---

## 1. 核心结论（可放在 PPT 首页）

**「原型即代码 + 五件套」在 AI 时代非常适合人机协作**：把产品意图写成可运行、可版本化、可 diff 的代码，并按固定模块结构交付，AI 与人共享同一套事实来源，而不是猜设计稿或口头交接。

---

## 2. 为什么对 AI 友好（总表）

| 做法 | 为什么对 AI 友好 |
|------|------------------|
| **设计规范 / 色板 / 原子组件都是代码** | 真源唯一，AI 不必猜 Figma 或截图里的数值 |
| **模块五件套**（Page / CSS / Interactions / mock / README） | 上下文边界清晰，一次只读一个模块，冷启动成本低 |
| **`packages` 共用**（`@trinity/ui`、`@trinity/tokens` 等） | 工程师不必重复抄 Button、Token；设计变更可沿依赖传导 |
| **`apps` vs `apps_ui` 分责 + 提交互斥** | 权责清楚，PR 好审，AI 也不容易在工程任务里「顺手改原型」 |
| **双仓（Trinity 真源 → 工程仓镜像发布）** | 产品沉淀与工程交付分离，发布 commit 成为工程师可感知的版本锚点 |

---

## 3. 五件套是什么（不是「自创玄学」）

五件套本质是页面级模块的 **五类职责拆分**，与成熟设计体系里的 **「组件 + Story + 文档」** 同一类思路，只是落到 Vue 原型工程：

| 文件 | 职责 | 类比 |
|------|------|------|
| `XxxPage.vue` | 视图结构 | 页面 / Story 的 DOM |
| `*.css` | 样式 | 视觉层 |
| `*Interactions.ts` | 交互、壳层逻辑 | 行为 / 事件 |
| `mock.ts` | 假数据与纯函数 | **数据契约**（Mock as contract） |
| `README.md` | 路由、依赖、交接、变更 | **交接说明** |

对 **产品经理**：每个模块自带说明书，评审与迭代有边界。  
对 **工程师与 AI**：接 API 时先换 `mock`，再动交互；README 降低幻觉。

规范全文（文件名、反例、体量分级）：[`01-原型与交付规范/Trinity原型模块目录与交付规范.md`](../01-原型与交付规范/Trinity原型模块目录与交付规范.md)。

## 4. 与常见方案的对比（汇报用简表）

| 方案 | AI 可读性 | 版本化 / diff | 适合 PM+工程长期协作 |
|------|-----------|---------------|----------------------|
| Figma 为主 | 弱（软证据） | 弱 | 中 |
| 静态 HTML 原型 | 中 | 中 | 中（易与工程双轨） |
| **Vue 五件套（本方案）** | **强** | **强** | **强** |
| 仅 PRD 文字 | 弱 | 中 | 弱（难对齐 UI 结构） |
