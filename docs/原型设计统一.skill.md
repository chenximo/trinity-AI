## 文档与技能命名约定（本仓库）

- **技能文件**：放在 **`docs/`**，文件名用 **中文** 概括用途，并以 **`.skill.md`** 结尾。示例：`docs/原型设计统一.skill.md`。
- **普通文档**：放在 **`docs/`**，**中文文件名**，读名知义；Trinity 相关建议 **`Trinity` 前缀**（如 `Trinity原型版式与视觉规范.md`、`Trinity对话页原型说明.md`）。
- **AI 生成提示词**（给 Figma Make、对话里生成静态页等的一次性指令稿）：同样放 **`docs/`**、中文文件名；文首用引用块标注 **`文档类型：AI 生成提示词`** 与 **`与实现对齐`**（写明对照的页面路径、**校验日期**，或写明 **已与当前实现脱钩**）。避免被当作 PRD 或与 `Trinity原型版式与视觉规范.md` 同级的单一事实来源。
- **前期方案对比稿**（多方案并列评审、探索用提示词或说明）：文首增加 **`文档状态：归档（前期方案对比）`**，并写明**非交付口径、不默认随现网迭代**；**文件名建议含 `（归档）`**（如 `…生成提示词（归档）.md`），便于目录与搜索一眼区分。若某一稿升格为现行规格，应去掉文件名中的归档标记、改写文首状态并合并进「普通文档」或对应模块说明，避免长期双轨。
- **不推荐**：含糊英文 slug 作主文件名（如 ~~`PROTOTYPE_DESIGN.md`~~）。
- **在 Cursor 中使用**：优先将 **`docs/原型设计统一.skill.md`** 加入技能引用；本仓库同时在 **`.cursor/skills/prototype-design-unification/SKILL.md`** 保留简短入口（链接到上述正文），便于沿用默认 skills 目录的发现方式。

# 产品原型视觉体系统一与交互规范

在改动 **Trinity 系列静态原型**（`TrinityAI/`、`TrinityAI_Admin/`、`AIRouter/` 等）时，把 **单一事实来源** 当作约束，避免单页「自创」色值、字体或间距。

## 必读规范与设计系统

开始较大范围的原型/UI 改动前，先读：

- [docs/Trinity原型版式与视觉规范.md](Trinity原型版式与视觉规范.md) — 版心、顶栏/正文边距、**§1.4 字号与字重阶梯**、页面模板归属、校验表。

**设计系统（指令化，但以本仓库为最高优先级）：**

1. **Token 优先**：主色、灰阶、圆角、阴影、间距凡已有变量或已有组件样式，**一律复用** `Trinity/assets/trinity-base.css` / `TrinityAI_Admin/admin.css` / 母版页中的定义；禁止为「好看」在单页堆魔法数。
2. **新增裸数值时的节奏**：在确无变量可用时，间距优先落在 **4 的倍数**（如 4 / 8 / 12 / 16 / 24 / 32px）上，与现有页面节奏一致；随后应提炼为变量或写入 `Trinity原型版式与视觉规范.md` 与全局样式，避免一稿多标准。
3. **字体层级**：Inter + Noto Sans SC；**字号与字重阶梯以 `Trinity原型版式与视觉规范.md` §1.4 为准**（`L1`–`L6`），不另起刻度；若业务确需新档位，须同步改 §1.4 与全局样式。
4. **与 Ant Design / Arco 等的关系**：可作 **交互与信息架构** 的参照（模态遮罩与取消/确认、表单错误态、加载占位等），但 **实现不得假设已引入 Ant/Arco 组件库**；样式与类名仍须落在本项目 token 与现有 HTML/CSS 模式上，避免第三方主题覆盖品牌规范。

若文档与实现冲突，以 **文档 + 全局 CSS 变量** 为准；改实现后应同步更新文档中的变量名或数值说明。

## 多模态输入（参考图、竞品截图、手绘）

当用户提供 **截图、设计图或照片** 时，在动手前与用户或上下文对齐意图（若未说明则默认 **结构优先**）：

| 模式 | 做法 |
|------|------|
| **结构 / 布局参考**（推荐默认） | 提取信息层级、区块划分、主次要操作；用本项目组件与变量重建，**不**追求像素级复刻截图中的色值与字号。 |
| **高保真还原**（仅当用户明确要求） | 尽量对齐视觉，但仍将颜色、圆角、间距 **映射到** `:root` 变量或现有类，避免大量一次性 HEX。 |

流程建议：识别文案与控件语义 → 生成/修改 HTML 骨架 → 用变量与现有模式上样式 → 对照参考图做一轮自检（对比度、对齐、状态是否缺失）。

## 模块原型：对话里定下来的事要主动写进文档

与用户一起 **设计某个模块的原型** 时，不要只把结论留在对话里；在当次或紧随其后的编辑中 **主动创建或更新** 该模块对应的设计记录，至少覆盖：

| 块 | 写什么 |
|----|--------|
| **设计思路** | 目标用户与场景、要解决什么问题、关键取舍、与现有产品其他模块的关系。 |
| **产品交互** | 主路径与分支；**状态**（空、加载、错误、成功反馈如 Toast/行内提示）；**表单与输入**的校验规则（必填、格式、错误文案展示位置）；入口出口；与侧栏/顶栏/弹层等的衔接。若与实现不一致，以**即将落地或已落地的原型**为准更新文档。 |
| **产品参考** | 对标产品或界面（名称 + 链接或截图路径）、借鉴点与**不**借鉴点；多模态输入时注明上文「结构优先 / 高保真」中的哪一种。 |

**存放约定（按优先顺序）：**

1. 若仓库已有该模块 PRD：在 PRD 同目录或 `docs/` 下 **增补「原型 / 交互摘要」小节**。
2. 否则在 **`docs/`** 下新建 **中文文件名** 的模块说明（如 `Trinity对话页原型说明.md`），避免再建仅英文 slug 的深层目录作为主入口。
3. 若用户指定路径，以用户指定为准，但三类信息仍须写全。

**时机**：对话中方案已定，或刚改完对应 HTML/CSS/JS 时，**同一轮或下一轮**更新文档；未定论可先列「待确认」，定论后再改为正式描述。

## 单一事实来源（不要散落魔法数）

| 范围 | 文件 / 约定 |
|------|-------------|
| 用户侧 / Cloud 全局 token、顶栏、内页 gutter | `Trinity/assets/trinity-base.css` |
| 云落地首页局部 | `TrinityCloud/home.html` 内 `:root`（见 `Trinity原型版式与视觉规范.md`） |
| 管理台 | `TrinityAI_Admin/admin.css` |

**禁止**：在单页内联或局部 `<style>` 中写新的主色蓝、新字体栈或与全局不一致的圆角体系（除非用户明确要求实验页且会回收）。

## 字体与主题色

- **字体与字号**：**Inter + Noto Sans SC**；正文行高约 `1.5`；**字号阶梯见 `Trinity原型版式与视觉规范.md` §1.4**。
- **主色**：`#2563eb` 仅通过 `var(--blue)` / `var(--adm-blue)` 使用。
- **辅助色与按钮**：使用已有 `--blue-soft`、`--blue-ring`、`--text`、`--muted`、`--border` 等；主操作 `btn-gradient`（或项目内等价），次要描边/幽灵与现有 hover、focus 一致。

## 风格、组件与响应式

- **圆角**：与 `--radius`、`--adm-radius` 对齐。
- **列表 / 选择行 / 弹层**：与邻近页面已有组件对齐；优先复用类名与结构。
- **下拉菜单行（统一「图标 + 标题」）**：浮层/`<details>` 内可点行一律 **`display: flex; align-items: center; gap`**；左侧 **`span.*-item-ico`（`aria-hidden`）+ 线性 SVG（约 18×18，`stroke`）**，右侧 **`span.*-item-label`**。默认图标色 **`var(--muted-2)`**。**与主渐变主操作（如对话发送 `.orc-send`）同列、需视觉对齐时**：悬停 / `:focus-visible` 使用 **`background: var(--grad)`**、**`color: #fff`**、图标 **`currentColor` 随字变白**，阴影与发送钮同系（如 **`0 1px 4px rgba(37, 99, 235, 0.3~0.35)`**），`:focus-visible` 外加 **`outline: 2px solid var(--blue-ring)`**；**`[open]` 的 `<summary>` 触发器**亦可同系渐变以表明展开态。轻量侧栏菜单若不需与主按钮抢视觉，可退化为 **`var(--blue-soft)` + `var(--blue)`** 字色。禁用项保持灰态且 **不改变** 悬停背景。参考：`TrinityAI/app/chat/index.html` **「更多」**（`.orc-preset-more-item`）；新开下拉勿再写纯文字行或各页一套 hover 色。
- **响应式**：新页需说明或实现 **PC 与窄屏** 行为（如侧栏折叠、单列、触控热区）；断点与现有 `TrinityAI/app/chat/chat-openrouter.css` 等中的 `@media` 策略保持一致，并在模块文档中写明关键断点与布局变化。
- **模板**：按 `Trinity原型版式与视觉规范.md` §3 选择母版页。

## 交付与评审

- **可访问性**：表单控件有关联 `label` 或 `aria-label`；图标按钮有 `title`/`aria-label`；焦点可见（与现有 `focus`/`ring` 一致）；关键操作不仅靠颜色传达状态。
- **说明 vs 主排版**：面向评审的**长说明、截图对齐类「标注」**优先放在 **ⓘ 帮助气泡**（如对话页 `#orc-help-tip-popover` + `<template id="orc-help-tpl-*">`），与 `data-orc-prototype-annotation` 一并可脚本剥离；**避免**在输入框、主按钮旁再叠一整条常驻灰字（除非 PRD 要求正式产品文案）。底栏示例见 `TrinityAI/app/chat/index.html` 中 `orc-composer-help-btn` / `orc-help-tpl-composer`。
- **评审入口**：在模块文档中写明静态页路径（如 `TrinityAI/app/chat/index.html`）；若团队使用 Figma、Axure、内网预览等，**补充链接或导出资产路径**，便于业务与开发对齐。

## 改动后自检（可对照 `Trinity原型版式与视觉规范.md` §5）

- [ ] 未引入与 `Trinity/assets/trinity-base.css` / `TrinityAI_Admin/admin.css` 冲突的主色或字体栈；新增文案字号落在 **§1.4** 约定区间内。
- [ ] 顶栏与正文留白符合 `--page-gutter`、`--home-content-inline`、`--adm-page-inline` 等约定。
- [ ] hover/focus/ring 与全局一致。
- [ ] 交互文档与界面一致：空/加载/错误/成功是否有对应说明或占位实现；表单校验与错误提示是否可评审。
- [ ] 响应式与 a11y 在模块文档或注释中有交代（若该模块有移动端场景）。
- [ ] 若调整全局变量或布局规则，已更新 `docs/Trinity原型版式与视觉规范.md` 并全局检索相关变量。
- [ ] **模块原型** 会话：已更新或新建模块文档（设计思路 / 产品交互 / 产品参考）。
- [ ] **下拉菜单行**：新增浮层/菜单项采用 **icon + title**，高亮与 **`--grad` / `--blue-ring`** 等与主操作对齐（见本技能「下拉菜单行」）；与已有 `.orc-preset-more-item` 模式一致或可抽到 `trinity-base.css` 后再引用。

全局视觉与版式以本技能与 **`docs/Trinity原型版式与视觉规范.md`** 为准；**模块级** 思路、交互与参考以约定模块文档为准，与 PRD 互补。字段含义、接口约定仍以 PRD 或代码注释为准。
