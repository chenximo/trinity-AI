# AI API 聚合平台 · 产品全景与介绍

> **文档用途**：说明这个平台**是干什么的**、**提供哪些能力**，方便销售、售前、客户成功与研发对齐同一张「功能地图」。  
> **版本**：v1.2（2026-05-11）— 修订见文末。

---

## 1. 它是干什么的

这是一个 **AI API 聚合平台**（偏 **B2B / B2B2C**）：把多家 **上游模型供应商** 的能力接进来，通过 **统一的 API 网关** 对外提供调用；企业以 **租户** 身份接入，按合同与套餐使用模型，平台负责 **路由、计量、计费、风控与运营配置**。

可以把它理解成：**企业不用分别对接每家模型厂商**，在一条统一通道上完成鉴权、调用、看用量、管密钥和对账；平台方则用一套后台管理所有客户、线路与策略。

---

## 2. 用来解决哪些问题（典型场景）

| 场景 | 平台起什么作用 |
|------|------------------|
| **多模型、多厂商** | 一次对接平台，按需选用不同模型与线路，减少自建路由与适配成本。 |
| **稳定性与成本** | 同模型可配置多条供应线路，支持优先级、主备与降级，在故障或涨价时可控切换。 |
| **企业采购与对账** | 按组织、项目、合同维度看清用量与账单，支撑内部成本分摊与对客结算。 |
| **安全与合规** | 密钥生命周期、跨租户异常排查、用户注册审核、黑白名单与操作审计，满足企业治理要求。 |
| **运营与销售** | 上下架模型、配置套餐与刊例、维护客户与授信，支撑商业化运营。 |

## 3. 平台有哪些功能（能力地图）

下面按 **「能做什么」** 归类；**租户侧**（客户开发者用的控制台）与 **平台运营侧**（内部员工用的后台）分开写，避免混在一起。

### 3.1 租户 / 开发者侧（客户组织内使用）

面向签了合同、要在自己系统里 **调用 AI API** 的人，通常提供：

- **API 密钥**：创建、命名、轮换；控制谁能调用（与角色/权限配合）。  
- **额度与套餐**：可用量、周期、告警；与合同中的套餐绑定（具体形态依产品定义）。  
- **调用与用量**：按时间、模型、密钥等查看用量，便于排错与成本归因。  
- **活动 / 日志类信息**：关键操作、异常提示等（与产品形态一致即可）。

以上在工程上常集中在 **Account 控制台**（单页多区：密钥、额度、活动、用量等），与「平台内部运营后台」是 **两套界面、两类账号**。

### 3.2 平台运营侧（平台公司内部使用）

面向 **运营、财务、风控、超管** 等内部角色，在 **运营后台** 中配置与治理整条业务链，主要包括：

**接入与模型**

- **供应商管理**：维护各家供应商档案、结算信息、对接参数、连通性探测、密钥轮换策略等。  
- **模型与线路**：维护平台对外提供的模型目录、能力说明、上下架与灰度；为每个模型配置 **供应线路**（走哪条上游、区域、成本与优先级）。  
- **双类上游（产品概念）**：一类是模型 **原生研发商** 直连 API；一类是 **云/转售供应商** 提供的多模型线路（常带 Profile、JSON 等配置）。平台在网关侧做 **请求/响应映射** 与 **降级策略**，对客户仍暴露统一调用方式。  
- **路由与限流策略**：按客户、模型、标签等配置路由规则、Fallback 链、灰度比例、全局限流与客户级例外等。

**客户与商业**

- **客户与合同**：租户主数据、组织与项目、合同期限与折扣、授信与开票抬头等。  
- **用量与计费**：用量明细、配额监控、套餐 SKU、账单、调账与对账导出等。

**安全与治理**

- **API 密钥（平台视角）**：跨租户检索、脱敏查看、冻结与吊销、审计轨迹、风险标签（与监控联动）等。  
- **用户与认证**（终端用户，非员工）：注册审核、白/黑名单、实名或企业认证等。  
- **平台权限**：内部管理员、角色、菜单与数据范围等（与租户内角色分离）。

**内容与经营分析**

- **文档中心（运营）**：对外文档的版本、发布、可见范围（公开 / 指定客户等）。  
- **报表中心**：用量与消耗、收入试算等预置报表，多维分析与导出任务等。

**运维与系统**

- **工作台**：关键指标、待办入口、跳转到各模块（深度可分期迭代）。  
- **监控与运维**：实时或准实时大盘、错误与延迟、供应商健康、告警与维护窗口等。  
- **系统与合规**：全平台操作审计、敏感操作与批量导出审批、特性开关、全局参数等。

### 3.3 本仓库里，功能模块主要在哪（`views`）

在 Trinity 工程里，**各 app 的用户界面功能**基本按模块落在 **`src/views/`** 下一层目录（再配 `mock.ts`、`*Interactions.ts`、样式等）；**路由与壳**在路由表与 `admin-shell` / `shell` 里把模块拼成完整应用。

**运营后台（平台员工）** — `apps/trinity-ai-admin/src/views/`

| 目录 | 对应产品能力（与上文 §3.2 大致对齐） |
|------|--------------------------------------|
| `admin-shell/` | 侧栏、顶栏、面包屑、Tags、子路由表；未单独做页的模块先走占位子路由 |
| `admin-dashboard/` | 工作台 |
| `admin-ops/` | 监控与运维 |
| `admin-billing/` | 用量与计费 |
| `admin-suppliers/` | 供应商 |
| `admin-models/` | 模型管理 |
| `admin-customers/` | 客户与合同 |
| `admin-keys/` | API 密钥（平台视角） |
| `admin-users/` | 用户与认证 |
| `admin-access/` | 平台权限 |

详设里还有 **客户与合同、文档中心、报表中心、系统与合规** 等一级域：侧栏子菜单与路由已预留，多数仍以 **`AdminStubPage`** 占位，后续再按批次换成与上表同构的 `admin-*/` 实页。（**API 与路由策略 §4.6 / P5：v1 不做**，侧栏已注释。）

**用户站（租户、访客）** — `apps/trinity-ai/src/views/`

| 目录 | 说明 |
|------|------|
| `account/` | 租户 **Account 控制台**（密钥、额度、活动、用量等；与运营后台分轨） |
| `home/` | 官网 / 落地首页等 |
| `models/` | 用户侧模型目录等（与运营侧「模型管理」分工不同） |
| `chat/` | 对话类产品页（若产品线包含） |
| `shell/` | 用户站壳布局与主导航注入 |

需要 **路由名、hash 与静态页对照** 时，以各目录下的 **`README.md`** 为准（例如 `views/account/README.md`）。

`apps/trinity-portal/src/views/` 多为 **开发枢纽** 入口骨架；聚合 API 相关的产品界面仍以 **`trinity-ai-admin`** 与 **`trinity-ai`** 的 `views` 为主。

## 4. 运行时大致怎么转（帮助理解「功能」落在哪里）

1. **运营后台** 预先配好：模型、线路、客户套餐、路由与限流等 **策略**。  
2. **租户** 在控制台创建 **API 密钥**，应用带着密钥调用 **统一网关**。  
3. **网关** 做鉴权、选路、限流与埋点，请求转到对应 **上游供应商**，响应再经必要映射返回客户。  
4. **计量与账单** 根据流水聚合用量、生成账单与报表；**审计与风控** 记录关键操作与异常密钥行为。

**边界（避免期望错位）**：平台聚焦 **聚合调用、线路与商业化配套**，不是「自研大模型训练平台」；终端对话式 C 端产品与「企业调 API」可以是同品牌下的不同产品线，需在对外话术中单列说明。
