## 1. 背景与目标

### 1.1 产品是什么

面向 **B2B / B2B2C** 的 **AI API 聚合**：上游对接多家 **供应商**，下游服务 **租户（客户组织）** 的调用与计费。本文档所述 **「运营后台」** 仅服务 **平台内部员工**，用于配置模型与线路、管理 **租户主数据与合同账单（运营视角）**、全平台治理类能力。**租户自助**（API 密钥、额度、活动、用量等 **Account 控制台五区**）不在本文展开，真源见 **`apps/trinity-ai/src/views/account/README.md`**。

### 1.2 建设目标

| 目标 | 说明 |
|------|------|
| **可运营** | 模型上下架、多供应商路由与主备切换、客户与套餐全生命周期可管。 |
| **可计量** | 用量、消耗、按模型 / 按客户 / 按供应商多维聚合；支撑对账与报表。 |
| **可管控** | 用户审核、黑白名单、权限分级、关键操作留痕。 |
| **可扩展** | 模块边界清晰，便于后续接工单、计费引擎、多区域等。 |

### 1.3 非目标（v1 明确不做或仅预留）

- 终端用户侧 **对话式产品 UI**（与聚合 API 控制台区分，除非产品定义为同一品牌下的子模块）。  
- **自研大模型训练** 平台（仅管理「调用哪个上游模型」）。  
- 完整 **财务 ERP**（发票 / 税务规则以「对接导出 + 人工」为主时可降级为二期）。  
- **租户侧 Account 控制台 UI 与交互说明**（真源 **`apps/trinity-ai/src/views/account/`**；本文仅互链，不双轨维护该域细节）。

> 分模块放大见 **§2**（信息架构与导航）；本节约为**一页鸟瞰**，评审与 onboarding 时优先看本节。

## 3. 角色与权限（概要）

建议采用 **RBAC + 数据范围**（如「仅本客户组」「全平台」）。菜单与 **§2 总表** 一级目录对齐后，按角色授予 **模块 + 数据范围**（见 **§4.12** 细化为菜单 / 按钮 / 字段）。

| 角色 | 典型职责 | 数据范围建议 |
|------|----------|----------------|
| **超级管理员** | 管理员账号、全局配置、供应商主数据 | 全平台 |
| **运营** | 模型上下架、路由、公告、文档上架 | 全平台读；写按菜单授权 |
| **商务 / 客户成功** | 客户账号、合同、套餐变更申请 | 客户维度 |
| **财务** | 账单、发票、对账、报表导出 | 财务域 + 脱敏客户信息 |
| **风控 / 合规** | 用户审核、黑名单、异常用量、审计查询 | 风控域 |
| **客服（只读偏多）** | 客户信息查询、用量解释、文档指引 | 只读 + 有限写（工单若上） |

**随手记已有**：管理员管理、用户与权限。  
**本文增补**：显式区分 **「平台员工角色」**（本文 RBAC）与 **「租户内角色」**（若存在，属 Account 控制台或未来租户 RBAC，**不在**运营后台首期）；租户侧 UI 真源见 **`views/account/README.md`**。

## 4. 分模块详细设计（按 §2 顺序）

以下为全文档 **「分」**：与 **§2 总表** 同一列序。每个 **一级模块** 下：**模块总述**（边界与目标）→ **二级页面** 表（页面职责 + **页内主要功能与说明**）。跨域技术子题以 **§4.x.x** 嵌套，避免打断 IA 叙述。

### 4.1 工作台

**模块总述**：运营后台 **默认落地页**；以 **少跳转** 暴露全站健康度、收入相关信号与待处理线索，并通过 **深链** 进入各业务域完成闭环。v1 以 **单页聚合** 为主；v2 再拆「待办中心」「快捷入口」独立子页（与 **§2 总表** 一致）。

| 二级页面 | 页内主要功能与说明 |
|----------|---------------------|
| **概览 KPI** | v1 核心：卡片式 KPI；时间范围切换；**只读**；每卡可跳转到 **监控 / 用量 / 客户 / 用户审核** 等模块详情。 |
| **待办中心（v2）** | 跨模块 **人工队列** 聚合（审核、风险、商务类）；筛选、（可选）指派 / 已读；条目跳转 **目标模块列表或详情抽屉**。 |
| **快捷入口（v2）** | 可配置卡片 / 链接区；常用操作 **一键直达**（新建租户、SKU、上下架、全站密钥检索等）；可按 **角色或个人收藏** 排序。 |

#### v1 / v2 分期（与 §2 对齐）

| 分期 | 范围 | 说明 |
|------|------|------|

### 4.5.1 双通道供应：原生研发商 API₁ 与云供应商 API₂（契约 · 适配 · 降级）

模型能力对外可由 **两类上游** 提供，协议与字段往往 **互不兼容**，平台需在网关侧做 **契约维护** 与 **入出转换**。

| 通道 | 主体示例 | 典型形态 | 本文档记法 |
|------|-----------|-----------|------------|
| **API₁** | 模型 **原生研发商**（如 DeepSeek、OpenAI 等） | 官方直连或官方文档约定的 HTTP / SSE 形态 | 每家研发商一套 **官方契约** |
| **API₂** | **云厂商 / 转售供应商**（如腾讯、阿里、火山及其他） | 各家云对同一逻辑模型再封装，路径、鉴权、返回结构可能与 API₁ 完全不同 | 每家供应商一套 **云侧契约** |

**JSON 真源（按官方统一维护）**

- 平台为 **「每个原生研发商」** 与 **「每个云供应商」** 分别维护 **可版本化的 JSON 配置**（字段名、枚举、鉴权头、错误码映射等），**对齐各自官方文档**；更新时走 **评审 → 发布 → 回滚**（可与「文档中心」或独立 **契约中心」v2 联动）。  
- **线路级**配置可 **引用** 全局 Profile 并 **覆盖** 局部字段（如 `base_url`、`model_id`、区域 endpoint）。  
- 运营后台 **「供应商」** 主数据以 API₂ 类为主；**「原生研发商」** 可作为独立主数据类型或挂在 **「模型管理」主数据 / 供应线路** 中维护（实现二选一，**数据模型上**须区分两类主体）。

**输入 / 输出适配（平台自研网关）**

- **输入**：租户调用 **平台统一入参**（canonical request）→ **InputMapper**（由 JSON + 代码生成或脚本）→ 转为当前线路要求的 **上游请求体 / 头 / URL**。  
- **输出**：上游 **原始响应**（API₁ 与 API₂ 结构可完全不同）→ **OutputMapper** → 收敛为 **平台统一出参**（canonical response），供租户 SDK、计量、日志一致消费。  
- **同一逻辑模型** 绑定不同线路时，**须分别配置** 输入映射、输出映射（可复用同一 Profile 的模板，但允许 per-route 覆盖）。

**降级策略（可编辑优先级）**

- 每条 **供应线路** 带 **优先级序号**（或主 / 备标签）；上游失败、超时、限流或返回可识别错误码时，按策略 **顺序尝试下一条**（与 §4.6「Fallback 链」一致）。  
- 运营后台提供 **可视化编辑优先级**、**临时置顶备路**（应急）；变更写入配置并 **审计**。

**ASCII（单租户一次调用在网关内的展开）**

```
  Canonical Request
         │
         ▼
  ┌──────────────┐     JSON Profile     ┌──────────────┐
  │ InputMapper  │◀────（研发商或云）────│ 配置存储     │
  └──────┬───────┘                      └──────────────┘
         │
         ├──▶ 线路1: API₁ 原生商 A  … 失败/超时
         │              │
         │              ▼ 降级（按优先级）
         └──▶ 线路2: API₂ 云厂商 B  …
                        │
                        ▼ raw
                 ┌──────────────┐
                 │ OutputMapper │──▶ Canonical Response
                 └──────────────┘
```

**业务目标**：风控与合规：密钥泄露、滥用时 **冻结 / 吊销**，并留审计。

**核心能力（与上表对应）**：全平台密钥检索（脱敏展示）、绑定客户与项目、创建人、**最近调用 IP / 地域**（若政策允许）、吊销与轮换记录。

**边界**：租户在 **Account 控制台** 自助创建 / 轮换密钥的产品与交互，见 **`apps/trinity-ai/src/views/account/README.md`**；运营后台本节仅描述 **跨租户治理**（冻结、吊销、审计、风控查询），两套能力 **文档分轨、产品分端**。

### 4.9 文档管理（运营侧）

| 二级页面 | 页内主要功能与说明 |
|----------|---------------------|
| **文档列表** | 类型（公开 / 客户专属 / 内部）、状态、负责人；筛选与批量操作。 |

| **发布与回滚** | 发布审批（可选）；一键回滚到历史版本；CDN 或静态站点刷新说明（工程）。 |
| **可见范围** | 公开、登录可见、指定客户组；与 **客户与合同** 组织 ID 绑定。 |

**业务目标**：控制 **对外文档** 的版本与可见性（公开文档 vs 某大客户专属 Release Note）。

**核心能力（与上表对应）**：Markdown / 外链、版本、发布、回滚；可选与 **模型能力说明** 双向链接。

### 4.11 用户管理（终端用户，非员工）

**模块总述**：面向 **注册到平台的外部用户**（非平台员工）：增长与风控平衡；与 **§4.7 客户组织** 下的主账号 / 成员关系可关联（数据模型级）。

| 二级页面 | 页内主要功能与说明 |
|----------|---------------------|
| **用户列表** | 检索、状态、所属客户组织；进入详情、审核、黑名单操作。 |
| **审核队列** | 待审列表、**单条**通过/拒绝；**批量**、**原因模板**、与 **工作台 / 待办 v2** 联动见 **§4.11.1（v2）**。 |
| **白名单** | 邮箱域名或手机号段放行规则；与注册流程耦合说明。 |
| **黑名单** | 全局拒绝登录 + 拒绝调用（策略可配置）；**审计必填落库**见 **§4.11.1（v2）**。 |
| **实名 / 企业认证** | 个人实名、企业执照上传与人工审核流；**材料下载权限控制**、与 **用户 / 组织主数据自动联动**见 **§4.11.1（v2）**。 |

**业务目标**：增长与风控平衡。

**随手记能力对齐**：白名单、用户列表、黑名单、审核、认证。

**建议补充状态机**：注册 → 邮箱验证 → **待审核** → 通过 / 拒绝；**企业认证** 子流程（上传执照、人工审核）。  
**黑名单**：全局拒绝登录 + 拒绝调用（策略可配置）。

#### 4.11.1 v1 / v2（或 v1.5）分期 — 工程与原型范围

与 **§4.1** 工作台「待办中心 **v2**」分工一致：**§4.11 主路径在 v1 跑通**；效率、合规与跨模块集成在 **v2（或 v1.5）** 加深，**依赖账号服务、对象存储、审计落库**后再上更划算。

| 分期 | 范围（能力） | 说明 |
|------|----------------|------|
| **v1（最小） / 当前原型** | **用户列表**（检索、状态、组织摘要）；**审核队列**上 **单条**通过/拒绝；**白名单 / 黑名单**规则维护；**实名 / 企业认证**以 **申请记录 + 单条审** 为主；用户与 KYC **允许分表维护**（不做强制主数据联动） | 可先 **前端原型 + `localStorage`** 或薄后端演示 **§4.11 主路径**；不把「批量、模板、待办聚合」绑死为 v1 验收项。 |
| **v2（或 v1.5）** | **审核原因模板**（拒绝/驳回可选原因、可统计）；**批量通过/拒绝**（事务、权限、操作人/批次留痕）；与 **工作台待办 v2** **深链与状态回写**；**KYC 通过后自动写用户 / 组织标签或认证态**；**认证材料下载权限**（水印、角色、审计）；**黑名单与敏感操作审计必填落库**、更完整 **审计检索与保留策略** | 属 **效率 + 合规 + 集成**；与 **§4.13** 导出审批、审计可衔接；**核心审核域仍在 §4.11**，增强不拆到其他一级模块除非产品另定。 |

**产品压 v1 范围时**：可将上表 **v2** 行整体列入「v2 清单 / 路线图」单独排期，而不改 §2 一级目录名称（「用户与认证」仍一处收口）。

## 5. 相对随手记的「增补清单」（便于评审勾选）

下列为随手记未逐条展开、但聚合平台后台**常见必备或强建议**项：

1. **计费与套餐**：SKU、配额、账单周期、调账与发票状态机（**§4.3**、**§4.7**）。  
2. **监控与告警**：供应商与线路健康、维护窗口（**§4.2**）。  
3. **路由与 fallback / 灰度**：与多供应商模型策略一致（**§4.6**）。  
4. **审计与合规**：操作留痕、导出审批、密钥治理（**§4.8**、**§4.13**）。  
5. **客户组织与子账号**（可选）：若 B2B 合同主体下有多项目团队（预留）。  
6. **工单 / 客服**（可选二期）：与「客服」角色联动。  
7. **Webhook 投递日志 / API 版本**（可选）：若平台对客户提供回调能力。  
8. **对账**：平台计量 vs 供应商账单导入（财务强需求时单独立项）。

## 6. 关键业务流程（文字级）

### 6.1 新模型上架（运营）

配置逻辑模型 → 绑定至少一条供应商线路并探测通过 → 设置刊例 / 内部成本 → **上架** → （可选）灰度指定客户 → 全量开放。

### 6.2 主备切换（应急）

监控发现主线路异常 → 运营在模型线路页 **提升备路优先级** 或启用 **fallback** → 公告 / 客户通知（可选）→ 事后补事件单与复盘字段。

### 6.3 客户开通与计费

创建客户组织 → 绑定合同与套餐 → 开通调用权限 → 用量入账 → 周期出账 → 开票流程结束。

### 6.4 流程 ASCII（与 §6.1–6.3 对应）

**6.1 新模型上架**

```
 [新建逻辑模型] --> [绑定线路+探测] --> [定价/成本] --> [上架]
                                                  |
                                                  +--> (可选) [灰度客户] --> [全量]
```

**6.2 主备切换（应急）**

```
 [监控告警] --> [运营打开模型线路] --> [提升备路优先级 / 启用 fallback]
        |                                        |
        |                                        +--> [公告/通知(可选)]
        +--> [记录事件 / 复盘字段]
```

**6.3 客户开通与计费**

```
 [创建客户组织] --> [合同+套餐] --> [开通调用] --> [用量持续入账]
                                                        |
                                                        v
                                              [周期出账] --> [开票结束态]
```
