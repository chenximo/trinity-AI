# 创建后端需实现页面与功能（分版本）

## 1. 目标与范围

本文面向 `trinityai-admin` 管理后台，基于当前已落地流程与数据表，设计一套可分阶段上线的页面与功能方案。目标是：

1. V1 先把必须能力做完整：基础数据可维护、日志可查询、可监控、后台用户可创建、可分配查询权限、所有后台写操作可审计入库。
2. V2/V3 在 V1 基础上增强风控运营、审批与自动化能力。

---

## 2. 现状能力盘点（作为设计基线）

### 2.1 已有后台接口（`/v1/admin/*`）

- 认证：`/v1/admin/auth/login`、`/v1/admin/auth/verify-credential-password`
- 模型管理：`/v1/admin/models`
- 供应商管理：`/v1/admin/providers`
- 平台密钥管理：`/v1/admin/platform-api-keys`（含密钥明文查看需 gate token）
- 用户 API Key 管理（运营视角）：`/v1/admin/api-keys`
- 监控：`/v1/admin/monitor/*`（SSE、token、spend、active users、stability）
- 文档：`/v1/admin/docs`

### 2.2 现有关键表（与页面设计强相关）

- 后台域：`admin_user`、`admin_doc_page`、`monitor_sse_sample`
- 模型与路由域：`model_profile`、`model_supply_route`、`platform_api_key`、`model_route_api_key_binding`、`model_family`、`model_supply_vendor`
- 用户与 API Key：`app_user`、`workspace`、`user_api_key`
- 用量与计费：`usage_event`、`billing_line`、`wallet_account`、`wallet_credit_ledger`、`wallet_debit_ledger`、`payment_order`
- 风控：`risk_rule`、`risk_ip_action`

---

## 3. 信息架构（页面总览）

建议后台导航按 6 个主模块组织：

1. **总览**
2. **基础数据**
3. **日志与查询**
4. **风控**（运行监控见侧栏「实时大盘」`ops/live`）
5. **后台用户与权限**
6. **系统审计**

---

## 4. V1（必须上线）页面与功能

V1 原则：先覆盖“可维护、可查询、可追溯”，不追求复杂审批流。

### 4.1 总览页（Dashboard）

**页面目标**
- 让管理员进入后台后 10 秒内判断系统是否健康、是否有异常消耗。

**核心功能**
- 监控卡片：当前 SSE、近 5m token、近 5m spend USD、活跃用户数、成功率。
- 快捷入口：模型管理、平台密钥、日志查询、风险规则、审计日志。
- 异常提示：最近 30 分钟风控命中量、网关 429 占比、上游 5xx 占比。

**数据来源**
- `monitor_sse_sample` + `usage_event` 聚合接口（现有 `/v1/admin/monitor/*`）。

---

### 4.2 基础数据模块

#### 4.2.1 模型管理页

**功能**
- 列表/搜索/分页：按状态、模型名、供应商筛选。
- 新建模型（基础属性、价格参数、能力标签）。
- 编辑模型（上下线、价格、排序、展示信息）。
- 关联路由信息查看（可跳转路由详情）。

**涉及表**
- `model_profile`、`model_family`、`model_origin_vendor`

#### 4.2.2 供应商管理页

**功能**
- 供应商新增/编辑/删除。
- 查看供应商下模型数量、可用路由数量。

**涉及表**
- `model_supply_vendor`（及其引用关系）

#### 4.2.3 平台密钥管理页（上游 Key）

**功能**
- 列表展示掩码密钥、状态、过期时间、绑定路由数。
- 新增/编辑/删除平台密钥。
- 明文查看需要二次口令 gate token（已与现有接口一致）。
- 校验规则：禁止无到期策略、禁止空用途说明。

**涉及表**
- `platform_api_key`、`model_route_api_key_binding`

#### 4.2.4 路由绑定页

**功能**
- 按模型查看路由与密钥绑定关系。
- 支持绑定/解绑、优先级调整、启停。
- 变更预览：展示“变更前后”路由结果摘要。

**涉及表**
- `model_supply_route`、`model_route_api_key_binding`

---

### 4.3 日志与查询模块（V1 必须）

#### 4.3.1 用量日志查询页

**功能**
- 查询维度：时间窗、模型、用户、API Key 名称、状态、workspace。
- 列表字段：请求 ID、模型、token、成本、时延、提供方、路由、创建时间。
- 支持导出 CSV（复用 usage 导出能力）。

**涉及表**
- `usage_event`

#### 4.3.2 计费流水查询页

**功能**
- 出账流水（扣费）查询：按 workspace/user/requestId/时间窗。
- 入账流水（充值）查询：区分 Stripe、手工充值。
- 账单联动：可跳转对应 `billing_line` 或 payment 订单。

**涉及表**
- `wallet_debit_ledger`、`wallet_credit_ledger`、`billing_line`、`payment_order`

#### 4.3.3 用户 API Key 查询页（运营）

**功能**
- 按用户/工作区/状态查询 API Key。
- 修改限额、重置窗口、过期时间；支持撤销。
- 展示策略版本与最近更新时间。

**涉及表**
- `user_api_key`（及策略窗口使用表）

---

### 4.4 风控与运行监控（V1 必须）

> **侧栏命名**：运行观测为独立单页 **「实时大盘」**（`admin-ops/`）；限流/拉黑配置与动作为折叠模块 **「风控」**（`admin-risk/`）。下文 §4.4.1 与 §4.4.2/4.4.3 分属两模块。

#### 4.4.1 实时监控页（侧栏「实时大盘」`ops/live`）

**V1 必做（验收 §9）**
- SSE open、token by model、spend、active users、stability 五类图表。
- 统一时间窗（默认 2h）与自动刷新（默认 15s）。
- 超窗限制（最大 24h）与缓存降级标记（stale）展示。

**原型增广（详设 §4.2 实时大盘 + 单页分区；mock 与 UI 已齐，后端可分期接）**

页面自上而下（与工作台分工见模块 README §1.1）：

1. **值班 KPI**：QPS、错误率、P99（建议 `GET /v1/admin/monitor/summary` 与五图同窗返回）。
2. **趋势 + 错误构成**：近 7 日调用量趋势、近 24h 错误码占比（`usage_event` 聚合）。
3. **模型调用占比**：按模型请求占比 + Token 占比对照（`GET .../model-share`）。
4. **五类监控时序**：与 §4.1 工作台监控卡同源，大盘展示同窗迷你柱（可折叠）。
5. **错误 / 延迟 Top**：错误码、路由、线路、次数 → 可下钻用量日志；维度 P95。
6. **供应商健康摘要**：线路成功率、延迟、容量、状态（与供应商模块、探测联动）。
7. **维护预告（只读）**：计划窗口与负责人；配置页可后续独立。

模块说明、分区顺序与 mock 键名：[`admin-ops/README.md`](../src/views/admin-ops/README.md) **§1.1、§2、§5**。

**涉及表**
- `monitor_sse_sample`、`usage_event`（及供应商线路健康、维护计划等扩展表，按实现分期）

#### 4.4.2 风控规则页（侧栏「风控」→ `risk/rules`）

**功能**
- 管理全局规则：窗口、rate limit 阈值、blacklist 阈值、TTL、启停。
- 规则校验：`rate_limit_threshold <= blacklist_threshold`。
- 规则历史变更记录可查看。

**涉及表**
- `risk_rule`

模块说明：[`admin-risk/README.md`](../src/views/admin-risk/README.md) **§1.1、§5**。

#### 4.4.3 风控动作日志页（`risk/action-logs`）

**功能**
- 查询 IP 被限流/拉黑记录：IP、动作类型、命中次数、起止时间、规则来源。
- 支持按 action_type、scope_type、IP、时间筛选。

**涉及表**
- `risk_ip_action`

---

### 4.5 后台用户与权限模块（V1 必须）

> 你要求“后台用户可创建、可分配查询权限”，V1 必须落地。

#### 4.5.1 后台用户管理页

**功能**
- 创建后台用户（用户名、初始密码、状态）。
- 重置密码、冻结/解冻、查看最近登录时间。
- 账号禁用后立即失效已签发后台 JWT。

**当前不足**
- 现有 `admin_user` 仅账号与密码哈希，需要补充状态与安全字段。

#### 4.5.2 角色管理页

**功能**
- 创建角色（如 `super_admin`、`ops_admin`、`finance_viewer`、`risk_viewer`）。
- 配置角色权限点（资源 + 动作）。
- 支持复制角色模板。

#### 4.5.3 权限分配页（用户-角色绑定）

**功能**
- 给用户分配多个角色，支持生效/失效时间。
- 支持“仅查询权限”模板（只允许读日志与监控，不允许写）。

#### 4.5.4 数据范围权限页（查询范围控制）

**V1 建议最小实现**
- 限制可查询的 workspace 范围（全部/指定列表）。
- 限制可查询模块（用量、计费、风控、审计）。
- 限制可查看敏感字段（例如平台密钥明文、用户隐私字段）。

---

### 4.6 系统审计模块（V1 必须）

> 你要求“后台操作每次改了什么需入库”，V1 必须全覆盖写操作审计。

#### 4.6.1 操作审计日志页

**功能**
- 查询所有后台写操作：谁、何时、改了什么、改前/改后、来源 IP、traceId。
- 支持按模块/动作/操作者/资源ID/时间筛选。
- 支持导出（脱敏后导出）。

#### 4.6.2 安全事件页

**功能**
- 登录失败、越权访问、credential gate 验证失败、批量敏感操作告警。

---

## 5. V1 必须新增的数据模型（支撑权限与审计）

### 5.1 RBAC 相关表（新增）

1. `admin_role`：角色定义  
2. `admin_permission`：权限点定义（如 `monitor:read`、`risk:write`）  
3. `admin_user_role`：用户与角色绑定  
4. `admin_role_permission`：角色权限绑定  
5. `admin_data_scope`：用户/角色的数据范围（workspace 与模块）

### 5.2 审计相关表（新增）

1. `admin_audit_log`
   - 核心字段：`operator_admin_user_id`、`module`、`action`、`resource_type`、`resource_id`、`before_json`、`after_json`、`changed_fields_json`、`request_id`、`ip`、`ua`、`created_at`
2. `admin_security_event`
   - 核心字段：`event_type`、`severity`、`subject`、`detail_json`、`request_id`、`ip`、`created_at`

### 5.3 `admin_user` 扩展字段（建议）

- `status`（active/frozen）
- `last_login_at`
- `pwd_updated_at`
- `created_by`
- `updated_by`

## 8. V2 版本（运营增强）

### 页面新增

1. **发布中心**
   - 模型配置/价格变更发布单、影响预览、回滚入口。
2. **告警中心**
   - 监控阈值配置（SSE 高水位、429 突刺、上游 5xx 突刺）与通知策略。
3. **批量任务中心**
   - 批量 API Key 策略调整、批量冻结/解冻、批量标签。
4. **报表中心**
   - 按日/周/月的成本、收入、毛利趋势与 TopN 用户/模型。

### 能力增强

- 引入审批流（高危变更需二次审批）
- 增强数据范围到“组织/租户/业务线”
- 审计日志差异可视化（字段级 diff）

---

## 9. V3 版本（治理与自动化）

### 页面新增

1. **策略编排中心**
   - 风控、限流、计费策略可视化编排。
2. **自动化运营中心**
   - 自动封禁、自动解封、自动告警升级、自动回滚策略。
3. **审计分析中心**
   - 审计事件关联分析、异常行为画像、追踪链路图。

### 架构演进

- 后台 IAM/RBAC 独立服务
- 审计日志入仓（OLAP）支持长期分析
- 规则配置与执行引擎分离

---

## 10. V1 验收标准（与你要求逐条对齐）

1. **基础数据可维护**  
   模型、供应商、平台密钥、路由绑定、文档均可在后台维护并可追溯。

2. **日志可查询**  
   用量日志、计费流水、风控动作、后台操作审计均可多维检索。

3. **可监控**  
   SSE/token/spend/active users/stability 五类监控可视化可用，支持时间窗与刷新。

4. **后台用户可创建**  
   可创建/冻结/重置后台账号，支持角色分配。

5. **可分配查询权限**  
   支持只读角色、模块级查询权限和 workspace 数据范围限制。

6. **后台操作每次改了什么需入库**  
   所有写操作写入 `admin_audit_log`，包含 before/after 与 changed fields。

---

## 11. 建议实施顺序（V1 内部迭代）

1. 第 1 周：RBAC 最小模型 + 后台用户管理页 + 审计中间件
2. 第 2 周：基础数据页完善（模型/供应商/平台密钥/路由绑定）
3. 第 3 周：日志查询页（用量/计费/风控）+ 权限细化
4. 第 4 周：联调压测与验收（权限回归、审计完整性、监控稳定性）
