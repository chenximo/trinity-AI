### 资深 PM 能力（进阶 · 按需补齐）

> **定位**：上树 **【高级】** 覆盖 0→1 文档与交付；**【资深】** 补 **决策深度与影响力**。不强制一次写全——你在实际工作中填一项，就在「我的落点」链出或改 `【待补充】` 为真源路径。

| 能力 | 资深 PM 通常要回答什么 | 建议落点 | 状态 |
|------|------------------------|----------|:----:|
| **假设 & 证伪** | 哪些 belief 错了就 pivot？怎么验？何时 kill？ | L0 附录 · 立项评审附件 | 待补充 |
| **商业粗算** | 定价带 · COGS/采集成本 · LTV 量级 · 待拍板项 | §1.1 · [AI API 商用计费](ai-api-platform/commercial-billing) · [GEO 计费](geo/geo-billing-industry) | 草案 |
| **GTM / 销售包** | 谁买单 · 试用/POC 标准 · 什么条件下不商用 | §1.1 试用与商用范围 · 定价页 PRD | 待补充 |
| **Discovery 机会池** | 访谈→机会→实验→是否进 roadmap（≠ 功能 backlog） | 各线 `product-backlog` 旁或独立 YAML/飞书 | 待补充 |
| **Portfolio 取舍** | 多产品线资源争用 · build vs buy · 技术债 vs 新线 | [站点总览](index.md) · 各线 L0 §0.2 楔子 | 待补充 |
| **向上 & 跨团队** | 管理层 3 页叙事 · 资源/人力争取 · 无汇报线协调 | 月报模板 · 评审 deck（不进客户侧手册） | 待补充 |
| **领域判断力** | 「外行 PM vs 懂行 PM」怎么答 | 各线 PM 补充（例 [GEO](geo/pm-handbook)） | GEO 已有 |

**自检（资深门禁 · 立项 / 商用前可选）**：

- [ ] 至少 3 条 **可证伪假设** 已写清验证方式与截止日期  
- [ ] **商业粗算** 有数量级（不必精算），待拍板项已标出  
- [ ] **计费三处口径**（对外 docs · 用户控制台 · 运营后台）已对齐  
- [ ] **商用验收**：充值/订阅 → 调用或采集 → 扣费可见 → 可对账  
- [ ] **P0 闭环** 能对应一个可演示、可收钱的样本客户场景  
- [ ] **不做清单** 已与管理层或评审会对齐  
- [ ] Discovery 产出的 **机会** 与工程 **需求** 未混在同一 backlog

### 1.1 商业化与计费（PM 必备模块）

> **定位**：工作树独立一枝，**跨 L0–L3**；不是只写 L0 §0.3 一行就结束。B2B / 按量 / 采集型产品 **商用前必做**。

**PM 要回答的 5 个问题**：

| # | 问题 | 典型落点 |
|---|------|----------|
| 1 | **卖什么单位** | 订阅档 · Credits · 品牌数 · prompt 数 · Key · 席位 |
| 2 | **什么行为触发成本（COGS）** | 每次 API · 每次采集 · 每平台 · 存储天数 |
| 3 | **套餐怎么限才毛利成立** | 配额表 · 频率 · 启用数 · 历史保留 |
| 4 | **用户在哪看用量、在哪升级** | 定价页 · 账户设置 · 控制台 `#usage` |
| 5 | **本阶段商用卖到什么边界** | 6.30 范围 · 试用上限 · 明确「不卖」 |

**各线计费专文（真源 · 逐步补齐）**：

| 产品线 | 专文 | 状态 |
|--------|------|:----:|
| **GEO** | [业界计费与监测周期](geo/geo-billing-industry) | ✅ 草案 |
| **AI API 聚合** | [商用计费与充值](ai-api-platform/commercial-billing/) · [全球支付](ai-api-platform/commercial-billing/global-payment) | ✅ 草案 |
| **AI Cloud** | （待建） | 待建 |
| **Agent 二期** | 随 SDK/Web 立项再写 | 规划期 |

**商用前 checklist（PM · 与研发/运营对表）**：

- [ ] L0 §0.3 定价带与套餐维度已写，**待拍板**项已标出  
- [ ] 计费专文（或 §0.3 附录）有 **COGS 数量级** 与控成本杠杆  
- [ ] **计量维度表**：用户可见文案 ↔ 后台字段 ↔ 成本来源 — 三列对齐  
- [ ] 定价页 / 账户页 / 运营 billing **同一套单位**（不出现「Credits」与「元」混用无换算）  
- [ ] 试用边界写清（额度 · 天数 · 到期降档行为）  
- [ ] 商用验收一句话可测（见上树 **商用验收闭环**）  
- [ ] 隐私 / 条款 / 计量说明 **不矛盾**（链 [legal](legal/)）

**与角色分工**：PM 定 **维度与口径** · 研发做 **计量真源与限流** · 运营/商务做 **合同价与特例** · 财务/法务做 **主体与条款**。

## 二、研发协作规范

### 2.1 需求评审流程

```
PM 起草 PRD / 规格 → 小组内部评审 → 技术可行性评估 → 正式评审会 → 确认排期
```

**评审会必问清单**：

- 这个功能解决 **谁的** 什么问题？
- **不做** 会怎样？
- 有没有 **更简单** 的做法？
- 数据从哪里来？（新采集还是复用现有数据）
- 边界条件是否穷举？
- 异常情况如何表现？
- 什么叫「做完」？（验收标准是否明确）

### 2.2 需求优先级

| 级别 | 含义 |
|:----:|------|
| **P0** | 必须做：不做产品不可用 |
| **P1** | 应该做：影响体验或竞争力，本周期内完成 |
| **P2** | 可以做：锦上添花，有资源再做 |
| **P3** | 暂不做：明确本期不做，放入需求池 |

**排期真源**：L0 §0.4 / 产品设计分析中的 P0/P1/P2；Sprint 周次为 **示例节奏**，不入设计真源。

### 2.3 迭代节奏（示例）

按产品线替换模块名；GEO 示例如 [GEO · PM 补充 · 迭代节奏](./geo/pm-handbook#二迭代节奏示例)。

```
Sprint 1–2：  核心主链 · 可演示最小闭环
Sprint 3–4：  差异化能力 · 诊断 / 规则 / 对比
Sprint 5–6：  验证闭环 · 用户反馈
─────────────── P0 可卖 / 可验收
后续 Sprint： P1 加深 → P2 规模化
```

## 三、用户验证机制

### 3.1 验证节奏

| 阶段 | 验证方式 | 频率 | 对象 |
|------|---------|------|------|
| 需求阶段 | 用户访谈 | 按需 | 3–5 个潜在用户 |
| 设计阶段 | 原型测试 | 每功能 | 2–3 个目标用户 |
| 开发阶段 | 内部 Dogfood | 持续 | 团队 |
| 上线前 | Alpha 内测 | 1–2 周 | 5–10 个种子用户 |
| 上线后 | 数据 + 反馈 | 持续 | 全体用户 |

### 3.2 种子用户（通用原则）

- 正在经历 **真实痛点**，而非「帮忙看看」
- 对早期不完美有容忍度，愿意给 **可执行反馈**
- 所在场景与 L0 **P0 用户** 一致
- 产品线专属画像与渠道 → 各线 PM 补充（例 [GEO 种子用户](./geo/pm-handbook#三种子用户)）

## 四、竞品跟踪机制

### 4.1 竞品分级

```
Tier 1 — 直接竞品：每周跟踪
Tier 2 — 相关竞品：每月跟踪

```

## 五、合规前置（通用框架）

产品设计阶段与法务 / 后端对齐；**产品线特有红线** 写在各线专文。

| 合规项 | 设计要求 | 责任方 |
|--------|---------|--------|
| 用户数据隐私 | 配置与行为数据加密、最小采集 | PM + 后端 |
| 第三方 ToS | 调用 / 采集不违反平台服务条款 | PM + 法务 |
| 内容与广告合规 | 生成 / 建议内容不含虚假、极限词、未授权承诺 | PM + 法务 |
| GDPR / 个保法 | 收集、删除、导出权 | PM + 法务 |
| 知识产权 | 产出物版权归属明确 | PM + 法务 |
| 可审计性 | 关键操作留日志、可回溯 | PM + 后端 |

## 六、PM 周常清单

```
每周必做：
□ 产品数据看板（注册、活跃、留存、付费 / 调用量等）

□ 与研发同步：进度、阻塞项
□ 与至少 1 个用户交流

每两周必做：
□ Sprint Review + Retro
□ 路线图 / roadmap 与 L0 优先级对齐

每月必做：
□ 用户研究小结（本月新发现）
□ 竞品深度分析（或更新 Tier 1 摘要）
□ 下月 OKR
□ 向管理层汇报产品进展
```
