# 需求分析助手 · 产品需求分析

> **目录**：[智能助手工具](../) · 需求分析助手  
> **性质**：**内部自用**效能工具（非对外产品、不走完整 L0/L1/L2 手册树）。  
> **用户**：PM、技术支持（收 **钉钉群** 反馈，整理进需求池 / Bug 表）。  
> **状态**：路线 A 落地中（**钉钉**）— 等应用发布审批  
> **待办**：[待办清单](./todo) · **工程**：[路线 A 落地文档](./route-a) · `trinity-AI/tools/dingtalk-requirement-inbox/`  
> **关联**：[产品经理工作手册 · §1.2 产品待办池](../../产品经理工作手册#12-产品待办池多渠道需求池) · [飞书 Bug 表](https://qcn81yhei1l2.feishu.cn/sheets/PjnVs7bmphodaKtOkkycpvxmnne) · [产品待办池](../../ai-api-platform/#产品待办池)

## 1. 背景与问题

| 现状 | 痛点 |
|------|------|
| Bug、需求、排障讨论集中在 **固定钉钉群** | 长对话多，PM 需通读才能摘录；群内 **可直接贴截图** |
| 真源分散：飞书 Bug 表、`product-backlog.yml`、周计划 | **缺「群聊 → 结构化候选」** 一层，易漏、重复劳动 |
| 历史上用过 TAPD | 现以手册 + 钉钉讨论 + 飞书 Bug 表为主；**收件箱不另建库** |

**典型场景（验收样本）**：群内 30 分钟 Codex / CC Switch 排障对话 → 应摘录为 **3～4 条** Bug/需求，而非 30 条消息或一条模糊摘要。

---

## 2. 产品一句话

**在固定钉钉需求群里，PM @需求助手 整理一段对话，自动拆成若干条可确认的 Bug/需求，写入钉钉「需求收件箱」多维表。**

---

## 3. 目标与非目标

### 3.1 P0 目标

- 支持 **线程级** 整理（一段对话 → 多条候选）
- 每条含：**类型、标题、摘要、建议模块、原消息链接、提出人**
- 消息含图时标 **`含截图`**（钉钉群内可直接贴图；P0 识别图片消息，P1 可附图片链接）
- 全部先进收件箱，状态 **待确认**；**PM 确认后** 再人工转 Bug 表 / backlog

### 3.2 明确不做（P0）

| 不做 | 原因 |
|------|------|
| 自动写入 Bug 表 / `product-backlog.yml` | 避免 AI 污染真源 |
| 全群消息自动收录 | 噪音大；只用 **@整理** 触发 |
| 自建业务数据库 | 进线真源 = **钉钉多维表** |
| 替代周会排期 | 仍人工进 `week-progress` |
| 对外商用 / 多租户 | 内部狗食 |

---

## 4. 用户与用法

| 角色 | 动作 |
|------|------|
| 反馈方（支持、研发、客户成功等） | 在 **固定需求群** 按群规发消息（可长对话、可附图） |
| PM | 对话告一段落后发送 **`@需求助手 整理`**（或回复该话题后 @整理） |
| PM | 打开 **需求收件箱** 表，核对 → 改状态 → 周会转入 Bug 表 / [产品待办池](../../ai-api-platform/#产品待办池) |

**群规（建议写入群公告）**：

1. 本群用于发布需求、Bug、体验反馈；闲聊请去其他群。  
2. 长讨论结束后由 PM **@需求助手 整理**，不要期望每条消息自动进表。  
3. 勿在群内贴 API Key、客户隐私；敏感信息用文字概括即可。

---

## 5. 端到端流程

```text
固定钉钉群（需求发布）
        │
        ▼  @需求助手 整理
        │
   整理服务（路线 A · 钉钉机器人 Stream + 薄后端）
   · 拉取会话近期消息（Chat.Message.Read / 群消息记忆）
   · LLM 摘录 → JSON 候选列表（含图片消息识别）
   · 写入钉钉多维表（状态=待确认）
        │
        ▼
   钉钉「需求收件箱」多维表  ← 进线真源
        │
        ▼  PM 确认 / 编辑 / 忽略
        │
   ┌────┴────┐
   ▼         ▼
飞书 Bug 表   product-backlog.yml（note 含收件箱编号 + 群链）
   │         │  ← 仍手工同步，不自动写
   └────┬────┘
        ▼
   week-progress（周会人工排期）
```

## 6. 实现方案

### 6.1 技术选型（路线 A）

| 组件 | 选型 | 说明 |
|------|------|------|
| 编排 | **薄后端服务**（FastAPI） | 见 [路线 A 落地文档](./route-a) |
| 通道 | **钉钉企业内部应用 · 群机器人** | Stream 模式收 @（无需公网回调） |
| 理解 | **Trinity API**（OpenAI 兼容） | 摘录 + 分类 |
| 存储 | **钉钉多维表**（钉钉文档） | 需求收件箱；不建 MySQL |
| 确认 | **表内改状态** | P0 用 `sessionWebhook` 文本回复 |

### 6.2 触发口令

| 口令 | 行为 |
|------|------|
| `@需求助手 整理` | 以当前会话上下文整理（默认：最近 **50 条** 或 **2 小时** 内消息） |
| `@需求助手 整理 昨天` | P1：按时间范围扫（可选） |

**P0 仅实现第一种**；时间窗口通过环境变量配置。

### 6.3 处理步骤

1. **触发**：钉钉群内 @机器人，正文含 `整理`  
2. **拉消息**：`Chat.Message.Read`（聊天消息记忆）或群历史 API，默认最近 **50 条 / 2 小时**  
3. **预处理**：拼成 `messages[]`（sender、time、text、has_image、message_link）；**识别 `picture` 类型消息**  
4. **LLM**：系统 Prompt + 用户消息 JSON → 输出 **严格 JSON**（见 §7）  
5. **写表**：每条 candidate → 钉钉多维表新增记录  
6. **回复**：群内文本：`已整理 N 条，请在「需求收件箱」确认（待确认）`

### 6.4 钉钉应用权限（开通 checklist）

- [ ] 机器人接收消息（Stream：`/v1.0/im/bot/messages/get`）  
- [ ] **`Chat.Message.Read`**（聊天消息记忆，用于拉近期群聊；需向钉钉申请灰度）  
- [ ] 钉钉多维表：新增记录  
- [ ] 机器人加入 **固定需求钉钉群**

## 7. 摘录规则与 LLM 输出

### 7.1 类型定义

| type | 含义 | 下游 |
|------|------|------|
| `bug` | 可复现缺陷、报错、功能不可用 | 确认后 → Bug 表 |
| `feature` | 新能力、体验改进 | 确认后 → product-backlog |
| `doc` | 文档、配置、最佳实践缺口 | 确认后 → backlog 或文档任务 |
| `question` | 已当场解答的咨询 | 一般 **忽略** 或仅归档 |
| `noise` | 闲聊、致谢、表情 | **不写入表** |

### 7.2 输出 JSON Schema

```json
{
  "session_summary": "两三句话概括本场对话在解决什么",
  "candidates": [
    {
      "type": "bug",
      "title": "一句话标题",
      "summary": "现象 + 环境 + 当前结论（不含密钥）",
      "module_suggestion": "平台侧 · 统一 API 基座",
      "reporter": "吕超群",
      "has_screenshot": true,
      "message_link": "https://...",
      "confidence": "high"
    }
  ],
  "ignored_reason": "已过滤的闲聊说明（可选）"
}
```

工程真源：`trinity-AI/tools/feishu-requirement-inbox/schema.json`

### 7.3 系统 Prompt 要点

- 你是 Trinity 内部 **需求整理助手**，输入是一段飞书群消息列表。  
- **任务**：输出 `session_summary` + `candidates`；一条真实问题一条 candidate；同一根因合并。  
- **必须提取**：断流/报错/不兼容/无法使用的现象；配置体验差可标 `feature` 或 `doc`。  
- **必须忽略**：纯致谢、表情、模型选型闲聊（除非引出产品问题）。  
- **module_suggestion**：在「用户侧 / 平台侧 / 运营 / 文档站 / 内部工具」内猜最接近模块。  
- **禁止**：编造未出现的信息；输出 API Key。  
- **has_screenshot**：任一条相关消息含图则为 true。  
- 只输出 JSON，不要 markdown 包裹。

工程真源：`trinity-AI/tools/feishu-requirement-inbox/prompt.md`

### 7.4 固定样本 · 期望摘录（Codex 断流 case）

输入：吕超群 × Codex × CC Switch × Opus 断流对话（约 30 分钟）。

**至少应产出**：

| # | type | title（示例） |
|---|------|----------------|
| 1 | bug | Codex + CC Switch + Opus：输出一段后断流 |
| 2 | bug | Trinity API 拒绝参数 `thinking`（Codex 外接） |
| 3 | doc / bug | CC Switch 开启路由后无法拉取模型列表 |
| 4 | doc / feature | Codex 直连 Trinity 配置复杂、 onboarding 需最佳实践 |

**不应产出**：单独一条「用户觉得 codex 不好用」或 20 条重复消息。

## 8. 钉钉「需求收件箱」多维表结构

> 在 **钉钉文档** 新建多维表，名称建议：**Trinity · 需求收件箱**。

| 列名 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| 创建时间 | 日期 | ✅ | 自动 |
| 批次 ID | 文本 | ✅ | 同一次「整理」共用，如 `20260608-1430-a3f2` |
| 类型 | 单选 | ✅ | Bug / 需求 / 文档 / 咨询 / 忽略 |
| 标题 | 文本 | ✅ | |
| 摘要 | 多行文本 | ✅ | |
| 建议模块 | 文本 | | PM 可改 |
| 提出人 | 文本 | | |
| 含截图 | 复选框 | | LLM 判断 |
| 原消息链接 | URL | ✅ | 会话或首条相关消息 link |
| 会话摘要 | 多行文本 | | 本场 `session_summary` |
| 状态 | 单选 | ✅ | 待确认 / 已确认 / 已转 Bug / 已转 backlog / 忽略 |
| 跟进人 | 人员 | | PM 指定 |
| 截止 | 日期 | | 梳理或跟进 deadline |
| 备注 | 多行文本 | | 转 Bug 单号、backlog 链接等 |

**状态流转**：

```text
待确认 → 已确认 → 已转 Bug / 已转 backlog
待确认 → 忽略
```

## 10. 验收标准（P0 上线）

- [ ] 固定需求群已接入机器人；仅 **@整理** 触发  
- [ ] **Codex 断流样本** 整理出 ≥ **3** 条有效 candidate，PM 认可 ≥ **2** 条可直接采纳  
- [ ] 每条写入收件箱：类型、标题、摘要、链接、状态=待确认  
- [ ] 闲聊不写入（或写入后类型=忽略，由 PM 批量清）  
- [ ] 试点 **1 周** 内有效入库 ≥ **10** 条（PM 确认）  
- [ ] 未确认行 **从未** 出现在 Bug 表 / backlog  

---

## 11. 风险与约束

| 风险 | 缓解 |
|------|------|
| LLM 漏提/误提 | 人工确认；保留原消息链接 |
| 群聊含敏感信息 | 内部群 only；Prompt 脱敏；真源以链接为准 |
| 服务/钉钉不可用 | 回复失败提示；PM 可手工建表行 |
| 线程边界不准 | P0 固定「最近 N 条」；P1 支持回复链 |
