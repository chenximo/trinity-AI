# 跨境投放 Agent · 产品预研分析

> **性质**：产品预研 / 内部试点方向讨论稿，**非**对外承诺、**非**落地实施步骤。  
> **背景**：公司有跨境投放相关业务；待 **Agent SDK + 可视化 Web** 成熟后，可作为 **Dogfood（自用试刀）** 场景验证底座。  
> **关联**：[产品定位与架构](./positioning-and-architecture) · [行业现状与跨境落地](./industry-cross-border) · [Agent SDK · 产品设计](./agent-sdk-product-design)

---

## 一、预研要回答什么

| 问题 | 本页结论（摘要） |
|------|------------------|
| 跨境投放 Agent 是什么？ | 把「多平台投放 SOP」从人工盯盘，逐步固化为 **可配置、可观测、可私有化** 的智能体能力；简单形态先做 **辅助决策**，复杂形态才是 **多 Agent 全自动闭环**。 |
| SDK + 可视化能带来什么？ | SDK：**多轮推理、调广告/数据工具、状态记忆、工作流编排**；Web：**运营配流程、看日志、用模板演示**。网关：**多模型、多语种、计量**。 |
| 方向有没有问题？ | **有业务、有付费语境、能验证 Trinity 差异化**（私有化 + 双交付），方向成立。 |
| 第一刀试什么？ | **不建议一上来就做「全自动投放闭环」**；建议 **分片试点**（见 §六），投放业务作 **北极星场景**，先期用 **客服 / 日报 / 合规抽检** 等较轻 Agent 验 SDK。 |

---

## 二、场景定义：简单 Agent vs 复杂正式 Agent

预研里常混谈的两层，需分开：

| 形态 | 典型能力 | 是否「长任务 / 多 Agent」 | 与 SDK 成熟度 |
|------|----------|---------------------------|---------------|
| **简单 Agent** | 多语种文案生成、投放数据问答、异常摘要、合规一句提示 | 否 · 分钟级 | **P0** 循环 + tools 即可试 |
| **中等 Agent** | 监控告警 → 人工确认 → 执行调价/关停；跨天线索跟进 | 轻度长任务 | **P1** 记忆 + 定时触发 |
| **复杂正式 Agent** | 多平台自动搭计划、7×24 ROI 自优化、投放↔客服↔合规多 Agent 协同 | 是 · 小时～天级 | **P2+** 工作流 + Web + 外部连接器 |

**跨境投放的「终局想象」属于复杂正式 Agent**；公司试刀时应先锁定 **可验收的一片**，再往上叠。

### 2.1 业务链路（目标态 · 讨论用）

```text
市场洞察 / 竞品素材
    → 多语种素材生成与本地化
    → 各平台广告计划创建（Meta / Google / TikTok 等）
    → 7×24 花费与 ROI 监控
    → 异常告警（超支、低效、违规素材）
    → 人工或自动执行：缩预算 / 换素材 / 扩量
    → （可选）联动客服 Agent 承接询盘
```

### 2.2 三类角色痛点（为何值得做）

| 角色 | 痛点 | Agent 价值假设 |
|------|------|----------------|
| 投放运营 | 多平台后台切换、跨时区盯盘、经验难复制 | SOP 固化、告警与建议、减重复劳动 |
| 管理层 | ROI 不透明、决策靠周报 | 实时摘要、可审计的执行记录 |
| 技术 / 合规 | API 异构、素材违规风险 | 统一 Tool 协议、合规校验节点 |

## 三、我方双能力能承接什么（预研视角）

不展开实现细节；只判断 **方向是否闭合**。

### 3.1 Agent SDK（执行底座）

| 能力块 | 在投放场景中的作用 |
|--------|-------------------|
| **多轮循环 + tools** | 读账户数据、拉报表、调「创建/更新广告」类 API（经封装）、生成文案 |
| **记忆** | 记住活动 ID、昨日策略、待人工确认项；跨天续跑 |
| **工作流** | 串起「监控 → 判断 → 人工审批 → 执行」；后期多 Agent 分工 |
| **私有化交付** | 账户与营收数据走客户/自有环境，调 Trinity 网关出模型 |

### 3.2 可视化 Web（孵化器）

| 能力块 | 在投放场景中的作用 |
|--------|-------------------|
| **Agent / 工作流配置** | 运营拖拽或表单配置投放 SOP，无需写代码 |
| **调试与观测** | 单步跑、看 tool 调用与记忆变量，修正策略 |
| **模板** | 「多语种素材流」「日报监控流」等 **演示级样板**，加速内部与客户 POC |
| **知识库** | 品类话术、平台政策摘要、历史高转化文案 |

### 3.3 多模型聚合网关（已有）

| 能力 | 在投放场景中的作用 |
|------|-------------------|
| 多模型路由 | 文案用强模型、分类/摘要用轻模型 |
| 多语种 | 同一套 Agent 切换区域语言与风格 |
| 计量与追踪 | 按客户/活动对账，Request-Id 贯穿调试 |

```text
Web 配置台（孵化投放样板）
        ↓
薄适配层
        ↓
Agent SDK（循环 · 记忆 · 工作流）
        ↓
Trinity 网关 + 广告/数据 Tool（自研或 ISV）
```

## 四、与公司业务的契合（Dogfood 价值）

公司有跨境投放相关业务的预研价值：

| 维度 | 说明 |
|------|------|
| **真实数据与 API** | 不必虚构场景；可验证 Tool 封装、权限、审计是否够用 |
| **内部容忍度** | 先辅助后自动，允许「建议 + 人工确认」模式 |
| **产品叙事** | 对外可讲「自家跑通过」的投放 Agent 样板，比纯 PPT 可信 |
| **风险可控** | 用子账号 / 小预算测试，避免一上来全自动烧预算 |

**预研建议**：与业务方对齐 **一片最小试点**（见 §五），书面约定 **哪些动作必须人工确认**（如：改预算、上新计划）。

## 五、建议的试点切片（由易到难）

在「跨境投放」北极星下，拆成可试的 Agent **切片**，而不是一次做全链路。

### 切片 A · 投放数据日报 / 异常解读 Agent（推荐 **最先试**）

- **做什么**：每日拉各平台花费、ROI、异常波动 → 生成结构化摘要 + 追问问答。
- **为什么先它**：只读为主、风险低；验 P0（循环 + tools + 网关多模型）即可。
- **成功标准**：运营愿意每天打开看；摘要准确率业务方认可。

### 切片 B · 多语种广告文案 + 合规提示 Agent

- **做什么**：输入商品与目标市场 → 出多版文案 + 敏感词/政策提示 → **人工选用**。
- **为什么**：不直接改账户；验多语种路由与知识库；和投放强相关。
- **成功标准**：采纳率、改稿轮次下降。

### 切片 C · 监控告警 → 人工确认 → 执行 Agent

- **做什么**：ROI 跌破线 → 告警 → 运营在 Web/IM 点确认 → Agent 调 API 缩预算或暂停。
- **为什么**：引入「半自动」；需 P1 记忆（记住待确认单）+ 审批节点。
- **成功标准**：误操作为零；单次处理时长下降。

### 切片 D · 简化工作流（素材 → 审 → 上架）

- **做什么**：线性 SOP，不追求 7×24 自优化。
- **为什么**：验 P2 工作流 + Web 画布；仍是投放域，但复杂度可控。
- **成功标准**：一条活动从配置到上线，人力步骤减少可量化。

### 切片 E · 复杂正式 Agent（多 Agent 全闭环）

- **做什么**：§2.1 全链路、多 Agent 协同、长任务。
- **为什么放后**：依赖多平台连接器成熟度、组织信任；属 **愿景**，非第一刀。
- **成功标准**：预研阶段 **不定义**；SDK/Web 成熟后再立项。

## 六、试点路线建议（SDK / Web 成熟度对齐）

| 阶段 | Trinity 能力（规划） | 建议试的 Agent | 与投放关系 |
|------|----------------------|----------------|------------|
| **试刀 1** | P0 SDK + 网关 | **切片 A** 日报解读；或 **多语种客服**（见 §七） | 投放域只读 |
| **试刀 2** | P0 + 知识库/RAG | **切片 B** 文案 + 合规提示 | 投放上游 |
| **试刀 3** | P1 记忆 | **切片 C** 告警 + 人工确认执行 | 投放核心 |
| **试刀 4** | P2 工作流 + Web MVP | **切片 D** 线性投放 SOP | 可视化孵化器 |
| **标杆** | P2+ / 生态 | **切片 E** 或对外 POC | 公司全业务试刀 |

**结论**：**跨境投放是北极星，不是第一刀。** 第一刀选 **切片 A 或 B**，业务同属投放线，SDK 压力小、见效快。

## 七、其他可优先试的 Agent（对照推荐）

若希望 **更快验证 SDK** 或 **并行降低投放资金风险**，可考虑：

| 候选 Agent | 复杂度 | 与跨境/公司业务 | 建议 |
|------------|--------|-----------------|------|
| **多语种售前客服** | 低～中 | 高 · 投放引流后承接询盘 | ⭐ 与投放 **串联** 的第二试点；P0 即可起步 |
| **商品页 / 广告合规抽检** | 低 | 高 · 出海刚需 | ⭐ 适合切片 B 合规节点单独试 |
| **投放数据日报解读** | 低 | 高 · 纯投放 | ⭐ **首选试刀**（切片 A） |
| **多语种广告文案生成** | 低 | 高 | ⭐ 首选试刀备选（切片 B） |
| **库存 / 物流查询助手** | 中 | 中 · 供应链 | 验 tools 多样性；与投放弱耦合 |
| **全自动多平台调价** | 高 | 高 | ❌ 不宜第一刀 · 资金与合规风险大 |
| **多 Agent 投放+客服+合规闭环** | 很高 | 高 | ❌ 复杂正式 Agent · 作终局 |

**推荐组合（公司内部试）**：

1. **试刀 1**：投放 **日报/异常解读**（或 **多语种客服** 若客服诉求更急）  
2. **试刀 2**：**文案 + 合规提示**（与投放同域）  
3. **试刀 3**：**告警 + 人工确认执行**（真正碰投放 API 写操作）  
4. **试刀 4 起**：Web 可视化配 **线性投放 SOP**，再谈多 Agent

## 八、依赖与边界（预研级）

| 类型 | 说明 |
|------|------|
| **Trinity 自建** | SDK 循环/记忆/工作流、Web 配置与调试、网关与计量 |
| **业务 / 技术共建** | Meta、Google、TikTok 等 **广告 API 连接器**；账户权限与 OAuth |
| **不建议 Trinity 独扛** | 投放优化算法、全量各国法规库、替代资深投手的「全自动策略」 |
| **人工兜底** | 凡涉及 **花钱、上线、封号风险** 的操作，预研默认 **人工确认** |

## 十、一句话结论

**跨境投放 Agent 方向正确，且与公司业务契合，适合作为 SDK + 可视化成熟后的标杆 Dogfood 场景；但预研与试刀应分片推进——先用「日报解读 / 文案合规」等简单 Agent 验证底座，再用「告警 + 人工确认」接触真实投放写操作，最后才做 Web 编排与多 Agent 全闭环。**
