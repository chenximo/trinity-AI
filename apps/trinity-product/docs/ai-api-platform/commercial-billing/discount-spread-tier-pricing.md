## 1. 要解决什么问题

上游协议里常见「挂牌 × 6.5 折」一类**成本折**。平台对外可以给客户不同深度的**销售折**（或等价的账号 `discount_factor`）。中间这一段差价，乘上客户实际用量，才是折扣维度的利润。

需回答：

1. 充值 / 月耗达到某档时，给客户几折，单位毛利与毛利率是多少？  
2. 月维度、年维度怎么设门槛？  
3. 价格打深了用量会涨——**总利润 = 单位毛利 × 数量**，如何找「最赚钱」的平衡点？  
4. 不同成本折的模型族（6.5 / 7.5 / 7.8 / 9.7）能否共用一张「全站折」？（结论：**不能**。）  
5. **同一对外模型名下有线路 1 / 2 / 3，成本折不同——对客打一价还是分线路价？薄成本线路要不要打折？**（见 §6）

本文先定**口径与试算框架**；下一步用具体 model + 线路填数验证后再改「待拍板」为「已确认」。

## 2. 概念与符号

统一用同一套**挂牌原价** \(L\)（供应商挂牌 / 约定牌价；与 [商务原底](../pricing-sources/supplier-cost-discounts) 对齐）对齐折数：

| 概念 | 符号 | 含义 | 示例 |
|------|------|------|------|
| 挂牌原价 | \(L\) | 对齐折数用的参照价 | 某模型入 token 挂牌 |
| **成本折** | \(d_{cost}\) | 相对挂牌的进货系数 | 6.5 折 → \(0.65\) |
| **销售折** | \(d_{sell}\) | 相对挂牌的对客系数 | 8.5 折 → \(0.85\) |
| 进货成本 | \(C = L × d_{cost}\) | 真实 COGS（未计支付费） | |
| 对客售价 | \(P = L × d_{sell}\) | 客户实付单价（或由刊例反推） | |
| **单位毛利** | \(m = P − C = L × (d_{sell} − d_{cost})\) | 单次 / 单位量赚多少 | |
| **单位毛利率** | \(gm = m / P = 1 − d_{cost}/d_{sell}\) | 俗称毛利率 | |
| 用量 | \(Q\) | 按挂牌美元等价消耗，或 token/张/秒换算后的量 | 月耗 / 年耗 |
| **总毛利** | \(\Pi = m × Q\) | 折扣差价贡献的利润 | |

记法约定（与商务台账一致）：

- 台账写 **「X 折」**（如 `6.5`），系数 = \(X / 10\)。  
- **成本折 ≠ 客户折扣**；成本折只进 COGS，不对外展示。

### 2.1 与现网刊例、账号折扣的关系

现网对外是 **Models 一口价**（含默认 markup，不展示加价公式），账号可有 `discount_factor`（实付 ≈ 刊例 × factor）。对内核算时可二选一表达，**全公司须统一**：

| 口径 | 做法 | 建议 |
|------|------|------|
| **挂牌对齐（推荐对内）** | 由刊例反推 \(d_{sell} = P_{list} / L\)；企业再折 → \(d_{sell,eff} = d_{sell} × f\) | 商务、财务主口径 |
| **刊例再折（对外话术）** | 「标价基础上再减 X%」= factor \(f\) | 销售话术可用；算毛利时换算回挂牌口径 |

**充 $N 到账 $N**（见 [MVP PRD](./commercial-billing-mvp-prd)）不变；企业让利走**用量折扣档**，不做「充值加赠 Credits」作为主策略（避免对账与 Stripe 吸收逻辑冲突）。

### 2.2 刻意不纳入本主公式的项

| 项 | 说明 |
|----|------|
| Stripe 等通道费 | 约 2.9%+$0.30，平台吸收；产品定价时可用「毛利率多留 2–3pt」覆盖，**不改写** \(d_{cost}\) |
| 缓存命中 | 见 **[缓存计费盈利](./cache-billing-profit)**（命中少报 δ；与主阶梯拆开） |
| 试用赠额 | 记 CAC，不进商用折扣差价表 |

## 3. 毛利目标与「最深能打几折」

### 3.1 建议目标（待拍板）

| 层级 | 建议 | 含义 |
|------|------|------|
| **默认对客** | 单位毛利率约 **22%–25%** | 公开刊例锚定区 |
| **阶梯底线** | 单位毛利率 **≥ 20%** | 任一自助/标准合同档不低于此 |
| **战略合同** | 可低至约 **15%–18%** | 须销售+财务特批；按模型清单锁价 |

产品定价主看 **刊例（或有效销售折）毛利率**；净利（Stripe、试用、退款）另列财务报表。

### 3.2 底线公式

给定成本折与目标毛利率 \(gm_{min}\)：

\[
d_{sell,min} = \frac{d_{cost}}{1 - gm_{min}}
\]

即：销售折**不能深于** \(d_{sell,min}\)（系数越大越贵；「最深折」= \(d_{sell,min}\)）。

**示例（\(gm_{min}=20\%\)）**：

| 成本折（台账） | \(d_{cost}\) | 客户最深销售折（系数） | 约几折 |
|----------------|--------------|------------------------|--------|
| 6.5 | 0.65 | \(0.65 / 0.80 = 0.8125\) | **≈ 8.1 折** |
| 7.5 | 0.75 | 0.9375 | ≈ 9.4 折 |
| 7.8 | 0.78 | 0.975 | ≈ 9.8 折 |
| 9.7 | 0.97 | 1.2125 | **挂牌之上仍须加价**；不能「打折促销」到挂牌以下 |

→ **阶梯必须按成本折族（或模型清单）拆开**；禁止「全目录统一 8 折」。

## 4. 总利润与「量价平衡」

单位赚得多但卖得少，总利润未必最优：

\[
\max_{d} \; \Pi(d) = L × (d − d_{cost}) × Q(d)
\]

其中 \(Q(d)\) 随销售折加深（\(d\) 变小）而上升（更多充值、更多切量）。最优档是使上式最大的 \(d\)，而非最贵或最折。

### 4.1 实操：三档试算（拍板前够用）

对**同一成本折族**，给定弹性假设后比总毛利：

| 试算档 | 毛利率锚点 | \(Q\) 假设 | 用途 |
|--------|------------|------------|------|
| 保守 | ≥ 25% | 当前真实用量 | 底盘 |
| 进攻 | ≈ 20% | 因折扣用量 ×1.3～2（销售给区间） | 候选标准阶梯 |
| 极限 | ≈ 15%–18% | 更大放量 | 仅战略大客 |

判断：

- 进攻档 \(\Pi\) 明显高于保守 → 阶梯写入进攻折；  
- 接近持平 → **优先保 20% 底线**（管理简单、抗上游调价）；  
- 极限档 \(\Pi\) 更高但踩底 → 合同特批，不进自助表。

### 4.2 示意（非拍板数）

假设 \(d_{cost}=0.65\)，某模型族月度「挂牌美元等价消耗」随折扣变化：

| 场景 | \(d_{sell}\) | 差价系数 \((d_{sell}-0.65)\) | 假设月 \(Q\)（挂牌 $） | 相对总毛利 \(\propto (d-0.65)×Q\) |
|------|--------------|------------------------------|----------------------|-----------------------------------|
| 坚挺 | 0.90 | 0.25 | 8,000 | 2,000 |
| 中折 | 0.85 | 0.20 | 14,000 | **2,800** |
| 近底线 | 0.81 | 0.16 | 18,000 | 2,880 |
| 过深 | 0.78 | 0.13 | 22,000 | 2,860（反降且近成本） |

→ 示意上「中折～近底线」总利润更优；**真实峰值以模型测算表替换 \(Q(d)\) 后确定**。

## 5. 阶梯折扣框架（草案）

### 5.1 门槛：预存 vs 消耗

| 用途 | 建议维度 | 说明 |
|------|----------|------|
| **利润核算** | 实际消耗 \(Q\)（月 / 年） | 总利润只认用量 |
| **档位升级（营销）** | 预存余额，和/或近 30/90 日累计消耗 | 单笔充 1 万但不用完 ≠ 月耗 1 万 |
| **年框合同** | 年消耗 commit + 预存 | 承诺量不达标可约定回溯公开价 |

时间维：

- **月**：自助调档、运营观察更灵敏。  
- **年**：大客合同、更深折须绑 commit，否则峰值不稳。

### 5.2 档位表示意（待测算填数）

以下为**结构模板**：折数仅对「成本约 6.5 折」族示意；Claude（7.8）、视频（9.7）等另表。

| 客户档 | 门槛（示意） | 建议 \(d_{sell}\)（6.5 成本族） | 约单位 GM | 备注 |
|--------|--------------|----------------------------------|-----------|------|
| Standard | 公开 | ≈ 0.90（或当前刊例反推） | ~25%+ | Models 一口价 |
| Plus | 月耗或预存 ≥ **$1k**（待拍） | ≈ 0.88 | ~26% | 轻让 |
| Growth | ≥ **$10k**（待拍） | ≈ 0.85 | ~23.5% | 主力企业档 |
| Scale | ≥ **$50k**（待拍） | ≈ 0.82～0.81 | ~20%～21% | 摸底线；宜年框 commit |
| Enterprise | Wire / 月结 / 定制 | 模型清单锁价 | 合同保底 | Contact Sales |

**对外**：可说「大额预存可享更优折扣 / 联系销售」。  
**对内**：每档必须有适用 `model_id` 清单 + \(d_{sell}\) 或 factor + GM 底线。

### 5.3 按成本折**系数档**运营（推荐，不必逐模型）

**可以、且建议这样做**：对客阶梯与「打不打折」按 **\(d_{cost}\) 系数带**（或后台「产品分组」上挂的同一折扣系数）批量处理，**不要求每个 model_id 单独拍一版折**。

记法对齐（勿混）：

| 说法 | 含义 | 系数 |
|------|------|------|
| 台账「6.5 折」 | 常见商务说法 | \(d_{cost}=0.65\) |
| 后台「折扣 0.65」 | 产品分组上的系数（见运营分组表） | 同上 |
| 「0.65 折」 | **避免使用**（易与 6.5 折混淆） | — |

同一系数档内（如所有 \(d_{cost}=0.65\) 的分组：Azure官方、aws-cache、HA-Claude、Qwen、MiniMax…）**共用一套销售折 / 阶梯规则**；\(0.60\)、\(0.55\)、\(0.30\)、\(1.00\) 各用另一套。

| \(d_{cost}\) 系数档（示例，以现网分组为参考） | 典型产品分组 | 对客处理原则（待测算填数） |
|-----------------------------------------------|--------------|----------------------------|
| **≤0.35**（极深，如 0.3） | default、Codex | 空间大；可做体验/Auto/阶梯主力，但仍算 \(gm\)；防滥用与质量投诉 |
| **0.50–0.55** | Grok、azure-claude | 中深；可进阶梯，让利幅度介于极深档与 0.65 档之间 |
| **0.60** | GLM、Kimi | 中等；标准阶梯，让利略保守于 0.65 若质量/容量更紧 |
| **0.65** | Azure官方、aws-cache、HA-Claude、Qwen、MiniMax 等 | **样板档**：用 §3.2 算最深销售折；企业 Plus/Growth 主打此档 |
| **0.78–0.85**（若存在） | 浅成本线 | 轻折或不进深阶梯；见 §6 多线路 |
| **≈1.0** | Claude官方 等 | **原厂高成本**；默认不打折或仅合同微让；可作品质点名加价 |

**批量规则（拍板用）**：

| 规则 | 说明 |
|------|------|
| 一档一策 | 同一 \(d_{cost}\)（或分组折扣字段相同）→ 同一套 \(d_{sell}\) 阶梯与「是否享受企业折」 |
| 分组继承 | 新建模型挂到已有产品分组 → **自动继承该组折扣档策略**，无需逐模型评审 |
| 例外白名单 | 仅高粘性战略款、严重亏损路由、合规限制款进「破档」清单 |
| 多线路 | 同一对外模型跨多个分组时，按 **命中分组的 \(d_{cost}\)** 套档，或按 §6 一口价 + mix |
| 禁止 | 全站一个销售折盖所有系数档 |
| 禁止 | 只按模型名定价、忽略分组上的成本系数 |

**和客户档位的关系**：  
客户档（预存 $1k / $10k…）决定「在该系数档的策略里取第几档销售折」；  
**系数档**决定「有没有资格打折、天花板多深」。两维交叉，而不是「每个模型 × 每个客户」二维爆炸。

```text
客户档（Standard / Plus / Growth…）
        ×
成本系数档（0.3 / 0.55 / 0.6 / 0.65 / 1.0…）
        →
该次调用适用的销售折或 discount_factor
```

## 6. 同模型 · 多线路（核心补充）

> 关联工程/运营：[模型上架与供应线路](../operations/models-routes)。  
> 真相：**客户以为一个 model；平台背后是多条供应线路，各自 \(d_{cost}\) 不同。**

### 6.1 典型情况（你们已出现的形态）

同一对外模型（或同族 model_id），例如：

| 线路 | 上游成本折 \(d_{cost}\) | 计划对客销售折 \(d_{sell}\) | 单位差价 | 说明 |
|------|-------------------------|-----------------------------|----------|------|
| **线路 1** | 6.5（0.65） | 7.5（0.75） | \(L×0.10\) | **有钱可赚**；GM = \(1-0.65/0.75 ≈ 13.3\%\)（若要坚持 ≥20%，对客应 ≥约 8.1 折，见 §3.2——**7.5 对客在 6.5 成本下偏深，须测算是否故意让利**） |
| **线路 2** | 7.8（0.78） | 7.8（0.78） | \(0\) | **按上游折平进平出 → 本线路折扣差价利润为 0**（未计 Stripe / 运维） |
| **线路 3** | （更高或持平） | 待定 | 待测算 | 要不要打折、打到哪，**目前未测算 → 本文先定规则再填数** |

要点：

1. **「给客户 7.5」必须相对同一挂牌 \(L\) 读**；且必须扣着**实际命中线路的 \(d_{cost}\)** 算 GM，不能拿线路 1 的成本去养活线路 2 的报价故事。  
2. 线路 2「打 7.8 与上游一致」= **战略角色**（扩容、兜底、对标原厂体感），不是利润主力；须在组合层面算总账。  
3. 若客户**可指定线路**且看见线路 2 与线路 1 同价或更便宜却更「官方」，流量会砸向薄利/零利线路 → **定价与路由策略必须一起定**。

### 6.2 对客价的三种产品形态（须先选主形态）

| 形态 | 客户看到 | 扣费价 | 适合 | 风险 |
|------|----------|--------|------|------|
| **A. 模型一口价 + 平台选路** | 一个 model 一个价 | 可选：统一 \(P\)；成本随命中线路变 | Auto / 默认线路；主推 | 必须用**加权成本**或**最差线路成本**守 GM |
| **B. 分线路报价** | `model` / `model@route` 或「线路品质档」 | 各线路自己的 \(P_r\) | 专业客户点名高稳线路 | 目录复杂；利差清晰 |
| **C. 一口价 + 线路权限** | 默认一口价；开通「优质线」另授权 | 默认线走利润线；点名线可加价或不折 | 企业套餐（轻量 vs 专业） | 需产品包装配合 |

**建议默认（待拍板）**：零售与多数 API → **形态 A**（或 A+C）；大客可点名 → **B 或 C 的点名加价**。  
禁止在未做加权之前，把「线路 1 的厚利假设」写进全模型阶梯折。

### 6.3 核算：按线路算，再聚合成模型

**单线路**（与 §2 相同）：

\[
m_r = L × (d_{sell,r} − d_{cost,r}), \quad
gm_r = 1 − \frac{d_{cost,r}}{d_{sell,r}}
\]

**形态 A 下模型级期望**（路由 mix：线路 \(r\) 流量占比 \(w_r\)，\(\sum w_r=1\)）：

\[
C_{mix} = \sum_r w_r \, C_r, \quad
m_{mix} = P − C_{mix}, \quad
gm_{mix} = m_{mix}/P
\]

守门两道（建议同时满足才进「可打折」池）：

| 守门 | 规则 | 用意 |
|------|------|------|
| **线路底线** | 凡**可被客户稳定点名或路由可能高占比**的线路，\(gm_r\) 不得长期＜约定底线（如 20%）；零利/负利线须标注角色 | 防被指定打穿 |
| **组合底线** | 默认路由下 \(gm_{mix}\) ≥ 目标（如 20%–22%） | 平台选路时的真毛利 |

**线路角色标签（对内必填，测算用）**：

| 角色 | 典型 | 对客折扣策略（原则） |
|------|------|----------------------|
| **利润主力** | \(d_{cost}\) 深，如 6.5；容量够 | 可进阶梯让利；\(d_{sell}\) 仍守 GM |
| **品质 / 原厂感兜底** | \(d_{cost}\) 浅，如 7.8；稳、合规体感 | **默认不加入深阶梯**；\(d_{sell} ≥ d_{cost} + buffer\)（buffer 覆盖 Stripe≈2–3pt，至少到名义微利）；或 **对客价对齐利润线一口价**（同一 \(P\)，本线赚更少甚至为零，靠路由少走） |
| **弹性扩容** | 贵或平，峰值才切 | 可不单独对外报价；内部清算；避免营销折引导客户点名 |
| **不宜商用促销** | 成本≈售价或倒挂 | 不上架促销、不进试用深池；仅故障切换 |

对照你们例子的**原则解**（数字待测算锁定）：

```text
线路1（0.65）：可做「赚钱线」→ 对客 d_sell 按 §3 定在 GM≥20% 一带（未必是 7.5）
线路2（0.78）：
  - 若对客也打 7.8 → 标记「零差价兜底」，默认路由权重应低，或
  - 更好：对客仍收与线路1相同的一口价 P（例如等价 8.x～9 折）
    → 线路2 仍可能薄利/零利，但不会比线路1 更「好看」而虹吸流量
  - 不建议：线路2 单独再深折促销
线路3：先标角色 + 算 gm_r / 进 mix，再决定「折不折、进不进阶梯」
```

### 6.4 「线路 2 / 3 要不要打折」决策序

按顺序过，避免拍脑袋：

1. **该线路是否对客户可见 / 可点名？**  
   - 否 → 不对外谈「折」；只定内部路由权重与清算。  
   - 是 → 进入 2。  
2. **\(d_{cost}\) 与目标 GM 下是否存在合法 \(d_{sell}\)**（§3.2）？  
   - 否（如成本已 9.7）→ **不打折**，只加价或不上架促销。  
   - 是 → 进入 3。  
3. **若与利润主力线路共用一口价 \(P\)，本线 \(gm_r\) 是否可接受（可为 0～薄利）？**  
   - 可接受且路由可控 → 可不对客户「再折」，只共享 \(P\)。  
   - 不可接受 → 提高 \(P\)、限制点名、或下调该线路权重。  
4. **企业阶梯折是否作用于该线路？**  
   - 仅 **利润主力 + mix 后仍达标** 的线路/组进入阶梯；  
   - 兜底/零差价线：**阶梯折不适用**或单独合同封顶。  
5. **填测算表**（§7）后再写进销售货架。

### 6.5 与模型分组、Auto 的关系（简记）

- **Auto / 默认选路**：应偏向利润主力线路；品质线作 fallback，避免「便宜点名线」反客为主。  
- **专业点名**：点名到浅成本线时，应用**更高有效价或不享受阶梯折**，避免客户用点名权做套利。  
- 钱包额度不变；变的是**命中哪条线的 C，以及这线有没有资格吃折扣。**

## 7. 模型 / 线路测算怎么做（下一步）

**测算粒度：一行 = 一条供应线路**（可再按 model 汇总 mix）。勿只按模型名一行糊弄。

| 列 | 含义 |
|----|------|
| model_id（对外） | 客户侧名称 |
| route_id / 供应商 | 线路 1/2/3 |
| 线路角色 | 利润主力 / 品质兜底 / 扩容 / 不宜促销 |
| 台账成本折 \(d_{cost}\) | [商务原底](../pricing-sources/supplier-cost-discounts) |
| 挂牌 \(L\)（入/出/缓…） | |
| 拟对客 \(d_{sell}\) 或刊例 \(P\) | 统一口价则多行同 \(P\) |
| \(gm_r\) | 本线毛利率 |
| 是否客户可点名 | 是 / 否 |
| 默认路由权重 \(w_r\)（假设） | |
| 是否享受企业阶梯折 | 是 / 否 / 有封顶 |
| \(Q_r\) 或占比假设 | |
| \(m_r×Q_r\) 与 mix \(gm\) | |
| 结论 | 折深建议 / 不上促销 / 仅兜底 |

优先测算顺序建议：

1. 选 **1 个高流量同模多线路** 样板（如 deepseek 同族多线路）→ 先跑通 §6 规则。  
2. **成本 6.5 利润主力线** — 定可让利阶梯天花板。  
3. **7.8 类平进平出/薄利线** — 明确：共享一口价 vs 零差价标签 + 路由权重。  
4. **7.5 / 国际浅折 / 9.x** — 检验「不进阶梯」。  
5. 汇总到 model 级 \(gm_{mix}\)，再定对外套餐与阶梯。

测算完成后：回填 §5.2 与 §6 示例数字，更新本文状态为「已确认」，并同步 [models-routes](../operations/models-routes) 与销售话术。
