## 一、一句话：GEO 业务到底是什么

**GEO（Generative Engine Optimization）** 是围绕生成式 AI 答案的 **品牌可见性经营系统**：

> 当用户在 AI 里提问时，让品牌 **进入答案正文、占据合理叙事位置、被当作可信证据**；并 **持续监测 → 发现缺口 → 改造内容资产 → 验证效果**。

与传统 SEO 的本质差异：

| 维度 | SEO | GEO |
|------|-----|-----|
| 竞争单位 | 关键词排名 | **问题集里的答案份额** |
| 内容单位 | 网页 | **证据块**（事实 + 边界 + 对比 + 更新） |
| 成功标准 | 点击、流量 | **进答案、首推、可被引用** |
| 运营形态 | 优化页面 + 外链 | **结构化资产 + 信源权重 + 全域监测迭代** |

### 玩法1：公域全网品牌 GEO（主战场）

**目标**：用户在豆包、ChatGPT、Kimi 等 **自然提问** 时，AI 优先输出我方信息。

**标准流程（行业「全栈」表述）**：

```text
┌──────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ 基线扫描  │ → │ 素材标准化    │ → │ 信源分层加权  │ → │ 全域播种分发  │ → │ 监测迭代      │
│ 上千问句  │   │ Schema/FAQ   │   │ T1官方>T2媒体 │   │ 百科/媒体/社区│   │ 7×24 复测    │
└──────────┘   └──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

### 玩法2～4（知悉即可）

| 玩法 | 核心 | Trinity 优先级 |
|------|------|----------------|
| **2 本地生活** | 城市+品类+POI+抖音同城 | 低 |

## 四、统一业务主链（不拆版本 · 六环 + 行业五步对照）

无论大厂、SaaS 还是自研，**业务本质都是一条闭环**。下面用 **两套命名对齐**（清华/产品六环 ↔ 行业五步）。

```text
  产品六环（Trinity 演示 / 手册）          行业常见五步
  ─────────────────────────────          ─────────────────
  ① 策略规划  问题集·品牌·竞品      ←→    （隐含在基线扫描前的词库建设）
  ② 监测采集  向 AI 提问·存档        ←→    基线扫描 / 7×24 复测
  ③ 测量      SOA·位置·竞品·情感     ←→    提及率·首答率·占位
  ④ 诊断      失声·弱提及·归因        ←→    缺口·幻觉·竞品占位分析
  ⑤ 优化行动  改文档/FAQ/对比/Schema ←→    素材标准化·播种·任务工单
  ⑥ 验证      R1 vs R2·趋势告警      ←→    监测迭代·效果对赌数据
```

### 哪些环节是「程序化」、哪些常接 AI

```text
  接入层采集 ─────────────── 工程自动化（调度、爬虫、限流）
  品牌识别/SOA/规则诊断 ──── 规则引擎为主，边缘 case 可接 LLM
  页面审计/证据密度 ──────── 规则 + NLP；深度可接 LLM
```

### ③ 测量 · SOA · 主实现方案（规则引擎 + 别名库 + 时序库）{#measurement-soa-impl}

> **工程真源细则**见 [技术架构 · 测量与 SOA](./tech-architecture#measurement-soa)。演示实现：`apps/trinity-geo/src/demo/analyze.ts`、`mvp/config/brand.json`。

**原则**：SOA 主路径 **程序化、可复现、可审计**；不用 LLM 直接「智能打分」。AI 仅作边缘标注增强（可选）。

```text
  ② 原始回答（采集库）
           │
           ▼
  ┌─────────────────┐
  │    别名库        │  品牌/竞品 canonical_name + aliases
  │  （测量配置）     │  ← 控制台「品牌设置」同步或写入
  └────────┬────────┘
           │ 字符串匹配 / 归一化 entity_id
           ▼
  ┌─────────────────┐
  │   规则引擎        │  if/else：提及 · 进正文 · 位置 · 竞品
  │  （单条标注）     │  → brand_mentioned, in_answer_body, position…
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │   时序库          │  按 round / 日期 / 平台 / 问题 存指标快照
  │  （聚合 SOA）     │  → Dashboard 趋势 · R1/R2 · 告警
  └─────────────────┘
```

| 组件 | 职责 | 典型数据 |
|------|------|----------|
| **别名库** | 认字：哪些写法算本品牌/竞品 | `entity_id`, `alias`, `entity_type`(brand/competitor) |
| **规则引擎** | 判分：进不进 SOA 分子、首推/末段、D1–D4 输入 | 配置表 + 代码；版本可追踪 |
| **时序库** | 记史：同一问题跨日变化、验证前后对比 | `question_id`, `round`, `soa`, `in_answer_body`, `collected_at` |

**SOA 口径（与 MVP 一致）**：

```
SOA = in_answer_body = Y 的采样数 / 该范围内总采样数
提及率 = brand_mentioned = Y 的采样数 / 总采样数   （可高于 SOA）
```

**与六环其它步骤的边界**：

| 步骤 | 关系 |
|------|------|
| ① 策略 | 别名、竞品在控制台维护 → **同步进别名库** |
| ② 采集 | 只写 **原始回答库**；不依赖别名库 |
| ④ 诊断 | **读** 测量标注结果；规则 D1–D4 与测量规则同域或下游 |
| ⑥ 验证 | **读** 时序库中 R1 vs R2 |

**别名库更新**：用户新增别名后，可触发 **批量重算** 近期标注（工作流，非 LLM）。配置落库、任务态与接口见 [品牌设置 PRD](../../../trinity-geo/marketing/console/brand-settings.md) §5–§6、[技术架构 · 别名同步与重算](./tech-architecture#alias-sync-recalc)。

**可选增强（非主路径）**：NER、模糊匹配、LLM 判「弱提及/情感」→ 结果写回标注表，**SOA 聚合公式不变**。

## 七、核心指标词典（全景统一口径）

| 指标 | 含义 | 计算直觉 |
|------|------|----------|
| **SOA** 答案份额 | 关键问题集里，品牌进入 AI **答案正文**的占比 | 进正文条数 / 采样总条数 |
| **提及率** | 回答里出现品牌名（含别名）的比例 | 可高于 SOA |
| **首推率** | 被作为第一个推荐/论述主体的比例 | 位置在前列 |
| **CCR** 引用捕获率 | 被当作 **证据源** 的比例（相对仅提及） | 需引用判定 |
| **竞品占位** | 同问题集下竞品 SOA / 首推对比 | 雷达、对比表 |
| **情感** | 提及时的正/中/负倾向 | 分类器或规则 |
| **幻觉** | AI 对品牌事实性错误描述 | 事实表比对 |
| **首答率** | 行业常用语，≈ 首推 + 进正文 | 与 SOA 口径需统一 |

## 八、诊断类型（④ 层 · 统一规则语言）

```text
                    测量结果
                       │
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
    品牌未出现     品牌出现      品牌出现
    竞品出现       未进正文      位置落后
         │             │             │
         ▼             ▼             ▼

         │
         └─ 问法含品牌仍无 ──► D2 品牌未收录/别名
```

| 代码 | 名称 | 优化方向（⑤） |

| **D3** | 弱提及 | 证据前置、证据密度 |
| **D4** | 叙事落后 | vs 竞品对比表、首推位素材 |

## 十、成熟 GEO 系统 · 六条判别（PM 自检）

```text
  [1] 跨多 AI 平台适配          ── 不止单一模型
  [2] 信源分级权重              ── 可调素材优先级
  [3] 自动化结构化                ── Schema / llms.txt / FAQ
  [4] 完整曝光监测                ── 提及·首答·SOA·趋势
  [5] 原始素材 vs 优化资产分离    ── 清洗·校验流水线
  [6] 可对接 RAG/Agent            ── 作前置优化中间件
```

**说明**：这是 **「成熟全栈」** 判别尺，不是「能不能上线」的门槛。监测型 SaaS 可先满足 **[4]**，逐步补齐其余。

```text
         ▼              ▼              ▼
   演示六环已通    采集工程化待补    结构化/Agent 在链路上
   (程序化)        (流程化)          (同图右侧，非另起炉灶)
```

### 与「必须上 Agent 吗」的结论（写进全景）

| 层次 | 是什么 | 是否必需 |
|------|--------|----------|
| **业务闭环** | ①～⑥ 六环 | **必需**（GEO 定义本身） |
| **流程化** | 定时采集、任务、告警、工单 | **商用必需** |
| **AI 单次调用** | 问题扩展、优化草稿、边缘标注 | **增强，非入门必需** |
| **Agent 多步自主** | 自动改站、发布、再测 | **高阶；底层仍是同一主链** |
