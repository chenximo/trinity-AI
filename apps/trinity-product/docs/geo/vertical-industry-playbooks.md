## 一、为什么需要行业 Playbook

GEO 的 **六环运营闭环**（策略 → 监测 → 测量 → 诊断 → 优化 → 验证）对所有 **全球化产品** 通用。  
行业差异不在「要不要做 SOA」，而在 **怎么配问题、怎么读信源盘、优先改什么内容**。

**行业 Playbook（行业引导包）** 的定义：

> 在同一套 Trinity GEO 平台上，为特定垂直行业提供 **可导入的样例、预期与动作模板**，降低 onboarding 成本，避免用户从空白表单开始。

**不是**：

- 每个行业一套产品、一套六环、一套定价
- 锁死配置（用户必须能改、能删、能覆盖）
- 本地生活 / POI / 纯小程序私域（见 [业务全景 · 玩法2](./business-landscape)，非当前 ICP）

**是**：

- 平台 **一个**；差异落在 **四层引导**
- 对外可称「XX 行业 GEO 解决方案」；对内仍是同一控制台与 API

## 二、设计原则

| # | 原则 | 说明 |
|---|------|------|
| P1 | **样例是起点，不是上限** | 模板可编辑；禁止「选行业即锁死问题集」 |
| P2 | **引导差异，不重复横切** | SOA 公式、D1–D5、S1–S6、R1/R2 全行业共用 |
| P3 | **先配题，再采数** | 没有问题集就没有 SOA 分母；行业包首要交付问题样例 |
| P4 | **品类 + 对比 ≥ 50%** | 避免只盯品牌词导致 SOA 虚高、采购决策题失声 |
| P5 | **口语化问法** | 用户真会问的句子，不是 SEO 关键词堆砌 |
| P6 | **动作写清 D* + S* + URL** | 待办必须可验收，见 [产品设计 · §0.6.7](./product-design-analysis#visibility-layers) |
| P7 | **R2 双指标** | 信源盘 Δ 往往早于 SOA Δ；验收时两者同看 |
| P8 | **宁少勿滥** | 2 个写满的 Playbook 胜过 8 个空壳解决方案页 |

## 三、四层引导模型

```text
                    ┌─────────────────────────┐
                    │   横切平台（不变）        │
                    │  六环 · SOA · D/S · R2   │
                    └───────────┬─────────────┘
                                │
        ┌───────────┬───────────┼───────────┬───────────┐
        ▼           ▼           ▼           ▼           ▼
     ① 问题集    ② 信源预期   ③ 优化动作库  ④ 演示样本   （+ 竞品种子）
```

| 层 | 引导什么 | 产品落点（现有 / 规划） |
|----|----------|-------------------------|
| **① 问题集** | 四类问法比例、15 条样例、反例 | `brand-settings` 行业 · `keywords` 快速填入 / 导入 |
| **② 信源预期** | 健康信源结构、读盘话术、S* 优先级 | `citations` 行业参考结构 · 诊断 ⓘ 文案 |
| **③ 优化动作库** | 同 D/S 规则下，行业默认待办模板 | `optimize` / `optimize-detail` 生成待办 |
| **④ 演示样本** | 1 条可讲完的 R1→R2 故事 | `mvp/data/` · 控制台深链 · 对外 Demo |

### 3.1 与 AI 推荐问题集的关系

| 来源 | 角色 |
|------|------|
| **行业 Playbook 样例库** | 稳定、可解释、适合 onboarding 与对外演示 |
| **AI 生成候选** | 在样例基础上扩写长尾场景（`keywords` 已有入口） |
| **用户自填** | 最终真源 |

推荐 onboarding 顺序：**导入行业 10 条样例 → AI 补 5 条 → 用户删改 → 配比检查通过 → 开采**。

### 3.2 产品暴露层级（L1–L3）

```text
L1 · 选行业（brand-settings）
      → 影响默认值、文案、模板库 ID
L2 · 从模板导入（keywords / competitors-manage）
      → 勾选批量加入，始终可改
L3 · 上下文提示（diagnosis / optimize / citations / reports 侧栏 ⓘ）
      → 「本行业通常先补 X」
```

## 四、Playbook 文档规格（维护模板）

每个行业 Playbook 独立 Markdown，建议路径：`docs/geo/industry/playbook-{slug}.md`（随行业增加而增）。

**必填章节**：

```markdown
# 行业 Playbook：{名称}

## 1. 适用 ICP 与边界
## 2. 问题集
  - 2.1 四类问法推荐比例
  - 2.2 样例问题（15 条，含类型 + 测什么）
  - 2.3 反例（3 条）
## 3. 竞品种子（5–8 家，可删）
## 4. 信源预期
  - 4.1 健康结构描述
  - 4.2 读盘检查清单
  - 4.3 本行业 S* 优先级
## 5. 优化动作库（D/S → 默认待办，5–10 条）
## 6. R2 验收预期（先进盘 / 先进答案）
## 7. 演示样本（控制台 / MVP 深链）
## 8. 变更记录
```

**控制台行业枚举对齐**（`brand-settings.html`）：

| slug | 显示名 | Playbook 状态 |
|------|--------|---------------|
| `ai-api` | AI / API 聚合 | ✅ 本文 §五 |
| `saas` | 企业 SaaS | 🟡 本文 §六（纲要） |
| `ecom` | 出海电商 / DTC | 🟡 本文 §七（纲要） |
| `consumer` | 消费电子 | ⬜ 待写 |
| `other` | 其他 | 无模板，仅通用配比规则 |

## 五、Playbook 范例：AI / API 聚合（写满）

> **典型样本**：[Trinity AI](../../../trinity-geo/mvp/config/brand.json) · 问题集 [questions.json](../../../trinity-geo/mvp/config/questions.json) · Q00 R1/R2 [mvp/data/](../../../trinity-geo/mvp/data/)

### 5.1 适用 ICP 与边界

| 项 | 说明 |
|----|------|
| **适用** | 大模型 API 网关、聚合平台、OpenAI 兼容中转、多模型路由基础设施 |
| **用户** | 开发者、平台工程、出海与国内并用的 SaaS 团队 |
| **信源载体** | 官网、开发者文档、定价页、GitHub / 开发者社区 |
| **不适用** | 纯闭源 SDK 无文档、只做私有部署无可引用 Web 页 |

### 5.2 问题集

#### 5.2.1 推荐比例（15 题基准）

| 类型 | 占比 | 说明 |
|------|:----:|------|
| 品类词 | 35–40% | 选型类目是否进答案（**最优先**） |
| 对比词 | 25–30% | 与 OpenRouter / 云厂商网关等直接对决 |
| 场景词 | 20–25% | Cursor、团队计费、降本等多模型接入场景 |
| 品牌词 | ≤ 15% | 测认知与 CCR，**不能**作为唯一 KPI |

**配比验收**：品类 + 对比 ≥ 50%；品牌词 ≤ 30%；每条为口语问句。

#### 5.2.2 样例问题（与 MVP 对齐）

| ID | 类型 | 问法 | 测什么 |
|----|------|------|--------|
| Q00 | 品类 | 推荐两款 API 聚合平台 | 品类首推是否包含我方（**P0 基准题**） |
| Q03 | 品类 | 国内有哪些 OpenAI 兼容的大模型 API 聚合平台？ | 国内生态选型盘 |
| Q04 | 品类 | 想一个 Key 调用多家大模型，有什么推荐？ | 统一 Key 需求 |
| Q05 | 品类 | 开发者怎么统一管理多个 LLM API？ | 治理 / 运维场景 |
| Q11 | 品类 | 豆包模型 API 哪里调用比较方便？ | 国内模型 + 聚合交叉 |
| Q06 | 对比 | Trinity 和 OpenRouter 哪个更适合国内开发者？ | 海内外聚合对决 |
| Q07 | 对比 | trinitydesk 和其他 API 中转平台比有什么优势？ | 中转 / 聚合叙事 |
| Q12 | 对比 | 直连火山引擎和走聚合平台调用豆包，怎么选？ | 云厂商 vs 聚合 |
| Q08 | 场景 | 我在做 AI 应用，怎么降低多模型接入成本？ | 成本场景 |
| Q09 | 场景 | Cursor 怎么配置第三方 API 网关？ | 工具链场景 |
| Q10 | 场景 | 团队想统一计费、统一看用量，选什么平台？ | 企业管控场景 |
| Q01 | 品牌 | Trinity AI 好用吗？适合什么场景？ | 品牌 + CCR 样本 |
| Q02 | 品牌 | trinitydesk.ai 是什么平台？ | 品牌识别 / 别名 |

> 完整 13 条见 `mvp/config/questions.json`；扩展至 15 条时可加「企业合规开票选哪个聚合平台」等场景词。

#### 5.2.3 反例（不要这样写）

| 反例 | 问题 |
|------|------|
| 「大模型 API 聚合平台 排名 2026 最好」 | SEO 堆砌，非用户口语 |
| 「Trinity AI」 | 单词品牌词，无决策信息 |
| 「API 聚合」 | 过短，AI 无法理解意图 |

### 5.3 竞品种子

| 竞品类型 | 示例 | 备注 |
|----------|------|------|
| 海外聚合 | OpenRouter、Portkey | 全球模型覆盖叙事 |
| 国内云网关 | 腾讯云 TokenHub 等 | 合规 / 企业级 |
| 开源自建 | LiteLLM、One API | 「可控 vs 省心」对比 |
| 开源推理 | 硅基流动等 | 成本 / 开源模型场景 |

录入时 **别名 ≥ 2 个/家**（见 `brand.json` `competitors[]`）。

### 5.4 信源预期

#### 健康结构（品类题 · 可提取平台）

| 层级 | 期望 | API 行业读法 |
|------|------|--------------|
| T1 官方 | 我方 + 竞品 doc / 产品页 | **选型题核心**；Q00 样本 R1 为 0/16 我方域 |
| T2 媒体 / 社区 | CSDN、少数派、GitHub Discussion | 第三方定型叙事 → S3 |
| T3 案例 | 客户集成故事 | 增强信任，非必须 |
| 不可提取 | 部分 API 无 `cited_urls` | 标 `citation_extractable=false`，归因靠 ①② |

#### 读盘检查清单

- [ ] 品类题参考链接中，是否有 **竞品官方 doc** 而无我方 doc？→ S2
- [ ] 我方域是否 **完全缺席**？→ S1
- [ ] 是否由 **横向评测文** 定型前两名？→ S3
- [ ] ChatGPT 与豆包 **信源结构是否割裂**？→ S6 / D5

#### 本行业 S* 优先级

**P0**：S1、S2（官方文档树、选型对比页）  
**P1**：S3（第三方评测渗透——系统提示缺口，不承诺上榜）  
**P2**：S4（页面可引用性 / 证据块）

### 5.5 优化动作库

| 诊断 | 默认待办模板 | 最小可执行动作 |
|------|--------------|----------------|
| D1 + S1/S2 | 新建「API 聚合选型」文档树 | `doc` 下 `/docs/api-aggregation`：定义 + 3 条事实 + 对比表 |
| D1 + S3 | 第三方对比内容缺口 | 运营：定向投稿 / 社区稿；对标 CSDN 类 URL |
| D2 | 品牌未识别 | 官网 About 补中英全称；`brand-settings` 别名对齐 |
| D4 + S2 | 对比叙事落后 | 首段补「适用团队规模 / 计费 / 国内线路」可验证事实 |
| D5 + S6 | 市场割裂 | 中文 doc 补国内场景；海外页补 global use case |

待办格式示例（与 MVP `actions.json` 一致）：

```text
D1 · S1+S2 · doc 建「API 聚合选型」官方文档树
对标：https://openrouter.ai/docs …
验收：Q00 信源盘我方域 M/N > 0 → 再采 R2 看 SOA
```

### 5.6 R2 验收预期

| 指标 | 本行业典型节奏 | Q00 样本 |
|------|----------------|----------|
| 信源盘 Δ | **往往先动** | R1 0 → R2 1（doc 进盘） |
| SOA Δ | **滞后** | R1/R2 正文仍为 0 → `partial` 合理 |
| CCR Δ | 品牌词先改善 | Q01 ChatGPT 样本：SOA + CCR 双 yes |

**对外话术**：「API 选型类问题，先争进信源盘，再争进答案正文——两步都要在验证页看见。」

## 八、跨行业共用：问题集编写规范

### 8.1 四类问法定义

| 类型 | 定义 | 通过标准 |
|------|------|----------|
| **品牌词** | 问法含品牌名或独有域名 | 测认知、CCR、别名覆盖 |
| **品类词** | 不问品牌，问品类 / 推荐 | 测选型类目 SOA（**决策漏斗上游**） |
| **对比词** | 明确 A vs B 或「和其他比」 | 测首推与叙事位置 |
| **场景词** | 具体人群 / 任务 / 约束 | 测长尾场景覆盖 |

### 8.2 配比与质量检查（Onboarding 自动化）

```text
✓ 总题数 ≥ 10（建议 15）
✓ 品类 + 对比 ≥ 50%
✓ 品牌词 ≤ 30%
✓ 无单字 / 无纯 SEO 堆砌
✓ 每条问题可在 ChatGPT / 豆包自然问出
✓ 对比题关联竞品已在 competitors-manage 录入
```

### 8.3 与六环的映射

| Playbook 层 | 六环 |
|-------------|:----:|
| 问题集 + 竞品 | ① 策略规划 |
| 样例采集规范 | ② 监测采集 |
| 信源预期 + SOA 口径 | ③ 测量 |
| D/S 解读话术 | ④ 诊断 |
| 动作库 | ⑤ 优化 |
| R2 预期 | ⑥ 验证 |

## 九、落地路线图（产品 / 内容 / 营销）

| 阶段 | 内容 | 工程 |
|------|------|------|
| **现在** | 本文 + API Playbook 写满 | `brand-settings` 行业枚举已有 |
| **P0** | 补 `playbook-saas.md` 或 `playbook-dtc.md` 其一 | `keywords` 按行业切换「快速填入」 |
| **P1** | 第二行业完整 R1/R2 样本 | 导入向导：选行业 → 勾选样例 → 配比检查 |
| **P1** | 信源页行业参考结构 | `citations` 侧栏 checklist |
| **P2** | 优化待办按行业拉模板 | `optimize` 生成 `D*+S*` 详情 |
| **P2** | 对外解决方案 landing | 营销页链控制台 Demo 深链 |

**对外命名建议**：

- 对内：**行业 Playbook / 引导包**
- 对外：**「{行业} GEO 解决方案」**（底层同一 Trinity GEO）

**定价建议**：早期 **专业版全含** Playbook，不做按行业加价；Enterprise 可提供「定制行业包 + 基准数据」。

## 十、反模式清单

| 反模式 | 后果 | 正确做法 |
|--------|------|----------|
| 每行业独立 SKU / 代码库 | 维护成本爆炸 | 四层引导 + 同一平台 |
| 8 个行业 Playbook 全空壳 | 销售无法演示 | 2 个写满再扩展 |
| 只给品牌词样例 | SOA 虚高，品类仍失声 | 强制品类+对比 ≥ 50% |
| 行业模板不可编辑 | 客户信任下降 | 导入 = 复制，非锁定 |
| 待办只写「D1 补 FAQ」 | R2 无法归因 | D* + S* + 对标 URL |
| 忽视信源盘只看 SOA | 误判优化无效 | R2 同时展示信源 Δ |
| 本地生活 / 纯小程序当主 Playbook | 与 ICP 不符 | 玩法1 全球化 Web 信源 |
