# GEO · MVP 实践手册（完整链路 · 可演示）

> **用途**：用 **1 个品牌、1–2 个 AI 平台、10–20 个问题**，亲手跑通 GEO **全链路**，用于对内对齐、对外演示、以及后续商用产品规划。  
> **定位**：与 [产品设计分析](./product-design-analysis) 同源（**可见性分层 ⓪–⑤ 见 [§0.6](./product-design-analysis#visibility-layers)**）；与 [原型页面清单](./v1-prototype-pages) 的商用界面对应——本文强调 **可跑通的最薄全链路**。  
> **不含**：定价、套餐、付费转化（商用规划另文）。  
> **可演示系统**：`apps/trinity-geo` → `/demo`（六环 UI，非仅文档）。

## 一、这份文档解决什么问题

| 场景 | 你能用它做什么 |
|------|----------------|
| **对内** | 向研发、设计、管理层说明：GEO 不是「监测报表」，而是 **答案经营系统** |
| **对外** | 向客户、投资人演示：从「发现问题」到「给出动作」到「验证效果」的完整故事 |
| **对自己** | 在写 PRD、画原型、排工程之前，先用 Excel/Notion **跑通一轮**，建立业务肌肉记忆 |

**实践规模（刻意做小）**：

| 维度 | 实践 MVP | 商用终局（对照） |
|------|----------|------------------|
| AI 平台 | **2 个**（建议 ChatGPT + 豆包） | 海外 5 + 国内 5 |
| 品牌 | **1 个** + **2 个竞品** | 多品牌、多租户 |
| 问题集 | **10–20 条**真实用户问法 | 上百关键词 + 自动扩展 |
| 采集 | **人工提供豆包回答（3～5 条即可）**；自动采集交工程师（见 [执行包 · COLLECTION-TBD](../../../trinity-geo/mvp/COLLECTION-TBD.md)） | 每日定时 + 存档 + 截图 |
| 诊断 | **规则表**（3–5 条） | 审计引擎 + 证据密度模型 |
| 优化 | **人工改 1–2 处内容** + 待办勾选 | 证据块生成、CMS 集成 |
| 验证 | **第二轮提问**对比 SOA | 自动周期采集 + 时间线 |

## 二、GEO 是什么（对内 / 对外统一口径）

### 2.1 一句话（电梯演讲）

> **GEO（生成式引擎优化）** 帮助品牌回答：当用户在 AI 里提问时，**我的品牌有没有进答案正文**、**竞品有没有占住叙事**、**我该改什么才能被 AI 安全引用**——并持续验证效果。

### 2.2 和 SEO 的区别（30 秒版）

| | SEO | GEO |
|--|-----|-----|
| 用户行为 | 搜索 → 点链接 | 提问 → **直接读答案** |
| 竞争单位 | 关键词排名 | **问题集里的答案份额（SOA）** |
| 内容单位 | 网页 | **证据块**（事实 + 边界 + 对比 + 更新） |
| 产品价值 | 带来更多点击 | **进答案、占叙事、可被引用** |

### 2.3 本产品做什么（系统定义）

GEO 系统 = **六环运营闭环** 的产品化（与清华报告同源）：

```
┌─────────────────────────────────────────────────────────────┐
│  ① 策略规划    定品牌、竞品、关键问题集（用户在 AI 里怎么问）   │
├─────────────────────────────────────────────────────────────┤
│  ② 监测采集    向各 AI 平台提问，存档原始回答（可复现）       │
├─────────────────────────────────────────────────────────────┤
│  ③ 测量        识别品牌是否进入答案正文 → 算 SOA、位置、情感  │
├─────────────────────────────────────────────────────────────┤
│  ④ 诊断        找出「失声 / 落后 / 弱提及」的原因类型         │
├─────────────────────────────────────────────────────────────┤
│  ⑤ 优化        给出可执行动作（改内容、补 FAQ、加别名…）      │
├─────────────────────────────────────────────────────────────┤
│  ⑥ 验证与运营  再采集 → 对比前后 SOA；异常告警、待办闭环      │
└─────────────────────────────────────────────────────────────┘
         ↑___________________________________|
                    持续运营，非一次性项目
```

**演示时要强调**：哪怕平台只有 2 个、优化只是改一段话，**六环都必须出现**，否则只是「AI 可见性监测工具」，不是 GEO 运营系统。

## 三、实践前准备

### 3.1 选一个练习品牌（建议）

选你熟悉、公开信息多、便于改官网/文档的品牌。示例类型：

- 出海 SaaS / 消费电子 / 跨境电商 DTC  
- **避免**：强监管行业首练（医疗处方、金融投顾），合规环在实践版仅做「检查清单」占位

本文下文以虚构品牌 **「NebulaTrack」**（出海项目管理 SaaS）为例，你可替换成真实客户。

### 3.2 工具清单

| 工具 | 用途 |
|------|------|
| **表格**（Excel / Google Sheet / 飞书） | 问题集、采集记录、SOA、建议、验证 |
| **ChatGPT**（网页或 API 任选） | 海外代表平台 |
| **豆包**（网页） | 国内代表平台 |
| **截图工具** | 每条回答留证（演示必备） |

### 3.3 下载模板结构

在表格中建 **6 张 Sheet**（名称可自定）：

1. `00_品牌与竞品`  
2. `01_问题集`

4. `03_SOA汇总`

6. `05_验证对比`

各 Sheet 字段见下文各阶段。

## 四、完整链路实践（六步）

### 阶段 ① 策略规划（约 30 分钟）

**业务目标**：定义「盯哪些问题、跟谁比」——没有问题集就没有 SOA 分母。

#### 操作步骤

1. **填写品牌档案**（Sheet `00`）

| 字段 | 示例（NebulaTrack） |
|------|---------------------|
| 品牌主名 | NebulaTrack |
| 别名（AI 可能写法） | Nebula Track, 星云轨迹, NebulaTrack PM |
| 官网 | https://example.com |
| 行业 | 出海 SaaS / 项目管理 |
| 核心竞品 A | Asana |
| 核心竞品 B | Monday.com |
| 竞品别名 | Asana, Monday, monday.com |

2. **编写问题集 10–20 条**（Sheet `01`）

覆盖四类意图（每类至少 2 条）：

| 类型 | 说明 | 示例问法 |
|------|------|----------|
| 品牌词 | 直接问品牌 | NebulaTrack 好用吗？ |
| 品类词 | 不问品牌，问品类 | 适合小团队的出海项目管理工具有哪些？ |
| 对比词 | 品牌 vs 竞品 | NebulaTrack 和 Asana 哪个适合远程团队？ |
| 场景词 | 具体使用场景 | 跨境电商团队怎么选项目管理软件？ |

3. **验收标准**

- [ ] 至少 10 条问题，且 **品类词 + 对比词 ≥ 50%**（避免只盯品牌词导致 SOA 虚高）  
- [ ] 每条问题都是 **用户真会问的口语**，不是 SEO 关键词堆砌  
- [ ] 竞品已录入且别名 ≥ 2 个/家  

**对应商用产品**：Onboarding、品牌/问题集/竞品配置（见 [原型清单 §5.1](./v1-prototype-pages#51-用户首次使用引导流)）。

---

### 阶段 ② 监测采集（约 1–2 小时）

**业务目标**：拿到 **可复现、可展示** 的 AI 原始回答——演示时比分数更有说服力。

#### 操作步骤

1. 对 Sheet `01` 中 **每一条问题**，分别在 **ChatGPT** 和 **豆包** 各问 **1 次**（实践版不做多次采样；商用版建议每题每平台 ≥3 次取稳定值）。

2. 写入 Sheet `02_采集记录`：

| 列名 | 说明 |
|------|------|
| `question_id` | 与问题集对应 |
| `question_text` | 原问法 |
| `platform` | `chatgpt` / `doubao` |
| `collected_at` | 日期时间 |
| `answer_full` | 回答全文（粘贴） |
| `screenshot` | 截图文件名或链接 |
| `round` | `R1`（首轮）/ `R2`（优化后验证轮） |

3. **采集规范**（保证可对比）

- 使用 **默认模型**（不刻意选最强/最弱变体），记录模型名  
- **不**在同一对话里连续问多题（避免上下文污染）；每题新会话  
- 国内平台若需登录，固定同一账号  

4. **验收标准**

- [ ] 10 题 × 2 平台 = **至少 20 条**完整记录  
- [ ] 每条有截图或可追溯链接  
- [ ] `round=R1` 全部完成后再进入阶段 ③  

**对应商用产品**：采集任务调度、AI 回答详情页、采集日志、回答存档。

---

### 阶段 ③ 测量（约 1 小时）

**业务目标**：把「一大段话」变成 **进没进答案、排第几、情绪如何**。

#### 3.1 单条回答标注规则（Sheet `02` 增加列）

| 列名 | 取值 | 判定说明 |
|------|------|----------|
| `brand_mentioned` | Y / N | 正文是否出现本品牌（含别名） |
| `mention_position` | 首推 / 前列 / 中段 / 末段 / — | 品牌**首次**出现位置 |
| `competitor_mentioned` | 无 / A / B / A+B | 竞品是否出现 |
| `sentiment` | 正 / 中 / 负 / — | 对该品牌的态度（未提及填 —） |
| `in_answer_body` | Y / N | 是否在**推荐/论述正文**中（非仅脚注式一闪） |

**`in_answer_body` 实践口径**：

- **Y**：品牌出现在推荐列表、对比段、结论段  
- **N**：仅在一句话带过、或完全未出现  

> 商用版 SOA 分母常用「采样回答总数」；分子为 `in_answer_body=Y`。实践版与商用口径保持一致。

#### 3.2 计算 SOA（Sheet `03_SOA汇总`）

> **商用实现**：**规则引擎 + 别名库 + 时序库** 为主（非 LLM 打分）。架构见 [技术架构 · 测量与 SOA](./tech-architecture#measurement-soa)；演示 `apps/trinity-geo/src/demo/analyze.ts`、`mvp/config/brand.json`。

**公式**：

```
SOA = 品牌 in_answer_body=Y 的回答数 / 该范围内总采样回答数
```

建议做 **三张汇总表**：

| 汇总维度 | 行 | 列示例 |
|----------|-----|--------|
| **总览** | NebulaTrack | SOA、提及率、首推次数 |
| **分平台** | ChatGPT / 豆包 | 各平台 SOA |
| **分问题类型** | 品牌词 / 品类词 / 对比词 / 场景词 | 各类型 SOA |

**示例（虚构数据）**：

| 维度 | 采样数 | 进答案数 | SOA |
|------|:------:|:--------:|:---:|
| 全平台 | 20 | 6 | 30% |
| ChatGPT | 10 | 5 | 50% |
| 豆包 | 10 | 1 | 10% |
| 品类词 | 8 | 1 | 12.5% |
| 对比词 | 6 | 2 | 33% |

#### 3.3 竞品对比（同 Sheet 或附表）

对每个问题标注：**谁进了答案、谁首推**。统计：

- **失声问题**：SOA=0 且竞品出现  
- **共现落后**：品牌和竞品都在，但竞品首推  
- **独占**：仅本品牌出现  

#### 3.4 验收标准

- [ ] 总 SOA、分平台 SOA、分类型 SOA 均已算出  
- [ ] 能列出 **Top 3 失声问题** 和 **Top 3 竞品占优问题**  
- [ ] 能指着一条 AI 回答详情解释「为什么这条算进/不算进答案」  

**对应商用产品**：可见性总览、关键词详情、竞品概览（见 [原型清单 §5.4](./v1-prototype-pages#54-soa-三层可视化--页面对照)）。

---

### 阶段 ④ 诊断（约 45 分钟）

**业务目标**：回答 **「为什么没进答案 / 为什么落后」**——归到 **可行动因类型**。答案侧用 **D1–D5**；若采集到参考链接，叠加 **信源缺口 S1–S6**（见 [产品设计 · §0.6](./product-design-analysis#visibility-layers)）。

#### 4.1 实践版诊断规则表（够用的 5 条）

对每条「失声」或「落后」记录，勾选主因（可多选）：

| 规则 ID | 条件（来自标注） | 诊断结论 | 典型根因 |
|---------|------------------|----------|----------|
| D1 | `brand_mentioned=N` 且竞品 Y | **品类失声** | 品类问题集上内容资产不足 |
| D2 | `brand_mentioned=N` 且问法含品牌名 | **品牌未识别/未收录** | 别名缺失、官网结构化差、品牌声量低 |
| D3 | `mentioned=Y` 但 `in_answer_body=N` | **弱提及** | 仅边缘出现，证据不足以进正文 |
| D4 | `position=末段` 且竞品 `首推` | **叙事位置落后** | 缺少对比数据、首段无核心事实 |
| D5 | 豆包 SOA 显著低于 ChatGPT | **市场割裂** | 国内平台语料与出海叙事不一致 |

#### 4.2 写入 Sheet `04_诊断与建议`

| 列名 | 说明 |
|------|------|
| `question_id` | 问题 |
| `platform` | 平台 |
| `diagnosis_id` | D1–D5 |
| `evidence` | 引用回答原文一句 + 截图编号 |
| `priority` | P0 / P1 / P2 |

**优先级建议**：

- **P0**：品类词/对比词失声且竞品首推（直接影响采购决策）  
- **P1**：品牌词未识别（别名/官网问题）  
- **P2**：仅位置偏弱  

#### 4.3 验收标准

- [ ] 每条 P0 问题都有 **证据句**（能从 R1 回答中复制出来）  
- [ ] 至少 3 条诊断，覆盖 **≥2 种** 规则类型  
- [ ] 能向听众解释：「不是 AI 不喜欢我们，是这类问题我们 **没有可引用证据**」  

**对应商用产品**：页面审计引擎（**P1**）；规则诊断引擎（**P0**）；见 [产品设计分析 §0.4](./product-design-analysis#module-priority)。

### 阶段 ⑤ 优化（约 1–2 小时）

**业务目标**：产出 **可执行、可勾选** 的改进行动——实践版不求 AI 自动生成，求 **人能在 1 小时内改完 1–2 项**。待办须写清 **`D* + S* + 对标 URL`**（有信源盘时），见 [§0.6 ④ 归因层](./product-design-analysis#visibility-layers)。

#### 5.1 建议类型与最小动作

| 诊断 | 建议类型 | 最小优化动作（实践） |
|------|----------|----------------------|
| D1 品类失声 | 内容补全 | 官网新增 1 篇 FAQ：「出海小团队项目管理工具选型」，首段给出明确定义 + 3 条量化事实 |
| D2 品牌未识别 | 别名/结构 | 官网 About 增加中英品牌全称；表格中补充别名 |
| D3 弱提及 | 证据前置 | 把「客户数/适用团队规模」移到文章首段 |
| D4 叙事落后 | 对比块 | 增加「vs Asana」对比表：适用团队、价格、出海场景 |
| D5 市场割裂 | 本地化 | 增加中文版产品页摘要（豆包可引用的中文事实句） |

#### 5.2 Sheet `04` 继续填写

| 列名 | 说明 |
|------|------|
| `action_id` | A1, A2… |
| `action_title` | 一句话动作 |
| `action_detail` | 具体改哪一页、哪一段 |
| `owner` | 负责人 |
| `status` | 待办 / 进行中 / 已完成 |
| `completed_at` | 完成日期 |

**实践要求**：**至少完成 2 条 P0 建议**（真的去改官网/文档/Notion 公开页），不要只写「建议优化」。

#### 5.3 证据块自检（优化后立刻做）

改完后用 **30 秒检查** 是否符合清华「证据块」最小结构：

- [ ] **核心事实**：有没有数字、日期、可验证陈述？  
- [ ] **定义**：有没有「XX 是指…」？  
- [ ] **边界**：有没有适用团队规模/不适用场景？  
- [ ] **对比**：对比词问题是否能在页面上找到对照？  
- [ ] **更新记录**：页脚或文首是否有「最后更新于 YYYY-MM-DD」？  

#### 5.4 验收标准

- [ ] ≥2 条建议状态为 **已完成**  
- [ ] 每条已完成建议能对应到 **一个具体 URL 或文档段落**  
- [ ] 证据块自检 ≥4/5 项通过  

**对应商用产品**：优化建议列表、优化详情（**P0 最薄层**）；证据块重构（**P2**）。

---

### 阶段 ⑥ 验证与运营（约 1 小时 + 等待）

**业务目标**：证明 GEO 是 **闭环**——改完之后，答案份额有没有变化。

#### 6.1 第二轮采集（R2）

1. 仅对 **动过手的 P0 问题**（建议 3–6 题）重复阶段 ②，`round=R2`。  
2. 若内容刚发布，**等待 24–72 小时** 再问（语料刷新无固定周期；演示时说明「真实商用按日/周监测」）。  

#### 6.2 Sheet `05_验证对比`

| 列名 | 说明 |
|------|------|
| `question_id` | 问题 |
| `platform` | 平台 |
| `soa_r1` | 首轮是否进答案（0/1） |
| `soa_r2` | 次轮是否进答案（0/1） |
| `delta` | ↑ / ↓ / → |
| `linked_action` | 对应 action_id |
| `note` | 观察说明 |

#### 6.3 运营动作（实践版）

| 运营项 | 实践做法 | 商用产品 |
|--------|----------|----------|
| **异动记录** | R2 仍失声的问题标红 | 邮件/企微告警 |
| **待办闭环** | Sheet 中 status 全追踪 | 优化建议状态机 |
| **周报** | 用 1 页 Notion 写：SOA 变化 + 下周动作 | PDF 报告（**P1**） |
| **复盘** | 哪些动作有效、哪些无效 | 效果验证模块 V2 |

#### 6.4 验收标准（整条 MVP 链路）

- [ ] R1 → 诊断 → 优化 → R2 **完整跑通**  
- [ ] 至少 **1 个问题** SOA 从 0→1 或位置从末段→前列（若无变化，能解释原因，如等待时间不足）  
- [ ] 能填完下面的 **演示叙事卡**  

---

## 五、对外 / 对内演示脚本（15–20 分钟）

按顺序讲，配合表格 + 截图。

| 分钟 | 环节 | 说什么 | 展示什么 |
|:----:|------|--------|----------|
| 0–2 | **是什么** | GEO 解决「进没进 AI 答案」；不是 SEO 排名 | §二 闭环图 |

**收尾金句**：

> 这就是我们的 GEO 系统：**不是一次性写稿，而是持续监测答案、诊断缺口、执行优化、用 SOA 验证。** 今天用 2 个平台、20 条问题跑通；商用版是同一链路的自动化与规模化。

## 七、从实践 MVP 到商用产品（映射表）

| 实践环节 | 你亲手做了什么 | 商用产品化（P0） | P1+ |
|----------|----------------|------------------|-----|
| ① 策略规划 | Excel 问题集 | Onboarding + 关键词/竞品 CRUD | 主题地图、意图分层 |
| ② 采集 | 手动提问+截图 | 定时任务 + 回答存档 API/Playwright | 10 平台、多轮采样 |
| ③ 测量 | 人工标注+公式 | 品牌识别 + SOA 引擎 + Dashboard | CCR 引用 vs 提及 |
| ④ 诊断 | 5 条规则 | **规则建议引擎** | 页面审计、证据密度 |
| ⑤ 优化 | 人工改网页 | 建议清单 + 状态 + 操作指引 | LLM 证据块、Schema |
| ⑥ 验证运营 | R2 对比表 | 周期采集 + 前后对比 + 邮件告警 | 周报 PDF、自动化闭环 |
| 治理合规 | 证据块自检清单 | — | 合规引擎、幻觉检测 |

**规划结论（建议写入后续 PRD）**：

1. **工程 MVP** 可先只做 ②③（监测+SOA），用于验证采集。  
2. **商用首发** 必须包含 ④⑤⑥ 的 **最薄实现**，否则对外不能说「GEO 运营系统」。  
3. **定价** 单独文档；本实践不涉及。

## 八、常见问题（演示时可能被问）

**Q：只测 2 个平台，结论靠谱吗？**  
A：实践版验证 **方法论和闭环**；商用版扩展平台数量与采样次数，公式不变。

**Q：SOA 和「提及」有什么区别？**  
A：提及 = 出现过名字；进答案正文（SOA 分子）= 出现在推荐/论述中。商用版还会区分「引用为信源」（CCR，**P1**）。

**Q：优化后 AI 没变，是不是 GEO 无效？**  
A：AI 有随机性和语料滞后。GEO 看 **趋势和结构性失声**，不是改一次必上榜。实践要诚实记录「未改善」并调整动作。

**Q：和 Profound 等竞品差异？**  
A：本实践未涉及定价；叙事上强调 **双市场同一面板 + 从监测到建议到验证的闭环**，而非仅分数。

**Q：要不要做页面审计？**  
A：实践版用 **证据块自检** 代替；商用上 **P1** 再上 URL 级审计。

### A.3 采集：两条路径（实践 vs 商用）

| 路径 | 做法 | 优点 | 局限 |
|------|------|------|------|
| **A. 实践 / 工程 MVP（推荐先做）** | 用 [Trinity API](https://doc.trinitydesk.ai/quickstart?lang=zh) 调 **豆包模型**：`POST https://api.trinitydesk.ai/v1/chat/completions` | **几乎零额外基建**；可脚本化、可入库；你们本行 | 测的是 **模型回答**，不一定是 **豆包 App 消费端** 的完整产品体验（搜索、插件、信源策略可能不同） |
| **B. 真实 GEO 渠道（商用要对齐）** | 豆包 **网页 / App** 里用同一问法提问 + 截图 | 与用户真实触点一致 | 需 Playwright 或人工；维护成本高 |

### A.4 最小采集脚本思路（路径 A · 伪代码）

```text
for each question in question_set:
  response = POST /v1/chat/completions
    model = <豆包模型 ID>
    messages = [{ role: user, content: question }]
  save(question, response, round=R1, platform=doubao-via-trinity)
  annotate brand_mentioned, position, in_answer_body
  if platform returns citation block:
    save cited_urls[] with category (competitor_official | third_party_review | brand_official)
    compute brand_domain_hits, gap_ids S1–S6
aggregate SOA
run diagnosis D1–D5 + source gaps S* → actions with benchmark URLs
(after content changes on trinitydesk.ai / doc.trinitydesk.ai)
re-run same questions with round=R2
```

单平台 + 自有网关时，**采集模块可从「10–15 人天 Playwright」降到「1–3 人天脚本」**。

### A.6 单平台工程量（Trinity 实例）

| 范围 | 难度 | 粗估 |
|------|------|------|
| Excel 实践（路径 A 手动调 API 或网页问豆包） | 低 | **0.5 天** |
