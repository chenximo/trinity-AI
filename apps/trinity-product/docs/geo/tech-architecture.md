## 二、逐模块技术拆解

### 模块一：监测与测量层

**核心挑战**：如何获取多个 AI 平台对同一个问题的回答，并从中提取品牌信息？

#### 2.1.1 多平台数据采集

| 技术 | 用途 | 选型说明 |
|------|------|---------|
| **Playwright / Puppeteer** | 无头浏览器控制，模拟真实用户在 AI 平台提问 | 国内平台（豆包、文心、通义、元宝）多为网页端，需真实浏览器渲染；Selenium 偏重，Playwright 更现代 |
| **官方 API 接入** | 通过各平台开放 API 批量获取回答 | ChatGPT API、文心一言 API、通义千问 API、DeepSeek API、豆包 API（部分平台有免费额度） |
| **RPA 层** | 对无 API 的平台做 UI 自动化兜底 | 增长超人驭控系统同款技术路线，SpringBoot + RPA |
| **代理 IP 池** | 避免被 AI 平台限流/封禁 | 芝麻代理、快代理等国内服务商 |

#### 2.1.2 回答采集关键技术细节

```
采集流程：
  关键词管理 → 定时任务调度 → 多平台并行采集 → 原始回答存储 → 结构化处理

关键难点：
  1. 不同平台 API 返回格式差异大 → 需统一抽象层
  2. AI 回答具有随机性 → 同一问题需多次采样取稳定值
  3. 速率限制 → 分布式调度 + 流量整形
  4. 登录态维护 → Cookie/Session 管理
```

| 技术环节 | 推荐方案 |
|---------|---------|
| 任务调度 | **Celery**（Python）或 **XXL-JOB**（Java） |
| 并行采集 | **asyncio + aiohttp**（Python）或 **Reactor**（Go） |
| 限流控制 | Token Bucket 算法，Redis 实现 |
| 原始存储 | MongoDB（非结构化原始回答）+ MinIO（截图存档） |

#### 2.1.3 品牌提取与识别 {#measurement-soa}

> **主实现方案（共识）**：**规则引擎 + 别名库 + 时序库** 为主；NER/LLM 为可选增强。业务全景见 [GEO 业务全景 · ③ 测量](./business-landscape#measurement-soa-impl)；演示代码 `apps/trinity-geo/src/demo/analyze.ts`。

**三件套分工**：

| 组件 | 职责 | 实现要点 |
|------|------|----------|
| **别名库** | 品牌/竞品名称归一化 | 表结构：`entity_id`, `canonical_name`, `alias`, `entity_type`, `enabled`；控制台「品牌设置」写入或同步 |
| **规则引擎** | 单条回答标注 + SOA 聚合 | if/else：提及、进正文、位置、竞品；诊断 D1–D4 可同进程或下游服务 |
| **时序库** | 指标按时间存储与查询 | 维度：`brand_id`, `question_id`, `platform`, `round`, `collected_at`；支撑趋势、R1/R2、告警 |

**处理流水线**：

```text
raw_answer（采集库）
    → 别名库匹配（brand + competitor aliases）
    → 规则引擎（in_answer_body, mention_position, …）
    → annotations 表（单条标注）
    → 聚合写入 metrics 时序表（SOA / 分平台 / 分类型）
    → Dashboard / 告警 / 诊断输入
```

**别名库（主路径）**：

| 能力 | 实现 |
|------|------|
| 精确/忽略大小写匹配 | 字符串索引或正则；MVP 用 `brand.json` 同级配置 |
| 实体链接 | `alias` → `entity_id`（如 `trinity`, `openrouter`） |
| 竞品与品牌同库 | `entity_type` 区分；测量时一并扫描 |

**品牌别名示例**（复杂性仍适用）：

```
"华为" 在 AI 回答中可能出现的表述：
  · 华为 · HUAWEI · 华为技术有限公司 · Huawei Technologies
  · 华为手机 · 华为 Mate 系列 · …
→ 全部归一化到 entity_id: "huawei"
```

**可选增强（非 SOA 主路径）**：

| 技术 | 用途 | 何时启用 |
|------|------|----------|
| NER / 模糊匹配 | 别名表未覆盖的变体 | 别名库维护成本高时 |
| LLM 单次分类 | 弱提及边界、情感、指代消解 | 人工复核队列或抽检 |
| Spark/Flink | 超大规模批量聚合 | 采样量极大时；中小客户应用层聚合即可 |

**原则**：不以 LLM 直接输出 SOA 分数；指标须可回放（原始回答 + 规则版本 + 标注结果）。

#### 2.1.4 指标计算引擎

| 指标 | 计算逻辑 | 技术实现（主路径） |
|------|---------|-------------------|
| **SOA 答案份额** | `in_answer_body=Y 的条数 / 总采样条数` | **规则引擎**聚合 → **时序库** |
| **提及率** | `brand_mentioned=Y 的条数 / 总采样条数` | 同上 |
| **首推率** | 品牌首次出现位置在前列的占比 | 规则：`mention_position` ∈ {首推, 前列} |
| **CCR 引用捕获率** | 作为证据源次数 / 提及次数 | 引用规则或模型（**独立扩展**，非 SOA 前提）；见 [产品设计 · §0.6 ② 引用层](./product-design-analysis#visibility-layers) |
| **AVI 可见性指数** | `f(频次, 位置, 情感, 平台权重)` | 加权配置表 + 时序库 |
| **趋势对比** | 不同时间段指标变化 | **时序库**按日/周/月 roll-up；SQLite/PG/ClickHouse 按规模选型 |

#### 2.1.4a 参考来源采集（信源层）{#cited-sources}

> **产品真源**：[产品设计分析 · §0.6 ③ 信源层](./product-design-analysis#visibility-layers)。

部分消费端（如豆包 App）在回答中显式列出「官方依据」「第三方参考」等 **全部 URL**。商用采集须：

| 字段 | 说明 |
|------|------|
| `citation_extractable` | 本平台/渠道是否返回可解析链接 |
| `cited_urls[]` | `{ url, label, category, entity_id? }` |
| `category` | `competitor_official` / `third_party_review` / `brand_official` / `neutral` |
| `brand_domain_hits` | 我方域在 `cited_urls` 中的条数 |
| `gap_type` | 归因 S1–S6（规则或人工） |

**样本数据**：`apps/trinity-geo/mvp/data/r1/cited_sources.json`（Q00 · 16 链 · 0 我方域）。

**存储**：与 `raw_answers` 同行或 `answer_citations` 子表；回答详情页展示信源盘 M/N。

**存储分层（建议）**：

| 层 | 内容 | 说明 |
|----|------|------|
| 采集库 `raw_answers` | 全文、截图、platform、round、`cited_urls` | ② 写入；③ 只读 |
| 标注库 `annotations` | 单条标注字段 | 规则引擎输出 |
| 别名库 `entity_aliases` | 品牌/竞品别名 | ① 配置同步 |
| 问题集 `monitoring_questions` | 监测问法、类型、启用状态 | ① 配置；② 采集任务输入；[问题集 PRD](../../../trinity-geo/marketing/console/keywords.md) |
| 时序库 `metric_snapshots` | SOA 等聚合结果 | ③ 写入；⑥ Dashboard/告警读取 |

#### 2.1.3a 控制台配置同步与历史重算 {#alias-sync-recalc}

> **产品需求真源（本页）**：[品牌设置 PRD](../../../trinity-geo/marketing/console/brand-settings.md) §8–§9。  
> 控制台「品牌设置」保存的是 **配置**；SOA 数字在 **标注库 + 时序库**，二者通过同步与重算衔接。

**配置写入（保存品牌设置）**：

```text
PATCH /api/console/brands/:brand_id
  → 事务写入 brands（主名称、官网、行业…）
  → 差量更新 entity_aliases（entity_type=brand, enabled）
  → 更新 brands.aliases_synced_at
  → 若别名集有变：创建 annotation_recalc_jobs（status=pending）
```

**侧栏「测量同步状态」↔ 存储映射**：

| UI 字段 | 数据来源 |
|---------|----------|
| 实体 ID | `brands.id` 或 slug（如 `trinity`） |
| 已启用别名 | `COUNT(entity_aliases WHERE enabled)` |
| 上次同步 | `brands.aliases_synced_at` |
| 待重算 | 最新 `annotation_recalc_jobs.status`（pending / running / done / failed） |

**历史重算工作流**（非 LLM，不重采集）：

```text
annotation_recalc_jobs（pending）
  → Worker 按时间窗读取 raw_answers
  → 用当前 entity_aliases + 规则版本重跑规则引擎
  → UPSERT annotations
  → 重聚合 metric_snapshots（按 question / platform / day）
  → job.status = done；Dashboard 读新时序
```

| 表（逻辑名） | 职责 |
|--------------|------|
| `brands` | 品牌主档；`aliases_synced_at` |
| `entity_aliases` | 别名库真源；控制台与竞品管理共用 |
| `annotation_recalc_jobs` | 重算任务队列（`brand_id`, `window_days`, `status`, `started_at`, `finished_at`） |
| `raw_answers` | 只读；重算输入 |
| `annotations` | 重算输出 |
| `metric_snapshots` | 重算后刷新 |

**MVP 样本**：`mvp/config/brand.json` 等价于 `brands` + `entity_aliases` 的文件态；无任务表时重算可手动脚本触发。HTML 原型为 Mock，不落库。

### 模块二：诊断与审计层

**核心挑战**：如何判断一个网页"适不适合被 AI 引用"？

#### 2.2.1 网页内容抓取与解析

| 技术 | 用途 |
|------|------|
| **Scrapy / Crawlee** | 批量网页抓取框架 |
| **BeautifulSoup / Jsoup** | HTML 解析 |
| **Playwright** | JS 渲染页面抓取（SPA 站点必需） |
| **Readability 算法** | 提取网页正文，去除广告/导航噪音 |

#### 2.2.2 证据密度评分（核心难点）

这是 GEO 产品最核心的技术差异化点——**评估页面作为"AI 可引用证据"的质量**。

```
证据密度评分维度：

1. 事实完整性检测
   · 是否有量化数据？（数字、百分比、金额）
   · 是否有明确定义？（"XX 是指..." 句式检测）
   · 是否有时间戳？（发布时间、更新时间）
   · 是否有出处标注？（引用来源、参考文献）

2. 边界清晰度检测
   · 是否说明了适用范围和限制条件？
   · 是否有"不适用于..."等边界语句？
   · 对比关系是否完整？

3. 结构可解析性
   · Schema.org 标记完整度
   · 标题层级（H1-H6）是否合理
   · 列表/表格等结构化元素使用

4. 更新时效性
   · 最后更新时间距今多久？
   · 是否有版本号？
   · 是否有更新日志？
```

| 技术 | 用途 |
|------|------|
| **文本分类模型** | 判断段落类型（事实/观点/定义/对比/边界） |
| **信息抽取（IE）** | 从文本中提取结构化事实三元组 |
| **Schema 验证器** | 检查 Schema.org 标记的完整性和正确性 |
| **正则 + 规则引擎** | 检测日期、数字、引用格式等结构化模式 |
| **Readability Score** | 可读性评估（汉语用偏旁复杂度、句长等） |

#### 2.2.3 技术 SEO 审计（AI 爬虫视角）

| 检查项 | 技术方案 |
|--------|---------|
| JavaScript 渲染依赖 | Playwright 渲染对比（是否依赖 JS 才显示核心内容） |
| meta 标签完整度 | HTML 解析 + 规则检查 |
| Schema.org 标记 | JSON-LD 解析 + 标准对比 |
| robots.txt / sitemap | 解析 + 可抓取性验证 |
| 页面加载速度 | Lighthouse / WebPageTest API |
| 移动端适配 | 移动端 UA 模拟 + 响应式检测 |

#### 2.2.4 审计报告生成

```
数据采集 → 多维度评分 → 问题聚合 → 优先级排序 → 建议生成 → 报告渲染

关键：不能只告诉用户"你有3个问题"，
     要告诉用户"你应该先修这个，因为它是被AI引用的第一道门槛"
```

### 模块三：内容优化与生成层

**核心挑战**：如何辅助用户把"普通页面"改造成"证据块"？

#### 3.1 文本处理

| 技术 | 用途 |
|------|------|
| **LLM（GPT-4/Claude/DeepSeek）** | 核心优化引擎：证据块重构、意图对齐改写 |
| **textstat / 可读性** | 优化前后可读性对比 |
| **difflib / diff-match-patch** | 优化前后差异可视化 |
| **Grammarly API / 写作猫** | 语法校对 |

#### 3.2 Schema 自动生成

| 技术 | 用途 |
|------|------|
| **信息抽取 + 模板填充** | 从页面内容自动生成 JSON-LD Schema |
| **Schema.org 类型库** | Article、Product、FAQ、HowTo、Review 等标准模板 |
| **验证器** | Google Rich Results Test API |

#### 3.3 语义网络构建（内部链接优化）

| 技术 | 用途 |
|------|------|
| **关键词聚类** | 将相关内容页面分组（K-Means / HDBSCAN） |
| **实体关系抽取** | 从页面内容抽取实体间关系 |
| **图算法** | PageRank / 社区发现 - 找出应该互相链接的页面群 |
| **Neo4j / NetworkX** | 存储和计算页面关联图 |

#### 3.4 多模态优化

| 模态 | 优化点 | 技术 |
|------|--------|------|
| **图片** | Alt 文本生成、OCR 验证 | CLIP / BLIP 图片描述 + PaddleOCR |
| **表格** | 结构化标记 | 表格检测 → JSON-LD Dataset Schema |
| **视频** | 字幕文本提取、场景描述 | Whisper 语音转文字 + 关键帧截图 |

### 模块四：策略与规划层

**核心挑战**：帮用户回答"我应该优化哪些问题？按什么顺序？"

#### 4.1 关键词/问题发现

| 技术 | 用途 |
|------|------|
| **AI 平台搜索建议 API** | 获取各平台的热门/联想问题 |
| **竞品关键词反查** | 从竞品 AI 表现反推其在回答哪些问题 |
| **用户评论/问答挖掘** | 从电商评论、知乎、小红书等提取真实用户问题 |
| **N-gram + TF-IDF** | 从数据中提取高频问题模式 |

#### 4.2 意图识别与分类

```
意图分类体系（用户从认知到购买的旅程）：

L1 痛点觉醒：  "XX 行业有哪些常见问题？"
L2 认知建立：  "什么是 GEO？和 SEO 有什么区别？"
L3 方案评估：  "GEO 工具有哪些？哪个好？"
L4 决策对比：  "Profound 和 Scrunch 怎么选？"
L5 购买引导：  "Profound 价格多少？如何购买？"
```

| 技术 | 用途 |
|------|------|
| **意图分类模型** | BERT 微调多分类（L1-L5 五级意图） |
| **Embedding + 聚类** | 将相似意图的问题归拢，避免重复优化 |
| **搜索引擎意图标注数据** | 百度/Google 搜索意图标注（信息型/导航型/交易型）辅助训练 |

#### 4.3 优先级排序算法

```
问题优先级 = f(
  搜索量/提问频率,      ← 有多少人在问
  商业价值,              ← 回答这个问题能带来多少转化
  当前覆盖度,            ← 我的品牌现在AI回答中出现了吗
  可优化空间,            ← 优化难度和预期提升空间
  竞品覆盖度             ← 竞品已经覆盖了多少
)
```

| 技术 | 用途 |
|------|------|
| **多因子加权评分** | 优先级排序核心算法 |
| **AHP 层次分析法** | 权重确定（可让用户自定义权重） |
| **竞品数据 API** | 获取竞品在各平台的覆盖度数据 |

#### 4.4 用户旅程建模

| 技术 | 用途 |
|------|------|
| **马尔可夫链** | 模拟用户从问题 A 到问题 B 的概率转移 |
| **图可视化（D3.js / G6）** | 将用户旅程地图可视化 |
| **GA4 / 百度统计 API** | 获取实际用户搜索行为数据 |

### 模块五：自动化运营层

**核心挑战**：把"监测→诊断→优化"串成自动运转的闭环。

#### 5.1 任务编排与工作流引擎

| 技术 | 用途 |
|------|------|
| **Temporal / Cadence** | 复杂工作流编排（最推荐，支持重试、补偿、可视化管理） |
| **Apache Airflow** | Python 数据管道编排（偏数据分析场景） |
| **Camunda / Flowable** | Java 业务流程引擎（偏企业审批流场景） |
| **Celery + Beat** | 轻量定时任务 + 工作流（适合早期 MVP） |

**典型自动化工作流**：
```
触发器：检测到竞品在"GEO工具对比"问题中新出现
 ↓
Step 1: 自动采集竞品被引用的页面内容
 ↓
Step 2: 对比分析竞品页面 vs 我方页面的证据密度
 ↓
Step 3: 生成优化建议任务，分配负责人
 ↓
Step 4: 优化完成后自动重新采集验证效果
 ↓
Step 5: 在仪表板更新指标
```

#### 5.2 引用缺口自动发现

| 技术 | 用途 |
|------|------|
| **差异算法** | 对比"AI 回答了但没提到品牌"的问题 vs "品牌有相关页面"的覆盖 |
| **语义相似度** | 计算用户问题和品牌内容的语义匹配度（Sentence-BERT / text2vec） |
| **阈值判断** | 语义相似度高但没有被引用 → 标记为优化机会 |

#### 5.3 内容自动刷新

| 技术 | 用途 |
|------|------|
| **CMS API 集成** | WordPress REST API / Strapi API / 自定义 CMS 对接 |
| **变化检测** | 对比原始内容和改写内容，只推送差异部分 |
| **发布管道** | 草稿 → 预览 → 审核 → 发布的自动流转 |

#### 5.4 告警通知

| 技术 | 用途 |
|------|------|
| **WebSocket / SSE** | 实时推送前端告警 |
| **企业微信/钉钉/飞书 Bot** | 通过 Webhook 推送告警到 IM |
| **邮件通知** | SMTP + 模板渲染 |
| **Prometheus + AlertManager** | 指标异常告警 |

### 模块六：治理与合规层

**核心挑战**：在国内强监管环境下，确保优化行为合法合规。

#### 6.1 合规审查引擎

| 审查项 | 技术方案 |
|--------|---------|
| **敏感词检测** | AC 自动机 + 敏感词库（定期更新） |
| **虚假信息检测** | 事实验证模型（Fact Verification）+ 权威信源交叉验证 |
| **不当承诺检测** | 规则 + 文本分类（"100%""绝对""保证"等危险表述） |
| **医疗/金融合规** | 行业专项规则引擎 + AI 辅助审核 |
| **广告法合规** | 极限用语检测（"最好""第一""唯一"等） |

**国内监管依据**：
- 《生成式人工智能服务管理暂行办法》
- 《互联网信息服务深度合成管理规定》
- 《广告法》 + 《反不正当竞争法》
- 各 AI 平台的内容政策与信源准入标准

#### 6.2 可回溯证据链

```
每条 GEO 优化建议必须包含：

  · 数据来源（哪个平台、哪个问题、哪个时间点采集）
  · 原始回答截图/文本（作为证据锚点）
  · 分析推理过程（为什么建议这样优化）
  · 优化前后对比（效果验证）
  · 操作人 + 时间戳

→ 完整审计日志 = 合规底线
```

| 技术 | 用途 |
|------|------|
| **区块链存证**（可选） | 关键证据的不可篡改时间戳存证 |
| **ELK 日志系统** | 收集和检索所有操作日志 |
| **数字签名** | 确保证据来源的真实性 |

#### 6.3 风险分级管理

```
风险评估矩阵：

              低业务影响        高业务影响
              ─────────        ─────────
  高合规风险   谨慎优化          重点治理
              （人工审核）       （法务介入）
              
  低合规风险   可自动化          标准优化
              （批量处理）       （半自动审核）
```

| 技术 | 用途 |
|------|------|
| **风险评分模型** | 行业敏感度 × 内容类型 × 平台政策 → 风险等级 |
| **审批工作流引擎** | 高风险操作必须多人审批 |
| **熔断机制** | 检测到异常行为模式自动暂停优化 |

#### 6.4 幻觉检测

| 技术 | 用途 |
|------|------|
| **多模型交叉验证** | 同一问题用多个 AI 模型回答，交叉比对一致性 |
| **知识库对账** | AI 回答 vs 品牌官方知识库，发现不一致 |
| **事实核查 API** | 对接第三方事实核查服务 |
| **异常检测算法** | 监控 AI 回答中品牌相关的异常变化 |

## 三、核心技术选型建议

### 3.1 整体技术栈推荐

```
推荐方案一线：Python（AI/数据密集型）

  后端：  FastAPI / Django
  异步：  Celery + Redis
  AI:     LangChain / LlamaIndex + 国产大模型 API
  数据：  PostgreSQL + ClickHouse + Milvus
  采集：  Scrapy + Playwright
  NLP:    transformers + text2vec + HanLP
  部署：  Docker + K8s

推荐方案二线：Java（企业级/高并发）

  后端：  Spring Boot / Spring Cloud
  任务：  XXL-JOB + Temporal
  数据：  MySQL + ES + ClickHouse
  采集：  Crawlee（Node.js 侧）+ 内部 API
  部署：  Docker + K8s + Nacos

推荐方案三线：Go（高性能采集层）

  采集：  纯 Go 高并发采集引擎
  网关：  Go API Gateway（Kong/Tyk）
  消息：  Kafka
  数据：  ClickHouse
  其余用 Python 微服务补充 AI 能力
```

### 3.2 按阶段的渐进式选型

| 阶段 | 技术栈 | 说明 |
|------|--------|------|
| **MVP（1-3月）** | FastAPI + PostgreSQL + Redis + Celery + Playwright + 前端 Vue3 | 快速验证核心监测+审计功能 |
| **成长（3-9月）** | + ClickHouse + Milvus + LangChain + Kafka + K8s | 数据量大后升级存储和计算 |
| **成熟（9月+）** | + Temporal + Neo4j + Flink + 多语言微服务 | 自动化闭环和多模态支持 |

### 3.3 关键技术难点排序

| 难度 | 技术难点 | 为什么难 | 建议方案 |
|:----:|---------|---------|---------|
| ⭐⭐⭐⭐⭐ | **证据密度评分模型** | 没有公开标准，需要自建评分体系和标注数据 | 人工标注 500+ 页面构建数据集，多维度回归模型 |
| ⭐⭐⭐⭐⭐ | **多平台稳定采集** | 平台频繁改版、反爬升级、登录态复杂 | 采集层单独服务，Playwright + 代理池 + 降级策略 |
| ⭐⭐⭐⭐ | **引用 vs 提及区分** | 需要训练分类器判断 AI 回答中品牌的角色 | 人工标注数据 + BERT 微调二分类 |
| ⭐⭐⭐⭐ | **品牌别名归一化** | 品牌表述多样性极高 | 自建品牌知识库 + 持续维护别名表 |
| ⭐⭐⭐ | **AI 回答随机性** | 同一问题重复问回答不同 | 多次采样取稳定值，统计置信区间 |
| ⭐⭐⭐ | **合规审查引擎** | 规则复杂，行业差异大 | 规则引擎 + LLM 辅助审核双通道 |
| ⭐⭐ | **数据可视化** | 指标多、维度多、需要灵活下钻 | ECharts + 数据下钻能力设计 |
