## 三、研发协作规范

### 3.1 需求评审流程

```
PM 起草 PRD → 小组内部评审 → 技术可行性评估 → 正式评审会 → 确认排期

评审会必问清单：
□ 这个功能解决谁的什么问题？
□ 不做会怎样？
□ 有没有更简单的做法？
□ 数据从哪里来？（新采集还是复用现有数据）
□ 边界条件是否穷举？
□ 异常情况如何表现？
□ 什么叫"做完"？（验收标准是否明确）
```

### 3.2 需求优先级判定

```
P0 — 必须做：不做产品不可用
P1 — 应该做：影响用户体验或竞争力，本月内完成
P2 — 可以做：锦上添花，有资源再做
P3 — 暂不做：明确本期不做，放入需求池
```

### 3.3 版本节奏建议

```
Sprint 1-2：   MVP 核心监测功能
Sprint 3-4：   监测增强 + 竞品对比
Sprint 5-6：   用户反馈修复 + 稳定性
─────────────────────────── V1.0 发布
Sprint 7-8：   内容审计引擎
Sprint 9-10：  优化建议 + 报告中心
Sprint 11-12： 用户反馈修复
─────────────────────────── V1.5 发布
```

## 四、用户验证机制

### 4.1 验证节奏

| 阶段 | 验证方式 | 频率 | 对象 |
|------|---------|------|------|
| 需求阶段 | 用户访谈 | 按需 | 3-5 个潜在用户 |
| 设计阶段 | 原型测试 | 每功能 | 2-3 个目标用户 |
| 开发阶段 | 内部 Dogfood | 持续 | 团队自己 |
| 上线前 | Alpha 内测 | 1-2 周 | 5-10 个种子用户 |
| 上线后 | 数据+反馈 | 持续 | 全体用户 |

### 4.2 种子用户画像

## 六、合规前置清单

### 作为 GEO PM，以下合规事项必须在产品设计阶段就考虑：

| 合规项 | 设计要求 | 责任方 |
|--------|---------|--------|
| 用户数据隐私 | 用户品牌/关键词/竞品配置数据加密存储 | PM + 后端 |
| AI 平台服务条款 | 采集行为不违反各平台 ToS（速率控制、用途限制） | PM + 法务 |
| 内容合规 | 优化建议不能包含虚假、夸大、医疗建议等 | PM + 法务 |
| 广告法合规 | 优化后的内容不出现"最好""第一""唯一"等极限词 | PM + 法务 |
| GDPR/个保法 | 欧盟/国内用户数据收集和删除权 | PM + 法务 |
| 知识产权 | 优化后的内容版权归属明确（属于品牌方） | PM + 法务 |
| 可审计性 | 所有操作留日志，可回溯 | PM + 后端 |

## 八、GEO PM 的核心判断力

作为 GEO 产品经理，你需要有能力判断以下问题——这些是 **"外行 PM"和"懂 GEO 的 PM"的分水岭**：

| 问题 | 外行 PM 的回答 | 懂 GEO 的 PM 的回答 |
|------|-------------|-------------------|
| 用户说"我想提升排名" | "好的我们做关键词优化" | "排名不重要，你真正需要的是在AI回答里的答案份额 SOA" |
| 用户说"帮我多写点内容" | "好的我们加内容生成功能" | "内容数量不解决问题，关键是证据密度和可引用性" |
| 竞品出了一个新功能 | "我们也做吧" | "这个功能解决用户的什么问题？和我们路线图的差异点是什么？" |
| 采集成本高怎么办 | "砍掉采集功能" | "监测是GEO的基础，降本可以用采样策略而非砍功能" |
| 用户说"这个报告不够好看" | "好的我们加更多图表" | "报告好不好看是次要的，能帮用户回答老板'AI里能看到我们吗'才是核心" |
