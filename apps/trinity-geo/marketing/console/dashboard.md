## 1. 这是什么页面？（终局定位）

**不是**运维监控报表（CPU、日志、队列）。

**是**品牌方登录后的 **经营驾驶舱 / 运营看板**——在 [业务全景 · 数据流](../../../../trinity-product/docs/geo/business-landscape.md) 里，它对应指标仓之上的 **`[运营看板]`**：

> 把六环闭环里已经发生的 **监测 → 测量 →（部分）诊断 →（部分）验证** 结果，压成 **一屏可读**，并指向各模块下钻。

用户第一眼应能回答：

| 问题 | 总览须覆盖 |
|------|------------|
| 品牌在 AI 答案里 **整体表现** 如何？ | SOA、趋势、周环比 |
| **海外 vs 国内** 是否一致？ | 双市场 Tab、分市场 SOA |
| **哪个平台 / 哪类问题** 拖后腿？ | 平台分布、回答流、问题类型摘要 |
| **竞品** 是否在挤占？ | 竞品对比摘要 |
| **接下来该干什么**？ | 诊断提示、待办/优化入口、告警摘要（终局） |

底层永远依赖 **② 采集 + ③ 测量**；总览 **不负责** 配词表、跑任务、改页面——那些在各模块子页。

| 对比 | 可见性总览（本页） | 监测 / 诊断 / 优化等子模块 |
|------|-------------------|---------------------------|
| 角色 | 闭环的 **读口** 与 **导航枢纽** | 闭环的 **操作台** |
| 典型问题 | 好不好？急什么？ | 怎么采？全文啥？改哪页？ |
| 类比 | CEO 周报首页 | 工单、明细、编辑器 |

官网 Hero 里的 Dashboard Mock = 本页的 **营销裁剪版**。

## 2. 与完整 GEO 主链的关系（六环 · 不拆版本）

```text
① 策略规划 ──► ② 监测采集 ──► ③ 测量 ──► ④ 诊断 ──► ⑤ 优化 ──► ⑥ 验证
                                      │         │              │
                                      └─────────┴──────┬───────┘
                                                     ▼
                                            【可见性总览 · 本页】
                                            聚合指标 + 异常信号 + 下钻入口
                                                     │
                                                     └──► 回到各环继续经营
```

| 六环 | 总览上的 **终局职责** | 说明 |
|:----:|----------------------|------|
| ① 策略 | **轻量摘要** | 当前监测多少词、几类问法覆盖；完整地图在策略/关键词模块 |
| ② 采集 | **状态与新鲜度** | 各平台最近采集时间、是否在采集中；日志在监测子页 |
| ③ 测量 | **核心主场** | SOA、提及、位置、情感、平台/时间/关键词三层 |
| ④ 诊断 | **异常卡片** | 失声类型（D1–D4）、审计红灯计数、幻觉/负面提示 |
| ⑤ 优化 | **待办摘要** | 进行中的优化任务、建议优先级 Top N |
| ⑥ 验证 | **效果条** | R1→R2 **信源盘 Δ** + SOA 变化；链 [效果验证](./verify.html) |

总览 **不是** 只服务「测量」一环；它是 **整条 GEO 经营闭环的读口**。测量最重，是因为没有 ③ 就没有可汇报的数字。

## 3. 与六大产品模块的映射（完整能力图）

依据 [产品设计 · 六大模块](../../../../trinity-product/docs/geo/product-design-analysis.md#三产品设计框架六大核心模块)：

| 模块 | 总览应反映的能力（终局） | 下钻去哪 |
|------|--------------------------|----------|
| **一、监测与测量** | SOA、AVI、平台分布、提及、情感趋势、**CCR 引用 vs 提及**、竞品首推/前三率 | 监测概览、关键词详情、回答详情 |
| **二、诊断与审计** | 诊断结论 Top（如 D1 失声）、可引用性审计红灯数、证据密度均分 | 诊断列表、页面审计 |
| **三、优化与生成** | 待办优化条数、预期 SOA 提升、最近发布证据块 | 优化建议列表、优化详情 |
| **四、策略与规划** | 问题集覆盖度、意图分层缺口提示 | 问题地图、关键词管理 |
| **五、自动化运营** | **异动告警**（SOA 跌、竞品新覆盖）、定时周报状态 | 通知中心、报告列表 |
| **六、治理与合规** | 幻觉/负面提及预警、合规风险计数 | 合规手册、风险详情 |

**设计原则**：总览只放 **聚合态 + 最急信号**；任何能力都有 **独立模块页** 承接细节，避免一屏堆成 BI 大屏。

## 4. 总览应呈现的「能力层」（终局清单）

不限于当前 HTML 已画的块——这是 **产品分析上的完整菜单**：

### 4.1 测量与可见性（核心层 · 必有）

| 能力 | 用户价值 |
|------|----------|
| SOA 总览 + 分市场 SOA | 进没进答案、哪边生态更弱 |
| 时间趋势（日/周/月） | 经营是否改善 |
| 平台级 SOA / AVI | 10+ 平台哪里掉队 |
| 关键词级摘要 | 哪道题最危险（Top 未进答案） |
| 提及 vs **引用（CCR）** | 被提到 vs 被当信源——两层竞争 |
| 情感倾向 | 推荐态 vs 中性提及 vs 负面 |
| 竞品同题 SOA 对比 | 相对位置，不只看自己 |

### 4.2 诊断与行动（闭环层 · 终局必有）

| 能力 | 用户价值 |
|------|----------|
| 诊断类型标签（D1–D4 等） | 失声 / 弱提及 / 竞品压制 / 叙事偏差 |
| 优化待办摘要 | 从「看见问题」到「知道改什么」 |
| 验证进度（R1 vs R2） | 改了有没有效 |

### 4.3 运营与治理（自动化层）

| 能力 | 用户价值 |
|------|----------|
| 邮件/企微告警摘要 | 不用盯盘 |
| 周报/月报生成状态 | 汇报材料 |
| 幻觉、合规风险提示 | 品牌安全 |

### 4.4 双市场（Trinity 差异化 · 贯穿各层）

海外 5 + 国内 5 **同一面板、同一问题集**——不是两个产品拼在一起。

## 6. SOA 三层 · 在总览上的终局分工

| SOA 层级 | 总览 | 完整下钻 |
|----------|:----:|----------|
| **时间趋势** | 全量展示 | — |
| **平台级** | 全量或 Top N | 关键词详情按平台 |
| **关键词级** | **摘要**（Top 风险问题） | 关键词管理 / 详情 |

关键词级不宜在总览堆完整表格——只露 **最该马上点进去的 5～10 条**。
