## 1. 背景与问题

### 1.1 用户原声

> 「品类词 SOA 是 0，是豆包没提我还是所有平台都没提？」  
> 「这条问法要不要优先做内容？」  
> 「我们有两个子产品，能分开看可见性吗？」

### 1.2 页面定位

| 项 | 说明 |
|----|------|
| **用户问题** | 「这道题（这条监测问法）我的可见性怎么样？」 |
| **产品角色** | 单 `question_id` 的 SOA 趋势 + 本题竞品排名 + 叙事标签 + 分平台摘要 + 最新回答入口 |
| **样本** | Q00「推荐两款 API 聚合平台」— **D1 品类失声**（SOA 0%，竞品进答案） |

---

## 2. 用户故事

```
作为 增长负责人，
我想要 从总览下钻到某条监测问法，查看各 AI 平台的 SOA、竞品对比与叙事标签，
以便 判断是全局失声还是个别平台问题，并点开具体回答取证。
```

## 3. 功能需求

### 3.1 布局

| 区域 | 说明 |
|------|------|
| 面包屑 | 总览 → 可见性下钻 → Qxx |
| 左侧栏 | 监测问题列表 + SOA 摘要；当前题高亮 |
| 主区 | KPI、趋势、竞品排名、标签、分平台表、回答列表 |

### 3.2 页头

| 元素 | 说明 |
|------|------|
| 分区标签 | 可见性 · 单题下钻 |
| 问法全文 | 与问题集一致 |
| 类型标签 | 品类词 / 品牌词 / 对比词 / 场景词 |
| CTA | 返回总览；查看最新回答 |

### 3.3 诊断横幅（条件展示）

当该题触发 D1–D4 时展示摘要条，链诊断模块。

**Q00 样本**：D1 + S1+S2+S3 — SOA 0%；豆包 R1 信源盘 **0/16**。

### 3.4 KPI 行

| 指标 | 说明 |
|------|------|
| SOA 7d | 该题近 7 日进正文占比 |
| 信源盘 | 可提取平台最近一轮我方域 M/N（如豆包 R1） |
| CCR | 被引为信源占比 |
| 提及率 | 品牌名出现（含未进正文） |

### 3.5 SOA 趋势

| 能力 | 说明 |
|------|------|
| 周期 | 日 / 周 / 月 |
| 数据 | `metric_snapshots` 按 `question_id` 过滤 |
| 竞品线 | 同题下 Top 竞品 SOA 趋势（可选 1–2 条） |

### 3.6 本题竞品 SOA 排名

横向条形排名，链 [竞品详情](./competitor-detail.md)。

### 3.7 叙事标签

从本题采样回答中抽取高频主题词；未进正文提及的标签降权显示。完整情感见情感读口。

### 3.8 分平台摘要表

| 列 | 说明 |
|----|------|
| 平台 | 10 平台之一 |
| 市场 | 国内 / 海外 |
| SOA 7d | 该平台该题 SOA |
| 信源 M/N | 该平台最近可提取回答的我方域命中（无则 —） |
| 竞品领先 | 进正文竞品名（若有） |
| 最近回答 | 时间 + 链回答详情 |

### 3.9 本题最新回答

列表：平台 · 轮次 · 日期 · 摘要 · 进答案徽章 · 诊断标记。

## 6. 一品牌多子产品 · 支撑方案

> **结论（MVP）**：无独立「子产品」顶栏模块，但可通过 **实体 + 问题绑定** 支撑；本页即子产品线可见性的 **下钻驾驶舱**。

### 6.1 已有能力（配置层）

| 机制 | 说明 | 原型位置 |
|------|------|----------|
| **别名库** | 同一品牌下多写法（如 Trinity AI、Trinity Desk、trinitydesk.ai）统一识别 | [brand-settings](./brand-settings.html) |
| **问题类型** | 品牌词 / 品类词 / 对比词 / 场景词 — 区分意图 | [keywords](./keywords.html) |
| **竞品实体** | 与品牌分表，`entity_type` 区分 | [competitors-manage](./competitors-manage.html) |
| **单题下钻** | 每题独立 SOA、趋势、竞品、标签 | **本页** |

### 6.2 推荐数据模型（P1，不阻塞 MVP 原型）

```
brand (1)
  └── product_line / sub_brand (0..n)   # 可选，如「Desk」「Cloud API」
        └── alias[]                      # 继承品牌别名库或覆盖
        └── question[]                   # 监测问法绑定 entity_id
```

| 场景 | 做法 |
|------|------|
| **仅 1 个子产品** | 不必建 product 实体；用 **品牌词 + 专属品类词** 即可 |
| **2–3 个子产品** | 每产品线一组监测问法 + 别名；左侧问题列表按 `entity_id` 筛选 |
| **多 SKU（手机等）** | P2：策略模块增加产品目录 + 按 SKU 聚合 SOA（参考 vivo 类客户） |

### 6.3 指标与总览

| 层级 | 聚合 | 页面 |
|------|------|------|
| 品牌 | 全问题集 SOA | 总览读口 ① |
| 子产品 / 产品线 | `WHERE entity_id = ?` | 总览可加筛选器；下钻仍为本页 |
| 单题 | `question_id` | **本页** |

### 6.4 与 Generforce「产品可见性」差异

- Generforce：左侧 **SKU / 产品目录**，右侧产品级 SOA。  
- Trinity：**左侧监测问题列表**（可挂 `entity_id`），右侧 **问法级 SOA** — 更贴合 B2B「意图监测」而非零售 SKU。  
- 若客户强需求 SKU 维度：在问题集增加 `product_line` 字段 + 总览筛选即可，无需复制 Generforce 双顶栏 IA。
