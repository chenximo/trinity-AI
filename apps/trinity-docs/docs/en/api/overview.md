# API overview

Trinity provides an **OpenAI-compatible HTTP API**. Requests are authenticated and routed through the Trinity gateway, then dispatched by the request body **`model`** (model ID).

## Base URL

| Item | Value (example) |
| --- | --- |
| Base URL | `https://api.trinitydesk.ai/v1` |
| Protocol | HTTPS |

Paths are relative to a base URL that includes **`/v1`** (for example `POST {TRINITY_BASE_URL}/chat/completions`).

## Authentication

Every request must include:

```http
Authorization: Bearer <TRINITY_API_KEY>
Content-Type: application/json
```

- Create API keys in the console. Keys usually start with **`xh-...`** (see [Manage API keys](../manage-api-keys.md)).
- Store only the key value in environment variables; do not include the `Bearer` prefix there.

## Tracing and settlement headers

Applies to **`POST /v1/chat/completions`** (text and image generation) and streaming responses. Video task creation follows gateway behavior.

| Header | Required | Purpose |
| --- | --- | --- |
| `X-Request-Id` | No | **Trace ID** for support and log correlation; max 128 chars; generated by the server if omitted |
| `X-Idempotency-Key` | No | **Settlement idempotency key**; same key in a workspace charges once on successful settlement; keep unchanged when retrying the same business operation |
| `X-Conversation-Id` | No | Conversation grouping ID; keep stable per Agent / chat for prompt cache; max 128 chars |
| `X-Session-Id` | No | Alias for `X-Conversation-Id`; used only when the latter is omitted |

Responses (including SSE) typically return `X-Request-Id` and `X-Settlement-Key`; `X-Conversation-Id` is echoed when provided.

Some text models support **prompt caching** (lower input cost when later turns reuse the same prefix). A stable `X-Conversation-Id` improves hit rate; check `usage.prompt_tokens_details.cached_tokens`. Billing and fields: [Chat completions Â· Advanced parameters Â· Prompt cache](./chat-completions-parameters.md#prompt-cache).

::: warning Billing
Without `X-Idempotency-Key`, each HTTP call is billed independently. When replaying the same business operation after a timeout, keep the settlement key unchanged; the trace ID may change.
:::

See request-header examples in [Create chat completion](./chat-completions.md).

## Supported capabilities

| Capability | Method | Path | Notes |
| --- | --- | --- | --- |
| List models | `GET` | `/models` | Models available to your API key; optional `?modality=text\|image\|video\|all` |
| Text / chat completions | `POST` | `/chat/completions` | Text, tool calling, multimodal input Parts |
| Image generation | `POST` | `/chat/completions` | Requires `modalities` with `image` + `image_config`; **not** `/images/generations`; synchronous long-running |
| Query image task | `GET` | `/image/tasks/{taskId}` | Compensating query after sync timeout (`408`); `taskId` is `trinity_task.task_id` |
| Video generation | `POST` | `/video/generations` | Creates an async task |
| Query video task | `GET` | `/video/tasks/{taskId}` | `taskId` from the create response, commonly `trinity_task.task_id` |

## Model ID (`model` field)

Set request body **`model`** to a **model ID** available to your account. Examples:

- Text: `gpt-5.5`, `gpt-4o`
- Image: `hunyuan-image`
- Video: `kling-2.6`

**How to find model IDs**:

- **API**: `GET /models` (optional `modality` filter)â€”see [List models](./models.md)
- **Console**: [Model catalog](https://trinity.ai/models)â€”copy the **model ID**

Use each list item's `id` as `model` in downstream requests. Unknown IDs return model-not-found style errors.

## Responses and errors

- Success: JSON or SSE (`stream: true`), aligned with OpenAI-style structures where applicable.
- Failure: OpenAI-style `error` object; see [Errors & debugging](../reference/error-codes.md).
